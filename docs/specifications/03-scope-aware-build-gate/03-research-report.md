## Summary

Follow-up research resolving the 5 open issues (ISS-001..ISS-005) blocking the scope-aware build-gate task. All five resolve to ACCEPTED design decisions with safe degradation — none block implementation, so openIssues is empty. ISS-001 (crate-name vs directory): Cargo's `-p spec`/`--package spec` selects by package NAME, not path (Cargo book, SRC-001), so a member at `crates/foo/` named `bar` makes `cargo -p foo` unresolved; this is safe for stockfan-server where dir==name by convention and is accepted as a known limitation this pass — the robust future fix is `cargo metadata --no-deps --format-version 1` (authoritative member name + manifest_path map), NOT raw `[workspace].members` parsing, because `[workspace].members` lists path globs, not package names. ISS-002 (layout): path-prefix scoping `crates/<pkg>/` means libs/xtask/bin/root-crate diffs yield `[]` → workspace-wide fallback (AC-01); no false scope, just lost scoping benefit — documented limitation. ISS-003 (git context): `git diff --merge-base <commit>` is exactly `git diff $(git merge-base <commit> HEAD)` (SRC-002); an unresolved ref (shallow/disconnected clone, detached HEAD, unborn branch, non-main default) exits non-zero → AC-01's try/catch degrades to `[]` → workspace-wide, and SUPER_DEV_GATE_BASE_REF covers non-main defaults; AC-07a's git-error→[] test pins this. ISS-004 (stderr tail): STDERR_TAIL_LINES=12 (build-runner.ts:147); a failure whose `--> <path>`/`-p <pkg>` marker falls outside the captured tail is unclassifiable → treated IN-SCOPE (conservative), so it never grants a false green but may cause an unnecessary retry/terminate — accepted; a cheap near-term mitigation is bumping the tail (12→~40), full fix is the deferred baseline-diff (#3). ISS-005 (clippy --all-targets): Cargo `-p` (package/workspace-member filter) and `--all-targets` (target-KIND filter = --lib --bins --tests --benches --examples, SRC-001) are orthogonal and order-insensitive, so `cargo clippy -p data --all-targets --quiet` lints every target kind of the `data` package while preserving the required `--all-targets --quiet` flags (AC-02) — confirmed correct. Net: the touched-set + conservative in-scope classifier design is sound and ready to implement; the in-scope verdict (gate.pass || gate.inScopePass) correctly stops the false-abort on pre-existing out-of-scope failures without ever risking a false green.

## Options Considered

### ISS-001 crate-name mapping: dir-name heuristic now, cargo metadata later (CHOSEN)

This pass maps `crates/<pkg>/` -> `-p <pkg>` assuming dir==package-name, which holds for stockfan-server by convention. Verified that `-p` is name-based (SRC-001), so divergence makes `cargo -p <dir>` unresolved rather than silently wrong — i.e. a hard gate failure, not a false pass, so the failure mode is safe-strict. Pros: zero new deps, one git diff, byte-identical backward compat. Con: loses scoping (falls to workspace-wide) or hard-fails the gate if a repo's dir!=name, which re-triggers the very false-abort we're fixing. ACCEPTED for this pass; robust future fix is `spawnSync('cargo', ['metadata','--no-deps','--format-version','1'])` and parse members' `name`+`manifest_path` for an authoritative path->name map — avoid hand-parsing `[workspace].members` since it lists path globs, not names. Note in 10-implementation-summary.md as future hardening.
### ISS-004 classifier posture: conservative default-IN-SCOPE with modest tail bump (CHOSEN)

For a HARD test oracle the asymmetry favors false-fail over false-pass: a false pass ships a regression, a false fail only costs a retry. Defaulting ambiguous/unparseable failures (incl. those truncated outside STDERR_TAIL_LINES=12) to IN-SCOPE, and granting inScopePass only when ALL failures map to untouched crates, guarantees no false green. Pros: cheap, never dangerous. Con: may cause an unnecessary retry/terminate when a real (in-scope) failure's path marker is clipped by the 12-line tail. Mitigation: bump STDERR_TAIL_LINES to ~40 (cheap, still bounded) to widen classification coverage; full layout/format-independence is the deferred baseline-diff (#3). ACCEPTED.
### ISS-003 base ref: git merge-base (default main, env-overridable) + total try/catch (CHOSEN)

`git diff --merge-base <commit>` diffs against the common ancestor of <commit> and HEAD (SRC-002), so a branch that rebased/merged main compares at the right point — more correct than a bare two-dot `main..HEAD`, which drifts/empties after a rebase. An unresolved ref (shallow/disconnected/detached/unborn/non-main) exits non-zero; AC-01's try/catch->[] degrades to workspace-wide and SUPER_DEV_GATE_BASE_REF covers non-main defaults. Pros: rebase-safe, total (never throws), env-escapable. Con: needs `main` (or configured ref) present; covered by the git-error->[] test (AC-07a). ACCEPTED.

### ISS-002 layout coverage: crates/ prefix with graceful [] fallback (CONFIRMED)

Scoping triggers only for `crates/<pkg>/...` diffs; libs/xtask/bin/root-crate edits yield no match -> [] -> workspace-wide (AC-01). This is safe-by-design: no false scope is ever applied, so behavior is byte-identical to today for non-crates diffs, and the in-scope classifier then counts every failure as IN-SCOPE (no false green). Pro: simplest correct scoping that needs no manifest parsing. Con: loses scoping benefit on non-`crates/` layouts. ACCEPTED as documented limitation; a future layout-aware detector could consult `[workspace].members` globs, but that overlaps with the ISS-001 cargo-metadata hardening.
