# Requirements: Scope-Aware Build Gate ‚Äî auto-scope build/test/clippy to touched crates and stop false-aborting on pre-existing out-of-scope failures

- **Date**: 2026-07-20
- **Author**: super-dev:requirements-clarifier
- **Type**: bug-fix
- **Priority**: high
- **Status**: draft

---

## Executive Summary

The super-dev build-gate (`src/build-runner.ts` + `src/stages/implementation.ts`) runs workspace-wide cargo build/test/clippy and then hard-terminates Stage 9 after 3 FAILs. On monorepos with PRE-EXISTING failures in crates the current branch never touched (e.g. stockfan-server spec 54: `reports_test.rs`, `stockfan-compute` clippy lints), this false-aborts Phase 1 while the phase's OWN code is green, so Phases 2‚Äì6 never run. Fix: (1) auto-derive the cargo package set from `git diff --merge-base main --name-only`, (2) scope ALL three gate commands (build + test + clippy) to that set, and (3) add an in-scope failure classification so the retry loop treats "all failures are pre-existing/out-of-scope" as phase-GREEN and only terminates-early on genuine in-scope failures. Backward compatible: empty scope or non-Cargo/non-git repos behave byte-identically to today.

## Acceptance Criteria

- **AC-01**: Add `detectTouchedCargoPackages(cwd, baseRef?): string[]` that spawns `git -C <cwd> diff --merge-base <baseRef> --name-only` (baseRef defaults to `"main"`, overridable via env `SUPER_DEV_GATE_BASE_REF`), maps every stdout line matching `crates/<pkg>/‚Ä¶` to `<pkg>`, dedupes preserving first-seen order (reuse the existing `dedupePreservingOrder` helper), and returns `[]` on ANY git error, empty diff, or a diff containing only non-crate paths. The helper MUST NEVER throw (try/catch the spawnSync). `runBuildGate` auto-derives the scoped set from touched crates when the caller passes no packages and `SUPER_DEV_BUILD_TEST_PACKAGES` is unset.
- **AC-02**: Generalize `scopedCargoTestArgs` into a shared `scopedCargoArgs(subcommand: "build"|"test"|"clippy", packages: string[], extraArgs?: string[]): string[]` emitting `["cargo", subcommand, ...packages.flatMap(p => ["-p", p]), ...(extraArgs ?? [])]`. Keep `scopedCargoTestArgs(packages)` as a thin wrapper returning `scopedCargoArgs("test", packages, ["--quiet"])` (existing call sites in verify.ts/implementation.ts + any tests unchanged), and add `scopedCargoBuildArgs(packages)` (`["cargo","build",‚Ä¶,"--quiet"]` unscoped form) and `scopedCargoClippyArgs(packages)` (keeps `--all-targets --quiet`). Empty `packages` ‚Üí byte-identical workspace-wide argv for all three (`["cargo","build","--quiet"]`, `["cargo","test","--quiet"]`, `["cargo","clippy","--all-targets","--quiet"]`).
- **AC-03**: In `runBuildGate`, when `cmds0.language === "rust"` AND the resolved package set is non-empty, override `build`, `test`, AND `typecheck` on the shallow-copy `cmds` object (today only `test` is overridden). When the set is empty, all three stay workspace-wide and the `cmds` object is byte-identical to current behavior. Package-set precedence (highest‚Üílowest): explicit `opts.testPackages`/`opts.packages` (incl. explicit `[]` = force workspace-wide) ‚Üí `process.env.SUPER_DEV_BUILD_TEST_PACKAGES` ‚Üí `detectTouchedCargoPackages(cwd)` ‚Üí workspace-wide. Detecting touched crates must not run when a higher-precedence source is present.
- **AC-04**: Extend `BuildGateResult` with `outOfScopeErrors: string[]` and `inScopePass: boolean`. After collecting `errors`, classify each error block: extract every source path it references via the `crates/<pkg>/` segment inside `--> <path>` markers AND `-p <pkg>` markers in cargo test-failure lines; an error is OUT-OF-SCOPE iff EVERY referenced crate is NOT in the resolved scoped set, otherwise IN-SCOPE. Compute `inScopePass = !gate.pass && gate.errors.length > 0 && gate.outOfScopeErrors.length === gate.errors.length` (all failures pre-existing/out-of-scope). When `gate.pass` is true, `inScopePass` is true and classification is a no-op. The classifier must NEVER throw; on any ambiguity it treats the error as IN-SCOPE (conservative ‚Äî does not grant false green).
- **AC-05**: In `implementation.ts` retry loop, a phase is GREEN when `gate.pass || gate.inScopePass`. When green via `inScopePass`, log: `Implementation ${phaseId} IN-SCOPE GREEN on attempt ${attempt} ‚Äî ${gate.outOfScopeErrors.length} pre-existing out-of-scope failure(s) ignored (crates: <comma-list>)`. The `if (!green) { ‚Ä¶ "failed after N attempts ‚Äî terminating early"; allGreen=false; break; }` path fires ONLY when NEITHER pass nor inScopePass holds after MAX_ATTEMPTS (genuine in-scope failures). `runBuildGate(setup.worktreePath, { signal: ctx.signal })` call signature unchanged ‚Äî scoping is derived internally.
- **AC-06**: Backward-compatibility / non-regression: non-Cargo repos, non-git dirs, no touched crates, and unset env vars produce IDENTICAL gate argvs and an IDENTICAL `runBuildGate` result to today (modulo the two additive fields), and the new in-scope classification never blocks when there are zero failures or no scoping is active (workspace-wide ‚áí every failure counts as in-scope ‚áí `inScopePass` stays false, preserving current abort semantics). `detectProjectCommands` purity is preserved (scoping applied only on the shallow copy).
- **AC-07**: New co-located unit/integration tests in `src/build-runner.test.ts` (and/or an `implementation` test) covering: (a) `detectTouchedCargoPackages` path‚Üípkg mapping, dedupe, order preservation, git-failure‚Üí`[]`, non-crate-only-diff‚Üí`[]`, base-ref override via env; (b) `scopedCargoBuildArgs`/`scopedCargoTestArgs`/`scopedCargoClippyArgs` argv construction including empty-set‚Üíbyte-identical workspace-wide; (c) in-scope classification with scoped set `{data}` where a failure in `crates/data/‚Ä¶` is IN-SCOPE and a failure in `crates/compute/‚Ä¶` is OUT-OF-SCOPE, plus `inScopePass` true only when ALL failures are out-of-scope; (d) a `runBuildGate` integration that stubs `spawnSync`/git-diff so a touched-`data` crate drives all three commands to carry `-p data`. `npm run typecheck` (tsc --noEmit, strict) and `npm test` (vitest run) ‚Äî existing AND new ‚Äî all pass. No new runtime dependencies; no new spawned processes beyond existing gate commands plus one `git diff --name-only`.

## Non-Functional Requirements

- Performance / cost: Auto-scoping adds at most ONE additional spawnSync per gate run (`git -C <cwd> diff --merge-base <baseRef> --name-only`) and is skipped entirely when a higher-precedence package source is present; on large monorepos it REDUCES wall-time by scoping the three cargo commands. The git-diff helper must use the existing spawnSync pattern with the resolved timeout envelope, never an unbounded shell.
- Robustness / safety: `detectTouchedCargoPackages` and the in-scope classifier MUST NEVER throw ‚Äî any git error, malformed output, or regex ambiguity degrades to workspace-wide behavior / conservative in-scope classification. This preserves the 'never false-abort on green, never false-pass on broken' contract.
- Backward-compatibility / determinism: empty resolved scope ‚áí byte-identical argvs and result to today; the two new `BuildGateResult` fields are additive and no-op when there are no failures or no active scoping. `detectProjectCommands` stays pure (overrides only on the shallow copy).
- Constraint isolation: pure TS change to `src/build-runner.ts`, `src/stages/implementation.ts`, and tests only ‚Äî nodes.ts / workflow.ts / pipeline.ts / render templates / control-flow engine untouched. Target repo never mutated by the gate.
- No new runtime dependencies; no new spawned processes beyond existing gate commands + one `git diff --name-only`.

## Open Questions

- When the resolved scoped set is EMPTY (workspace-wide), should `inScopePass` be forced false so the new field can NEVER grant a green the old code wouldn't? RECOMMENDED: yes ‚Äî treat workspace-wide as 'every failure is in-scope', so `inScopePass` is only ever true when active cargo scoping produced an all-out-of-scope failure set. This keeps non-Cargo and non-scoped runs byte-identical.
- `detectTouchedCargoPackages` reads `git diff --merge-base main` which reflects COMMITTED changes in the worktree. If the implementer created a brand-new crate not yet committed, it won't be scoped. RECOMMENDED: accept this and rely on the per-phase commit step having run before the gate (current implementation.ts commits AFTER green, so the FIRST attempt may run before the new crate is committed) ‚Äî note as a known edge case; fallback in-scope classification still protects against false-abort.
- Base-ref default is literally `"main"`. Some repos default to `master`/`develop`. RECOMMENDED: ship `"main"` as the literal default per spec (env `SUPER_DEV_GATE_BASE_REF` covers the rest) and note default-branch auto-detection as future work; if `main` doesn't exist, `git diff --merge-base` fails and the helper returns `[]` (safe workspace-wide fallback).
- Should the in-scope classification also be surfaced into the log/stderr tail consumed by the implementer's next attempt (so it does NOT retry against pre-existing failures)? RECOMMENDED: yes ‚Äî when `inScopePass` is the reason for green, the retry loop does not run another attempt, so no fix-loop waste; when genuine in-scope failures remain, feed only the IN-SCOPE subset into `attemptErrors` so the implementer fixes real failures, not pre-existing noise.
