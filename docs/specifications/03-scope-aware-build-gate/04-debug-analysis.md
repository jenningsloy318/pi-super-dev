# Debug Analysis: RCA: scope-blind build-gate false-fails then false-aborts on pre-existing out-of-scope failures

- **Date**: 2026-07-19

---

## Summary

On the 2026-07-19 stockfan-server spec-54 run, the timeout fix worked and Phase 1's OWN code was demonstrably green (agents verified 273 stockfan-data + 10 stockfan-api tests via `cargo test -p …`), yet Stage 9 implementation aborted after Phase 1 ("phase-01 failed after 3 attempts — terminating early"), so Phases 2–6 never ran. The build-gate reported FAIL ×3 on three pre-existing, out-of-scope failures: `reports_test.rs` (a different Reports-API feature), `stockfan-compute` clippy lints 18–30 (a different crate's tech debt), and `job_queries_test.rs`. Root cause is two compounding scope-blind defects. (A) `runBuildGate` (src/build-runner.ts:282) scopes ONLY `cmds0.test` at line 301 — `build` and `typecheck` (clippy) stay workspace-wide — and even that `test` scoping is gated on an explicit/env package list that was never set in this run, so all three commands ran workspace-wide `cargo build && cargo test && cargo clippy --all-targets`. (B) The implementation retry loop (src/stages/implementation.ts:43–78) has a binary 3-strikes-terminate: it retries up to MAX_ATTEMPTS=3 feeding `gate.errors` back to the implementer each round, but those errors are pre-existing/out-of-scope — the implementer cannot fix them — so all three attempts repeat-FAIL, then `if (!green) { … break; }` at :74–78 aborts ALL remaining phases. The gate cannot tell "my failures" from "the repo's pre-existing failures," so it false-aborts on green code. The fix set addresses both: (1) `detectTouchedCargoPackages(cwd, baseRef)` auto-derives the scoped crate set from `git diff --merge-base main --name-only`; (2) generalize `scopedCargoTestArgs` → `scopedCargoArgs("build"|"test"|"clippy", …)` so build+test+clippy ALL carry `-p` when the resolved set is non-empty (empty → byte-identical workspace-wide); (3) add `outOfScopeErrors[]` + `inScopePass:boolean` to `BuildGateResult` (classify by `crates/<pkg>/` from `--> <path>` markers and `-p <pkg>` markers; conservative → in-scope on ambiguity); (4) implementation.ts treats `gate.pass || gate.inScopePass` as phase-GREEN and only terminates-early on genuine in-scope failures. Backward-compatible: non-cargo, non-git, no-touched-crates, and unset-env paths produce identical results to today (the two additive fields never block when there are zero failures).

## Hypotheses

- H1 (70%): The gate ran all three commands workspace-wide. No scoping source was active (no `SUPER_DEV_BUILD_TEST_PACKAGES` env, implementation.ts:64 passes no `opts.testPackages`), AND even when scoping IS active commit 97fc4df6 only overrides `cmds.test` (build-runner.ts:301) leaving `build` and `clippy` workspace-wide. Prediction: out-of-scope crates' pre-existing failures (reports_test.rs, stockfan-compute clippy lints, job_queries_test.rs) flow into `gate.errors`; setting SUPER_DEV_BUILD_TEST_PACKAGES=stockfan-data would still leave cargo build and cargo clippy workspace-wide and FAIL. Confirmed by code read: only `cmds.test` is overridden.
- H2 (20%): Even with a touched-crate-scoped gate, the retry loop (implementation.ts:74–78) treats ANY non-pass as hard failure → `break` after 3 attempts → false-abort, because there is no in-scope/out-of-scope distinction. The implementer cannot fix pre-existing out-of-scope failures, so all 3 attempts repeat-FAIL identically. Prediction: a scoped gate that still leaks one out-of-scope failure (e.g. via a dependency compile/clippy ripple) still aborts the run. This is the second-order defect the in-scope classifier must defuse.
- H3 (8%): A genuine in-scope regression the implementer could not fix in 3 attempts. Prediction: re-running Phase 1's own `cargo test -p stockfan-data -p stockfan-api` in isolation FAILS. FALSIFIED by the run log — agents verified 273+10 tests green via package-scoped cargo, so the branch's own code passed.
- H4 (2%): Environmental flake (ENOENT, race, ETIMEDOUT). Prediction: failures are non-deterministic / tied to missing binaries. Largely falsified: the timeout fix produced no ETIMEDOUT, and the three failures are deterministic, named test files / clippy lints, not transient.

## Root Cause

Two compounding scope-blindness defects, neither of which can distinguish the branch's own failures from the repo's pre-existing failures:

(1) OVER-SCOPING OF COMMANDS — src/build-runner.ts `runBuildGate` (line 282) builds the shallow-copy `cmds` by overriding ONLY `cmds0.test` at line 301 (`{ ...cmds0, test: scopedCargoTestArgs(testPackages) }`). `build` and `typecheck` (clippy) are never scoped. Worse, `testPackages` resolves ONLY from `opts.testPackages` or `process.env.SUPER_DEV_BUILD_TEST_PACKAGES` (lines 291–294); there is no auto-derivation from what the branch actually touched. The implementation caller (src/stages/implementation.ts:64) invokes `runBuildGate(setup.worktreePath, { signal: ctx.signal })` with no packages and the env var was unset, so all three commands ran the blunt workspace-wide `cargo build && cargo test && cargo clippy --all-targets`, dragging in the pre-existing `reports_test.rs`, `stockfan-compute` clippy lints, and `job_queries_test.rs` failures that live in crates the branch never modified. Confirmed by grep: no `scopedCargoBuildArgs`/`scopedCargoClippyArgs`/`detectTouchedCargoPackages` exist; only `scopedCargoTestArgs` is defined (line 142) and called once (line 301).

(2) BINARY 3-STRIKES-TERMINATE — src/stages/implementation.ts retry loop (lines 43–78) sets `green=false`, loops `attempt 1..MAX_ATTEMPTS(=3)`, feeds the previous attempt's `gate.errors` back into the implementer prompt each round (line 54), and on each non-pass logs "attempt N/3 FAIL". Because the failures are pre-existing/out-of-scope, the implementer cannot remove them, so all three attempts FAIL identically. Then line 74 `if (!green) { … "failed after N attempts — terminating early"; allGreen=false; break; }` fires: the `break` exits the phase loop and aborts ALL remaining phases (2–6). There is no in-scope classifier and no `inScopePass` field on `BuildGateResult`, so the loop has no way to treat an all-out-of-scope failure as phase-green.

Composition: defect (1) injects out-of-scope failures into `gate.errors`; defect (2) escalates those into a hard abort. The fix mirrors this: (1) auto-scope build+test+clippy to touched crates via `detectTouchedCargoPackages` + a generalized `scopedCargoArgs`, and (2) add `outOfScopeErrors`/`inScopePass` classification so `gate.pass || gate.inScopePass` is treated as GREEN and only genuine in-scope failures terminate early.
