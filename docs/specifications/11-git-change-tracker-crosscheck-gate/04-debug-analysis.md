# Debug Analysis: False-GREEN root cause: implementation phase-green has no assertion tying the agent's claimed file changes to git ground-truth

- **Date**: 2025-07-28

---

## Summary

On current `main` (worktree `11-git-change-tracker-crosscheck-gate`, branch ahead of nothing; only spec docs untracked; `src/tracking.ts` absent; no `change-tracker.jsonl` anywhere), an implementation phase goes GREEN on a VACUOUS file claim. The phase-GREEN decision in `src/stages/implementation.ts:194` is `(gate.pass || gate.inScopePass) && deliverableCheck.pass`. The implementer's claimed `filesModified` (parsed at `implementation.ts:159-160`, accumulated into the summary at `:92/:219`) is advisory-only and NEVER compared to git. `runDeliverableCheck` (`build-runner.ts:1789`) is spec-OPT-IN: when `phase.deliverables` is undefined it early-returns `{pass:true,missing:[]}` (SCENARIO-014 backward-compat), so a phase whose author declared no deliverables has NO assertion against its claimed file changes. The build gate passes trivially because a never-created/unreferenced file compiles clean (`runBuildGate` runs whatever exists). The conjunction of these three gaps is the false-green: a phase claims to create `src/tracking.ts` (or a stockfan `e2e_x.rs`), changes nothing in git, and still passes build + deliverables → GREEN. The summary then lies ("Files Modified: src/tracking.ts"). spec-10 closed this *partially* (opt-in deliverable assertions) but only for phases whose SPEC author declared deliverables; it left the AGENT's self-reported `filesModified` entirely unverified. This analysis confirms the "before" buggy state and defines the change-tracker cross-check gate (AC-01..AC-11) as the second, agent-claim-directional closure.

## Hypotheses

- H1 (70%, CONFIRMED): The implementer's `filesModified` claim is advisory-only and never verified against git ground-truth. Prediction: a phase returning `filesModified:[X]` where git shows no change to X still goes GREEN. Evidence: `implementation.ts:159-160` only ACCUMULATES the array into `filesModified[]`; `:194` phase-GREEN never reads it; `prompts.ts:127` surfaces it in the summary as an unverifiable string. No `git diff`/`status` cross-check exists anywhere in the phase loop.
- H2 (20%, CONFIRMED): The deliverable oracle passes trivially for undeclared deliverables, removing the only existing assertion layer. Prediction: a phase with `deliverables===undefined` and a claimed-but-absent file → `deliverableCheck.pass===true`. Evidence: `implementation.ts:185` calls `runDeliverableCheck(setup.worktreePath, phase.deliverables, …)`; `build-runner.ts:1789` runDeliverableCheck early-returns `{pass:true}` when `deliverables` is undefined — exactly the stockfan Phase-5/6 shape (no deliverables declared, claim a file, never create it).
- H3 (10%, CONFIRMED, contributing condition): The build gate cannot detect a hallucinated file. Prediction: `gate.pass===true` even though file X was never created. Evidence: `runBuildGate` compiles/whatever currently exists; a file the implementer CLAIMS to create but never references compiles fine (no missing-symbol error), so build-green is orthogonal to the claim. Note: H1/H2/H3 are not mutually exclusive — their CONJUNCTION is the root cause; H1 is the dominant missing piece this fix adds (the git ground-truth), H2 is the spec-10 layer it must not regress, H3 explains why build-green alone is insufficient.

## Root Cause

Phase GREEN in `src/stages/implementation.ts:194` — `(gate.pass || gate.inScopePass) && deliverableCheck.pass` — has NO term binding the AGENT's self-reported file changes (`filesModified`, parsed `:159-160`) to git reality, and the only existing assertion (`runDeliverableCheck`) is opt-in by the spec author (undefined `deliverables` → trivial pass at `build-runner.ts:1789`). There is no per-phase git snapshot or delta, so three conditions collapse into one false-green: (1) the implementer claims `filesModified:[X]` but changes nothing in git; (2) the build passes because a non-existent/unreferenced file compiles clean; (3) deliverables pass trivially because none were declared. The claimed set is accumulated (`:92/:159-160/:219`) and echoed into the summary (`prompts.ts:127`) as an unverifiable string — pure advisory telemetry. Result: a phase can claim a deliverable it never produced and still be GREEN (the stockfan Phase-5/6 failure mode and the regression in AC-08). RECOMMENDED FIX (the spec, summarized): add `src/tracking.ts` `ChangeTracker` (begin/end at stage+phase granularity, git `rev-parse HEAD`+`status --porcelain` baseline then `diff --name-status <beginHead>` UNION `status --porcelain` delta, classify A/+untracked/M/D, never-throws → `<specDir>/change-tracker.jsonl`, reusing `spawnSync`+`dedupePreservingOrder` from `build-runner.ts:touchedFilePaths`); bracket ALL stages via an `ctx.events` "stage"/"phase" subscription (`nodes.ts:98/126/127` already emit them — minimal engine touch) and EACH phase in `implementation.ts` (begin before the attempt loop `:~140`, end after with the structured claim); switch `buildImplementPrompt` (`prompts.ts:120`) + review/fix prompt (`:140`) to structured `{filesCreated,filesModified,filesDeleted}` (legacy flat array tolerated); AND a `changeGate` into phase-green at `:194` where `claimedNotChanged` (claimed create/modify that git does NOT show) HARD-FAILS and is fed into the next attempt under `## Claimed changes not present in git — actually create/wire these` via the existing `attemptErrors`/`missingDeliverables` retry channel (respect `MAX_ATTEMPTS=:16`), `changedNotClaimed` is advisory-only, and git-unavailable → `changeGate.pass=true` (never block on infra); union `claimed.filesCreated` into deliverable `requireFiles` before `runDeliverableCheck` (`:185`) so spec-10 and tracking reinforce rather than diverge; thread the tracker via the `activeRun`/`setActiveRun`/`getActiveRun` module-singleton pattern (`extension.ts:68/131/148`) set from `state.setup` (specDirectory+worktreePath) in execute()'s entry and cleared in its finally (`extension.ts:391/446`). Theme methods stay method-style (no destructuring) to keep `tests/stream-theme-class-theme.test.ts` + `tests/render/real-theme-parity.test.ts` green.
