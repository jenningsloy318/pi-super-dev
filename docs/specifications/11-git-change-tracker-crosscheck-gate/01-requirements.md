# Requirements: Git-snapshot change-tracker with per-stage/per-phase bracketing and a claimed-vs-actual cross-check gate

- **Date**: 2026-07-21
- **Author**: super-dev:requirements-clarifier
- **Type**: enhancement
- **Priority**: high
- **Status**: draft

---

## Executive Summary

Replace the advisory flat `filesModified[]` with a structured `{filesCreated,filesModified,filesDeleted}` set and a new `src/tracking.ts` ChangeTracker that brackets EVERY stage (start+end) and EVERY phase (start+end) with a git snapshot, persisting an append-only `<specDir>/change-tracker.jsonl`. A git cross-check gate is AND-ed into phase-green (mirroring spec-10's deliverable AND): a phase that claims to create/modify a file git does not show changed hard-fails, closing the false-green root cause a second way. Tracking and spec-10 deliverable assertions collapse into one enforcement path (claimed.filesCreated auto-populates requireFiles); the whole subsystem never throws and degrades when git is unavailable.

## Acceptance Criteria

- **AC-01**: New `src/tracking.ts` exports a `ChangeTracker` constructed with `(specDir, worktreePath)`. `begin(unit: "stage"|"phase", id)` snapshots `git -C <wt> rev-parse HEAD` + `git -C <wt> status --porcelain` as the baseline; `end(unit, id, claimed?: {filesCreated,filesModified,filesDeleted})` re-snapshots and computes the DELTA as `git -C <wt> diff --name-status <beginHead>` (committed) UNION `git -C <wt> status --porcelain` (uncommitted/untracked), classifying into `gitActual:{created(A/+untracked), modified(M), deleted(D)}`, then CROSS-CHECKS vs `claimed` into `{claimedNotChanged[], changedNotClaimed[]}` and appends ONE jsonl line to `<specDir>/change-tracker.jsonl`. Reuses the existing `spawnSync` + `dedupePreservingOrder` git-union patterns from `src/build-runner.ts` (`touchedFilePaths`); no new runtime deps.
- **AC-02**: ChangeTracker is NEVER-throws: every git op is try/caught; on git failure it records `{gitUnavailable:true}` and returns a non-blocking `ChangeRecord` (verdict continues), so infrastructure absence never aborts a run. Conservative parse: a claimed file counts as `claimedNotChanged` only when git clearly shows no change.
- **AC-03**: Unit tests (mocked git) cover: created/modified/deleted classification; `claimedNotChanged` vs `changedNotClaimed`; git-unavailable → record-and-continue + non-blocking; begin/end bracketing emits start+end records; append-only jsonl (multiple `end` events → multiple lines, nothing overwritten).
- **AC-04**: ALL stages are bracketed (`tracker.begin("stage", stageId)` / `tracker.end("stage", stageId, claimed?)`) and implementation phases are ADDITIONALLY bracketed (`tracker.begin("phase", phaseId)` before the attempt loop / `tracker.end("phase", phaseId, claimedFromImplementer)` after). Implemented via a minimal-touch seam — PREFERRED an `ctx.events` "stage"/"phase" subscription or a runner-level wrapper (events already emitted in `src/nodes.ts` lines 98/126/127) over editing nodes.ts/workflow.ts/pipeline.ts internals. A synthetic-pipeline test asserts change-tracker.jsonl contains stage-start/stage-end for every stage AND phase-start/phase-end for the implementation stage, in correct nested order.
- **AC-05**: Tracker is threaded via ONE consistent mechanism mirroring `activeRun`/`setActiveRun`/`getActiveRun` in `src/extension.ts` (module-level per-run singleton set on execute() entry from `state.setup` providing `specDir`+`worktreePath`, cleared in the same finally). No leak between runs.
- **AC-06**: `buildImplementPrompt` (src/prompts.ts ~line 120) and the review/fix prompt (~line 140) change their output contract from `filesModified (array)` to `filesCreated (array), filesModified (array), filesDeleted (array)` with the instruction: "Report the exact files you created/modified/deleted — these are git-cross-checked; claiming a file you didn't change fails the phase." implementation.ts (~line 159) parses the structured set but STILL accepts a legacy flat `filesModified` array (backward-tolerant).
- **AC-07**: A `changeGate` is computed in `src/stages/implementation.ts` after `runBuildGate` (~line 165) and `runDeliverableCheck` (~line 185). `changeGate.pass === false` iff `claimedNotChanged` is non-empty (a created/modified claim git does NOT show) — the false-green killer; `changedNotClaimed` is advisory-only (logged, never fails). Phase GREEN (line 194) becomes `(gate.pass || gate.inScopePass) && deliverableCheck.pass && changeGate.pass`. A `claimedNotChanged` miss is fed into the next implementer attempt under a `## Claimed changes not present in git — actually create/wire these` section via the existing attemptErrors/missingDeliverables retry channel, respecting `MAX_ATTEMPTS`. Never throws; git-unavailable → `changeGate.pass = true`.
- **AC-08**: Regression test (mirroring `tests/implementation.test.ts` / `build-runner-deliverable-check.test.ts` patterns): a phase returning `claimed:{filesCreated:[X]}` where git shows NO new file → `changeGate.pass === false` → phase is NOT green, even though the build gate and deliverable check pass. The stockfan Phase-5/6 false-green (claimed a file that was never created/wired) becomes impossible.
- **AC-09**: Spec-10 bridge: before `runDeliverableCheck` is called, `claimed.filesCreated` is UNIONed with any spec-declared `deliverables.requireFiles`, so a file a phase claims to have created MUST exist — tracking and deliverable assertions reinforce rather than diverge.
- **AC-10**: The phase end-record's `gitActual` is surfaced as evidence: a `📝 N files changed (C/M/D)` line in the implementation summary, and the flat `filesModified` accumulation (implementation.ts lines 92/159/219) is derived from the structured set so the existing summary writer (prompts.ts line 127) keeps working.
- **AC-11**: `npm run typecheck` (tsc --noEmit) is clean and `npm test` (vitest run) is ALL green — existing suite plus new tracker/cross-check/structured-changes tests — with NO regression to runRedCheck, runDeliverableCheck, npm in-scope classification, scope-aware cargo gate, themed stream, mid-run input injection, dashboard, or real-theme parity (`tests/stream-theme-class-theme.test.ts`, `tests/render/real-theme-parity.test.ts` stay green). pi Theme methods are called method-style, NEVER destructured.

## Non-Functional Requirements

- Never-throw invariant: the tracker, all git ops, the cross-check, and the changeGate must degrade gracefully — git unavailable → record `{gitUnavailable:true}` and continue; parse ambiguity → conservative (treat claimed as not-changed only when git clearly shows no change); the subsystem sits in the hot path of EVERY run and must never abort or widen a phase's GREEN.
- Performance: git snapshots are bounded `spawnSync` calls (`rev-parse`, `status --porcelain`, `diff --name-status`) under the existing `resolveTimeoutMs` envelope; one append-only jsonl record per begin/end event (no rewrites, no cross-phase caching that could go stale); N stages × M phases yields bounded, linear I/O. No new runtime dependencies.
- Backward compatibility: legacy flat `filesModified` arrays from older agents are accepted; non-git worktrees are recorded as `gitUnavailable` and unblock (`changeGate.pass = true`); phases with no claimed changes trivially pass; no changes to backend selection (subprocess/session) or control-flow (branch/parallel/loop) semantics.
- Security/hygiene: the tracker writes only into `specDir`; git is spawned with discrete argv (`-C <wt> ...`, consistent with build-runner.ts Gap-4 hygiene) so file paths from agents cannot inject shell; the cross-check is one-directional (the AGENT claims, the framework verifies) — the framework never auto-generates claimed changes from git.

## Open Questions

- Threading mechanism: confirm the module-level singleton mirroring `activeRun` (preferred, minimal touch) over a RunOptions/StageContext field — and verify behavior if two super-dev runs ever overlap in one process (today runs are serialized, but confirm the singleton is cleared in the execute() finally).
- Record granularity: should `begin` events be persisted as `{event:"start"}` lines too (complete bracket trace) or only `end` records (smaller file)? The spec recommends start+end; confirm acceptable jsonl size for very long multi-phase runs.
- Cross-check scope vs spec-declared deliverables: the spec says claimed-changes is one-directional enforcement — confirm that UNIONing `claimed.filesCreated` into `requireFiles` does not create a circular double-count (an agent that omits a required file the spec already declared is still caught by the spec-declared requireFiles).
- Dashboard surfacing: is the `📝 N files changed (C/M/D)` line sufficient in the implementation summary, or does the dashboard also need a dedicated change-tracker panel consuming change-tracker.jsonl? (Listed as a future surfacing concern; out of scope for this cut unless confirmed.)
