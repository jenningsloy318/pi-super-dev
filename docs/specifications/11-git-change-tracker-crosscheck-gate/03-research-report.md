## Options Considered

### A. Change-set extraction: new classified extractor (recommended) vs reuse touchedFilePaths

Reuse `touchedFilePaths` (SRC-002) would be DRY but it returns flat string[] via `diff --name-only` + `ls-files --others` and INTENTIONALLY strips the A/M/D status letter, so it cannot feed the created/modified/deleted split or the cross-check. RECOMMENDED: a sibling `classifyChangedFiles(cwd, beginHead)` in tracking.ts using `diff --name-status <beginHead>` UNION `status --porcelain` (A/?? → created, M → modified, D → deleted), reusing the same two-bounded-spawns + `dedupePreservingOrder` (SRC-001, one-line export fix) never-throw envelope (BP-001/BP-002). Tradeoff: ~40 lines of near-duplicate git plumbing; justified because classification is the tracker's core deliverable and --name-status/--porcelain semantics (status letters, R rename detection, copy markers) differ from --name-only. Keep argv discrete (no shell:true).
### B. Tracker threading: module-level singleton mirroring activeRun (recommended) vs RunOptions/StageContext vs ctx.events-only

The activeRun/setActiveRun/getActiveRun triplet in extension.ts (SRC-004) is the proven per-run-singleton pattern — set on execute() entry from state.setup (already carries specDir+worktreePath), cleared in the same finally (:446). RECOMMENDED mirroring it: setActiveTracker/getActiveTracker set where setActiveRun is already called, cleared alongside it — zero inter-run leak, zero RunOptions signature churn, and the implementation.ts phase seam reads it without threading a new arg through every stage.run(state,ctx). Tradeoff vs ctx.events-only: events can capture STAGE enter/exit (SRC-003) but CANNOT carry the implementer's `claimed` set (lives in impl.control inside the phase loop), so a pure-event design still needs a direct call in implementation.ts — hence the hybrid: events for stages, direct calls for phases. RunOptions threading is the most explicit but touches the most call sites; overkill given the singleton already works for mid-run input.
### C. Stage-boundary capture: ctx.events subscription (recommended) vs runner-level wrapper vs nodes.ts edit

src/nodes.ts already emits `ctx.events.emit("stage", {id,label,status:"running"})` at :127 (enter) and `ctx.events.emit("stage", {id,label,status,error})` at :98 (exit, in the task-node closure, fired on both pass and fail). RECOMMENDED a thin tracker.onStageEvent subscriber wired once at execute() entry: status==="running" → begin("stage",id), else → end("stage",id) (claimed undefined for non-implementation stages — only gitActual evidence recorded). Fully decoupled from nodes.ts/workflow.ts/pipeline.ts internals (minimal-touch, satisfies AC-04's PREFERRED path, BP-005). Tradeoff: the event payload has no `claimed` field, so stage-level cross-check is unavailable — acceptable because the cross-check GATE is phase-scoped. A runner-level wrapper around stage.run would carry claimed but requires editing nodes.ts; reject unless events prove insufficient. Phase bracketing is NOT event-driven (the :126 phase emit is just a label marker) — two direct calls in implementation.ts: begin("phase",phaseId) before the for-attempt loop, end("phase",phaseId, claimed) after.
### D. Cross-check strictness: asymmetric hard/advisory (recommended, spec-faithful) vs symmetric vs advisory-only

RECOMMENDED the spec's asymmetric rule (BP-003): changeGate.pass===false iff claimedNotChanged non-empty (a created/modified file git shows unchanged = vacuous claim = the Phase-5/6 false-green root cause), while changedNotClaimed (git edits the agent didn't report) is LOGGED only. Rationale: claiming-done-but-undone is a correctness defect (false green); unreported edits are under-reporting (advisory, common with format-on-save). Per-bucket matching (ISS-03, RESOLVED) is deliberately NOT used — a file claimed `created` but git shows `modified` is NOT a failure (the change exists; classification fuzziness from rename/copy would cause false alarms); only claimed-and-totally-absent fails. Tradeoff vs symmetric hard-fail: a stricter gate risks flapping on R/C rename codes — mitigate by normalizing rename-targets (R99\told\tnew → include new) before the union. Tradeoff vs advisory-only: loses the false-green killer, the entire point of the feature. Feed claimedNotChanged into implParts under `## Claimed changes not present in git — actually create/wire these`, reusing the exact missingDeliverables→implParts retry shape (SRC-005, lines 153-154), respecting MAX_ATTEMPTS.
