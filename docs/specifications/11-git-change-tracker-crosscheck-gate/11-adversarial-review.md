# Adversarial Review: Adversarial Review — spec-11 Git Change-Tracker & Cross-Check Gate

- **Date**: 2025-07-21
- **Reviewer**: super-dev:adversarial-reviewer
- **Verdict**: REJECT

---

The feature is fully implemented and unit-tested, typecheck is clean, and the design is sound — but the headline requirement is dead in production wiring. src/stages/setup.ts imports `ChangeTracker` + `setActiveTracker`, carries a 7-line comment describing "install the per-run ChangeTracker singleton the instant the setup's worktreePath + specDirectory are finalized", then falls straight through to `abbreviatePath` and `return setup` WITHOUT ever calling `setActiveTracker(new ChangeTracker(...))`. A repo-wide grep confirms `setActiveTracker` is invoked with a non-null instance nowhere (only the teardown `setActiveTracker(null)` exists). Consequently `getActiveTracker()` is permanently `null` during a run, which means: (a) the workflow.ts stage subscription's `if (tracker && stage.id)` guard is always false → no stage-start/stage-end records; (b) the implementation.ts `const tracker = getActiveTracker(); if (tracker) tracker.begin("phase", ...)` is always skipped → no phase bracketing, no phase end-record, no crossCheck; (c) `computeChangeGate(tracker?.getRecord(...) ?? null)` always receives `null` → always `{pass:true}`. The false-green-killer gate (the whole point of the task) is inert; AC-2 (stage+phase bracketing), AC-4 (changeGate kills false-green), and AC-5 are unmet at the integration level even though the unit tests (which seed their own tracker) pass green. This is a textbook instance of the project's own learned.md score:62 anti-pattern ("an implemented-and-unit-tested-but-NOT-wired-in helper survives every review"). This alone forces REJECT. Secondary latent issues (path normalization, dangling begins on budget abort, unbounded jsonl) are also flagged but are moot until the wiring exists. Fix is one line in setup.ts: `setActiveTracker(new ChangeTracker(setup.specDirectory, setup.worktreePath))`.

### F-01: setActiveTracker never called with a real tracker — the entire feature is unwired dead code

- **Severity**: high
- **Lens**: Skeptic
src/stages/setup.ts (around lines 11 + 31-37): the module imports `{ ChangeTracker, setActiveTracker }` and contains a multi-line comment stating it installs the per-run singleton 'right after the setup stage populates them', but the install call is ABSENT — execution goes straight to `const relWorktree = abbreviatePath(...)` and `return setup`. A repo-wide grep (`rg setActiveTracker src/`) shows the symbol is assigned a non-null ChangeTracker nowhere; only `setActiveTracker(null)` in src/extension.ts:460 teardown. Result: getActiveTracker() is always null during a run, so src/workflow.ts:174 stage bracketing, src/stages/implementation.ts:143 phase begin/end, and the changeGate (implementation.ts ~272 `tracker?.getRecord(...) ?? null` → always null → computeChangeGate returns {pass:true}) are all inert. This voids AC-2, AC-4, AC-5 at the integration level — the false-green killer catches nothing. The unit/integration tests pass only because tests/tracker-bracketing.test.ts and tests/implementation-crosscheck-gate.test.ts seed their own tracker. Fix: add `setActiveTracker(new ChangeTracker(setup.specDirectory, setup.worktreePath));` before `return setup` in setup.ts. This is the project's own score:62 anti-pattern recurring.
### F-02: Cross-check uses raw string comparison — agent-reported path formats will trigger false claimedNotChanged failures

- **Severity**: medium
- **Lens**: Skeptic
src/tracking.ts computeCrossCheck() compares `claimed.filesCreated/filesModified` strings against git's repo-relative paths via plain Set membership (`gitCreatedOrModified.has(p)`). No path.normalize / relative-to-worktree / leading-`./` or absolute-path normalization is applied. git diff --name-status and git status --porcelain always emit repo-root-relative POSIX paths, but the implementer is an LLM that commonly reports absolute paths, worktree-rooted paths, or paths with a stray leading './'. Any mismatch → the claimed path lands in claimedNotChanged → changeGate.pass===false → wasted retry, and on persistent misreporting, MAX_ATTEMPTS exhaustion → phase RED even though the work was done. The spec's 'be conservative to avoid false alarms' guidance was only applied to the changedNotClaimed (advisory) direction; the gated direction is brittle. Mitigation: normalize both sides (path.relative(worktreePath, p) + POSIX + strip leading ./) before comparison, OR emit the expected path format in GIT_CROSSCHECK_WARNING.
### F-03: Budget-abort early-return leaves a dangling phase begin (no matching phase-end)

- **Severity**: low
- **Lens**: Skeptic
src/stages/implementation.ts: `tracker.begin('phase', phaseId)` runs once before the attempt loop; inside the loop `if (!ctx.budget.check()) { allGreen=false; return {...} }` returns BEFORE any `tracker.end('phase', ...)`. The change-tracker.jsonl then contains a phase-start with no phase-end for that phase (bracketing asymmetry), and the stale baseline stays in the tracker Map for the rest of the (still-running) run. Also note: a skipped stage (nodes.ts may emit a terminal status without a preceding 'running') triggers `tracker.end('stage', id)` with no prior begin, computing a gitActual from the current porcelain that can misattribute changes to a stage that never ran. Both are advisory-noise only (stages aren't gated), but they violate the 'clean start/end pair per unit' contract. Guard end() calls behind a baselines.has(key) check or emit begin on skip too.
### F-04: Phase bracketing deviates from spec text: begin once, end per-attempt

- **Severity**: low
- **Lens**: Architect
The spec/plan states 'tracker.begin("phase", phaseId) before the attempt loop and tracker.end("phase", phaseId, claimed) after it' (one bracket per phase). The implementation calls begin ONCE before the loop but end INSIDE the loop (once per attempt). This yields a jsonl trace of phase-start, phase-end(a1), phase-end(a2), ... — not a clean nested bracket. It is arguably BETTER (each attempt gets its own crossCheck, which is what makes the retry mechanism work, and endRecords keeps the last), but it diverges from the documented shape, the SCENARIO-008/009 'phase-start/phase-end pair' contract, and the bracketing test's stated nesting. Decide deliberately: either document the per-attempt-end semantics in the record (e.g. add `attempt` to the end record) or move end outside the loop and re-read the crossCheck from the prior attempt. Either way, update the prose to match the code.
### F-05: change-tracker.jsonl grows unbounded across runs/resumes in the same spec dir

- **Severity**: low
- **Lens**: Minimalist
src/tracking.ts appendRecord() is strictly append-only and setActiveTracker(null) only nulls the in-memory singleton — the durable file `<specDir>/change-tracker.jsonl` is never rotated, truncated, or run-prefixed. For a long-lived spec directory (many runs, retries, and especially RESUME which reuses the spec id), the file accumulates every begin/end from every historical run with no run boundary marker, making it hard to attribute records to a single run and unbounded in size. The 'durable per-run tracking file' wording implies a per-run section. Suggestion: prefix each run with a `{event:'run-start', runId, ts}` line at construction, or truncate on first begin of a run. Low impact (advisory artifact) but worth a delimiter.
### F-06: computeChangeGate lives in build-runner.ts but duck-types tracking.ts's ChangeRecord via `unknown`

- **Severity**: low
- **Lens**: Minimalist
computeChangeGate was placed in src/build-runner.ts 'for co-location with the other gates' but its input is typed `unknown` and it duck-reads `{gitUnavailable, crossCheck:{claimedNotChanged}}` — the exact shape of tracking.ts's ChangeRecord. This hides the real type dependency (build-runner now 'knows' about tracking's record shape through a string-keyed cast) and lets a caller pass any object. Co-location rationale is weak since the function is pure and owns no build-runner state. Moving it to src/tracking.ts (next to ChangeRecord and ChangeGateResult) would restore a single source of truth for the record shape and let the param be typed `ChangeRecord | null`. Cosmetic, but reduces implicit cross-module coupling.
