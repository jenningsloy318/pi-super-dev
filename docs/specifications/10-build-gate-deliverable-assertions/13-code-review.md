## Verdict: Blocked

The implementation is architecturally sound and the never-throwing `runDeliverableCheck` primitive (Layer 1) is well-built and well-tested — the AND-semantics logic, missing-list injection into the next implementer retry, `skipTests`-when-build-failed optimization, per-attempt cache reset, schema/normalizer/prompt elicitation (Layer 2), and backward-compat no-op path are all correct in logic. HOWEVER the change cannot be approved because it FAILS the two hard acceptance gates it explicitly owes (AC-07): `npm run typecheck` exits 2 with `TS2304: Cannot find name 'resetDeliverableCheckCache'`, and `tests/implementation-deliverable-wiring.test.ts` fails all 10 cases with a runtime `ReferenceError: resetDeliverableCheckCache is not defined`. The root cause is a one-line miss: `implementation.ts` calls `resetDeliverableCheckCache()` (line 180) but the import on line 14 omits it, AND the `vi.mock("../src/build-runner.ts")` factories in BOTH Phase-3 wiring test files only export `runRedCheck`/`runBuildGate`/`runDeliverableCheck` — so the function is undefined at runtime under the mock. This is a trivial fix (3 spots) but it is a build-breaker that falsifies the implementation summary's "typecheck + tests green" claim, so the verdict is Blocked. Secondary findings: the `requireTests` tolerant substring match is matched against the ENTIRE raw test-list stdout (cargo headers, file paths, vitest JSON), which risks false-positive matches (i.e. reintroducing the false-green class this feature exists to kill); there is no dedicated end-to-end stockfan regression test exercising the REAL checker (AC-06 is covered only via mocked-checker verdict composition + the primitive unit tests); and the spec prompt emits a redundant top-level `deliverables` bullet that may elicit a misplaced field. Dimension scores — Correctness 2/5 (red build/tests), Security 4/5, Performance 4/5, Concurrency 4/5, Maintainability 4/5, Testability 3/5, Error Handling 5/5, Data Integrity 4/5, Observability 4/5.

## Findings

### F-01: resetDeliverableCheckCache is called but never imported — typecheck fails (TS2304) and all 10 Phase-3 wiring tests throw ReferenceError

- **Severity**: Critical
- **File**: `src/stages/implementation.ts`
- **Line**: 180
AC-07 requires `npm run typecheck` strict-clean and `npm test` green; both are RED. `src/stages/implementation.ts:180` calls `resetDeliverableCheckCache()` (added for the per-attempt test-list cache reset), but the import at `implementation.ts:14` is `{ runBuildGate, runDeliverableCheck, runRedCheck, type GateOptions, type RedStatus }` — `resetDeliverableCheckCache` is missing. Result: `npx tsc --noEmit` => `src/stages/implementation.ts(180,5): error TS2304: Cannot find name 'resetDeliverableCheckCache'` (exit 2). At runtime the wiring tests also crash: `tests/implementation-deliverable-wiring.test.ts` fails 10/10 with `ReferenceError: resetDeliverableCheckCache is not defined` at `implementation.ts:180:5`, because the `vi.mock("../src/build-runner.ts")` factory only exports runRedCheck/runBuildGate/runDeliverableCheck and does not supply `resetDeliverableCheckCache`. This falsifies the implementation summary's claim that typecheck + tests are green, and means the core AND-semantics wiring (AC-03) is UNVERIFIED — every wiring scenario (SCENARIO-011..015) errors before asserting. Failure scenario: CI/pipeline gate rejects the PR; if forced through, every multi-attempt implementation phase crashes on the first deliverable-check attempt. Suggested fix: (1) `implementation.ts:14` — add `resetDeliverableCheckCache` to the named imports from `../build-runner.ts`; (2) in BOTH `tests/implementation-deliverable-wiring.test.ts` and `tests/implementation-deliverable-wiring-edges.test.ts` add `resetDeliverableCheckCache: () => {}` (or `vi.fn()`) to the `vi.mock` factory object. Confidence 1.0 — reproduced locally (tsc exit 2; vitest 10/10 fail).
### F-02: requireTests tolerant substring match against whole raw test-list stdout risks false-positive (false-green) matches

- **Severity**: Medium
- **File**: `src/build-runner.ts`
- **Line**: ~1730 (runDeliverableCheck requireTests loop)
`tolerantMatch(name, list.list)` matches each declared `requireTests` name as a substring (or regex) against the ENTIRE raw stdout of the test lister. For cargo `test -- --list` that stdout includes binary headers (`Running target/debug/deps/<crate>-<hash>`), file paths, module prefixes, and `[ignored]` tags; for `vitest list --json` it is the full JSON including `file`/`path` fields. A declared name like `"screen"` would PASS if ANY test anywhere (even an unrelated module) contains that substring, or if the string appears in a path/header. This is the dangerous direction for a deliverability gate: a false-positive requireTests match yields a false-green phase — precisely the class of bug this feature is built to eliminate (the stockfan false-green). The substring-OR-regex rule is documented as tolerant-by-design, but nothing anchors the match to a real test-name token. Failure scenario: spec author declares `requireTests:["screen_fallback"]`, the implementer never adds it, but the list already contains `screen_fallback_init` or a path containing `screen_fallback` → phase goes GREEN without the declared test. Suggested fix: for the vitest path, parse the JSON and match against `name`/`fullName` fields only; for cargo, match against lines trimmed of the ` ... [ignored]` suffix and require the match to be a full module-path token (e.g. split on `::`/whitespace and require an element equal to or ending with the declared name). At minimum, prominently document that declared names should be full test identifiers, not fragments.
### F-03: AC-06 stockfan regression test exercises a MOCKED checker, not the real runDeliverableCheck against temp files

- **Severity**: Medium
- **File**: `tests/implementation-deliverable-wiring.test.ts`
- **Line**: DELIVERABLE_FAIL fixture / SCENARIO-011
AC-04/AC-06 asks for a regression test 'simulating the stockfan Phase-5/6 false-green' proving `deliverableCheck.pass===false` with deliverables ABSENT even given a green stub build-gate, AND `pass===true` when present. The actual coverage splits the proof: `build-runner-deliverable-check.test.ts` proves the PRIMITIVE catches missing files/patterns/tests, and `implementation-deliverable-wiring.test.ts` proves the VERDICT COMPOSITION (`(gate.pass||gate.inScopePass) && deliverableCheck.pass`) by feeding a CANNED `DELIVERABLE_FAIL` through a fully-mocked `runDeliverableCheck`. Neither test runs the REAL checker against the exact stockfan shape (requireFiles:[X] + requireContains:[screen.rs→fetch_us_data] + requireNotContains:[screen.rs→with_retry\(\|\| fetch_fmp] + requireTests:[T]) in both the absent and present directions. So the integration that 'wires the contract end-to-end through the real checker' is asserted only by composition, not by execution. Failure scenario: a future refactor of runDeliverableCheck's file/regex semantics could pass both suites while silently mis-detecting the stockfan shape. Suggested fix: add `tests/implementation-deliverable-regression.test.ts` that, against a real temp worktree, calls `runDeliverableCheck` with the exact stockfan contract in absent-then-present states (this also survives F-01's mock-layer gap).
### F-04: skipTests:true suppresses requireTests from the missing-list across build-failing retries

- **Severity**: Low
- **File**: `src/stages/implementation.ts`
- **Line**: 185 (skipTests: !buildGreen)
When the build gate is not green, `skipTests: !buildGreen` skips the requireTests sub-check entirely (intended to avoid spawning the test-lister against a broken build and poisoning the cache — a good optimization). A side effect: `missingDeliverables` will NEVER contain a `missing test: <name>` entry while the build is failing, so the implementer retry prompt's `## Deliverables still missing` block never surfaces the test requirement until the build passes. If the build fails for all MAX_ATTEMPTS(3), the phase fails anyway on the build error (no false-green), so this is acceptable behavior — but it can mask the true root cause (a missing named test) in the audit log and can consume an extra attempt. Suggested fix: none required; add a one-line note in the existing comment that requireTests reasons are deferred until build-green so the behavior is explicit.
### F-05: Spec prompt emits a redundant top-level 'deliverables' bullet that may elicit a misplaced field

- **Severity**: Low
- **File**: `src/prompts.ts`
- **Line**: buildSpecPrompt data-to-return
In `buildSpecPrompt` the `phases` bullet now reads `{ name, description, deliverables? }` (correct — deliverables is per-phase), but a SEPARATE top-level bullet `- deliverables (optional, per phase): ...` is also added to the 'Data to return' list. An LLM could interpret the second bullet as a sibling top-level key (i.e. emit `deliverables` alongside `phases` rather than inside each phase). The typebox schema only accepts `phases[].deliverables`, so a misplaced top-level `deliverables` would be dropped/ignored on validation depending on `additionalProperties` handling — silently losing the contract rather than failing loudly. Suggested fix: drop the standalone top-level bullet and fold its instruction into the `phases` bullet (e.g. `phases: array of { name, description, deliverables? } where deliverables = {requireFiles, requireContains, requireNotContains, requireTests} declared when a phase's deliverable is not compiler-checkable ...`), or rename the bullet to `phases[].deliverables (optional) ...` to remove ambiguity.
