# Adversarial Review: Adversarial Review — Per-Phase Deliverable Assertions (build-gate AND-semantics)

- **Date**: 2026-07-21
- **Reviewer**: super-dev:adversarial-reviewer
- **Verdict**: CONTEST

---

The implementation correctly delivers the stated intent: `runDeliverableCheck(cwd, deliverables, opts)` enforces requireFiles/requireContains/requireNotContains/requireTests with tolerant matching, never throws (entire body wrapped in try/catch, SCENARIO-010), and the GREEN condition in implementation.ts:194 is exactly `(gate.pass || gate.inScopePass) && deliverableCheck.pass` — the proven root-cause fix for the stockfan false-green. Backward compatibility holds (absent deliverables → `{pass:true}` early return), the spec prompt elicits `phases[].deliverables`, the schema/normalizer round-trip the field as a typed `phase.deliverables`, missing deliverables are fed into the next attempt under the required heading, and `skipTests:!buildGreen` avoids spawning a test-lister on a broken build. `npm run typecheck` is strict-clean and all 1173 tests pass (53 new). The never-throwing invariant, AND-semantics, and the regression test (SCENARIO-011 + the stockfan-shaped wiring tests) are sound.

However, three medium-severity quality concerns warrant an author response before this is considered production-hardened: (1) the test-list cache is architecturally dead weight — ~60 lines of machinery (a process-global Map, a exported reset hook, a sentinel type, resolve()-key normalization, an entire SCENARIO-009 dedupe test) that can never serve a hit because the only caller resets it before every invocation and `loadTestList` runs exactly once per invocation; (2) the MAX_ATTEMPTS=3 budget is shared between build-fix and deliverable-fix retries, so a phase that burns two attempts on the build has only one left to satisfy its deliverable contract — premature termination while still making progress; (3) deliverable paths are resolved with bare `resolve(cwd, p)`, accepting `../` escapes that let a contract be satisfied by a file outside the worktree. None of these risk production failure, data loss, or a security breach (the feature is fail-closed and read-only), so this is CONTEST, not REJECT.

### ADV-01: Dead test-list cache — ~60 lines of complexity that can never serve a hit

- **Severity**: medium
- **Lens**: Minimalist
src/build-runner.ts:1639-1664 defines `testListCache` (a process-global `Map<string, TestListResult>`), `resetDeliverableCheckCache()`, the `TestListResult` sentinel type, and `resolve()`-based key normalization, plus ~40 lines of rationale ('SCENARIO-009 dedupe', 'run-boundary reset'). This machinery provides ZERO observable benefit: (a) `loadTestList` (build-runner.ts:1899 region) is called exactly ONCE per `runDeliverableCheck` invocation, so there is nothing to dedup within a call; (b) the only production caller, implementation.ts:180-185, calls `resetDeliverableCheckCache()` immediately BEFORE every `runDeliverableCheck`, so the cache is always cold when populated and is wiped before any cross-call read. The SCENARIO-009 test ('two requireTests-bearing phases sharing a cwd share one spawn') asserts behavior that the production wiring immediately defeats. The cache's comments even claim it 'dedupes a single runDeliverableCheck call's sub-checks' — vacuously true because there is only one sub-check call. Recommendation: delete `testListCache`, `resetDeliverableCheckCache`, the `TestListResult` sentinel, the `resolve()` keying, and the SCENARIO-009 cross-call dedupe test; let `loadTestList` return its result as a plain local. The reset-before-every-call is itself correct (a retry adds a test, so a stale cache would false-negative) — which is precisely why the cache can never safely dedup and should not exist. This is the single biggest over-engineering surface in the change.
### ADV-02: Shared MAX_ATTEMPTS budget between build-fix and deliverable-fix retries causes premature termination

- **Severity**: medium
- **Lens**: Architect
src/stages/implementation.ts:103 reuses the single `MAX_ATTEMPTS=3` loop (line 16) for BOTH build-gate failures AND deliverable-contract failures. A phase that needs two attempts to get the build green has only ONE attempt left to satisfy `requireFiles`/`requireContains`/`requireTests` — and if the implementer's first deliverable-bearing attempt still misses (e.g. creates the file but mis-wires the call site), the phase reports `failed after 3 attempts — terminating early` (implementation.ts:206) and aborts the entire run (`allGreen=false; break`) even though it was demonstrably converging. The task spec says 'MAX_ATTEMPTS respected' so this is compliant, but the coupling of two independent failure modes into one budget is a robustness regression vs. the pre-deliverable world, where 3 attempts were spent entirely on build/test. Recommendation: either give deliverable-fix attempts a separate small budget (e.g. deliverable failures don't count against build-fix attempts, or a `MAX_DELIVERABLE_ATTEMPTS` constant), or reset the attempt counter when the failure MODE switches (build-fail → deliverable-fail). At minimum, document that the budget is shared.
### ADV-03: Path escape — deliverable paths resolve outside the worktree with no clamping

- **Severity**: low
- **Lens**: Skeptic
src/build-runner.ts:1844-1854 (requireFiles) and 1856-1876 (requireContains) and 1884-1897 (requireNotContains) all resolve spec-supplied paths with bare `resolve(cwd, p)` / `join(cwd, file)`. A spec declaring `requireFiles: ["../../../../etc/hostname"]` or `requireContains: {file:"../../sibling/preexisting.rs", pattern:"fn main"}` is satisfied by a file OUTSIDE the worktree — defeating the deliverable contract the feature exists to enforce. `requireContains` additionally READS the out-of-cwd file's contents (no data leaks into logs — only pattern names appear in `missing`/`ran` — so this is not an exfiltration vector, but it is a contract-evasion vector). Since the framework's stated job is to ENFORCE contracts rather than trust the spec author, an unguarded path escape cuts against the feature's intent. Recommendation: clamp resolved paths to the worktree root (`resolve(cwd, p)` must startWith `resolve(cwd)`), and record `path escapes worktree: <p>` as a missing/failure when it does not. Low severity because exploitation requires a malicious or careless spec author, but the whole feature premise is that spec authors' deliverable claims are NOT trusted.
### ADV-04: Regex ReDoS surface — spec-supplied patterns compiled and run with no timeout

- **Severity**: low
- **Lens**: Skeptic
src/build-runner.ts:1699-1714 `tolerantMatch` does `new RegExp(pattern)` then `re.test(text)` synchronously on spec/LLM-supplied patterns. A catastrophic-backtracking pattern (e.g. `(a+)+$`) from a buggy or adversarial spec will hang the Node event loop with no timeout — the `spawnSync({timeout})` envelope only bounds the test-lister subprocess, NOT the in-process regex engine. Because patterns are LLM-generated and never validated, this is a plausible denial-of-pipeline vector, and it would manifest as a silent stage hang rather than a clean failure (the try/catch cannot catch an infinite regex loop). Recommendation: either sandbox the regex in a worker_thread with a hard timeout, reject patterns matching known-dangerous shapes (nested quantifiers), or set a Node-level limit. Low/UNCERTAIN severity because trigger requires a pathological pattern, but the blast radius (whole pipeline hang, no clean failure) is disproportionate.
### ADV-05: Substring fallback for requireTests cuts toward the exact false-green this feature targets

- **Severity**: low
- **Lens**: Skeptic
src/build-runner.ts:1899-1912 matches each declared test name with `lines.some((line) => tolerantMatch(name, line))`, and `tolerantMatch` falls back to `text.includes(pattern)` on any non-matching/invalid regex. A declared `requireTests: ["fetch"]` is therefore satisfied by ANY pre-existing test whose listed name contains 'fetch' — including unrelated tests that pre-date the phase. The per-LINE split (added per an inline review note) mitigates the worst whole-stdout false-hit, and the task explicitly requests tolerant matching, but the tolerance direction is noteworthy: it biases toward the SAME false-green failure mode (a name that superficially matches but is not the deliverable test) that the feature exists to prevent. Recommendation: default to a stricter match (e.g. `\bname\b` or anchoring to the lister's name delimiter) and only fall back to bare substring on an explicitly-declared `loose:true`; or document in the spec-prompt instruction that declared names should be sufficiently specific (the current instruction does not).
### ADV-06: requireNotContains fail-closed on a MISSING file is stricter than the task spec

- **Severity**: low
- **Lens**: Minimalist
src/build-runner.ts:1884-1897 treats a missing target file for `requireNotContains` as a FAILURE (`missing file: <file>`), with an inline comment justifying the choice. The task spec only mandated `unreadable → fail`; it was silent on missing files for requireNotContains, and a naive reading ('a missing file cannot contain the forbidden pattern → pass') would be equally defensible. The chosen fail-closed semantics are arguably MORE correct for the stockfan case (a missing screen.rs means the phase didn't deliver) and align with the anti-false-green intent, so this is not a bug — but it IS an undocumented deviation from the written spec that a future reader could trip on (e.g. a spec author who forbids a pattern in a file that is conditionally generated will see unexpected failures). Recommendation: either add a one-line note to the spec-prompt instruction describing the fail-closed-on-missing semantics for requireNotContains, or accept the deviation explicitly in the contract docstring.
### ADV-07: Pervasive 'review finding:' comment annotations bloat the module with resolved-iteration narration

- **Severity**: low
- **Lens**: Minimalist
src/build-runner.ts carries multiple multi-line 'Review finding:'/'review finding, HIGH'/SCENARIO-XXX comment blocks (e.g. 1613-1621, 1639-1664, 1877-1883, 1899-1912) that narrate review iterations the code has already resolved. While individually informative, in aggregate they add substantial noise at rest (several 10-15 line blocks) and conflate historical review process with present behavior — a maintainer reading the code cold cannot tell which 'review finding' notes still carry actionable weight vs. which are fossil records. The implementation.ts:178-186 block has the same character. Recommendation: distill each 'review finding' block to the one-line invariant it establishes (e.g. 'Reset cache before each attempt: a retry may add a test the stale list would miss') and drop the iteration narrative. Low severity — readability only, no behavioral impact.
