# Specification: Per-Phase Deliverable Assertions for the Build Gate

- **Date**: 2026-07-21

---

## Summary

The build gate marks a phase GREEN when the in-scope build and existing tests pass (`gate.pass || gate.inScopePass`). That condition is necessary but NOT sufficient: a phase whose deliverable is creating a file, wiring a call site from X to Y, or adding a named test can compile green while delivering nothing. This is the proven root cause of the 2026-07-20T11-27-04-004Z false-green in stockfan spec 54, where Phases 5 and 6 reported "completed successfully" but delivered nothing (a never-created test file compiles fine; unwired code is still a valid public fn; dead sources behind a `_ => {}` router arm compile and pass their own unit tests). The fix adds a per-phase DELIVERABLE CONTRACT — `requireFiles`, `requireContains`, `requireNotContains`, `requireTests` — that the spec AUTHOR declares per phase, AND-ed with build-green. A new never-throwing `runDeliverableCheck(cwd, deliverables, opts?)` primitive in `src/build-runner.ts` (sibling to `runRedCheck`/`runBuildGate`, reusing `detectProjectCommands`, `resolveTimeoutMs`, `readMaybe`, and ONE cached `spawnSync` test-list per cwd per run) enforces it. The implementation stage's GREEN condition becomes `(gate.pass || gate.inScopePass) && deliverableCheck.pass`; when build-green but deliverables FAIL the missing list is fed into the next implementer attempt under a `## Deliverables still missing — create/wire these` block, retrying up to MAX_ATTEMPTS (3). The specification prompt and the typebox schema are extended so phases carry an optional `deliverables` object, and `normalizePhases` is widened to preserve it so `phase.deliverables` is statically typed. Backward compatible: phases/specs with no deliverables behave identically to today (deliverableCheck.pass = true). Pure TypeScript scoped to four source files plus the phase-normalizer helper and one new test file; no control-flow (nodes.ts/workflow.ts/pipeline.ts), review/integration, or backend-selection changes; no new runtime deps. SCOPE NOTE: full "is the new source actually reachable at runtime" execution-based coverage (beyond grep/anti-grep on router arms) is explicitly future work — the `requireContains`/`requireNotContains` regex assertions are the pragmatic version of reachability. Local verification: `npm run typecheck` + `npm test`.

## Architecture

Three cooperating layers: a deterministic checker primitive, a spec-elicited declaration, and a single-line verdict composition at the ONE site that decides GREEN.

LAYER 1 — runDeliverableCheck(cwd, deliverables, opts?) in src/build-runner.ts (AC-01/02). Added as a sibling of runRedCheck/runBuildGate and follows the established never-throwing-gate-primitive invariant (the load-bearing one guarded by build-runner-nonregression.test.ts): the ENTIRE body is wrapped in try/catch; on any thrown error it returns {pass:false, missing:['<reason>'], ran:[...]} rather than propagating. It reuses the single sources of truth: detectProjectCommands(cwd) for project detection (so the test-list runner is chosen exactly as runRedCheck chooses its runner), resolveTimeoutMs(opts.timeoutMs) for the spawn envelope, readMaybe(cwd, file) for best-effort file reads, and spawnSync(argv[0], argv.slice(1), {cwd, timeout, encoding:'utf8'}) for the single test-list subprocess. All four sub-checks evaluate EVERY element (no short-circuit) so `missing` is exhaustive and `ran` is complete: (a) requireFiles — fs.existsSync(resolve(cwd,p)) per path; miss → `missing file: <path>`; (b) requireContains — readMaybe each {file,pattern}; empty text due to unreadable (vs genuinely empty) → `unreadable: <path>`, else new RegExp(pattern) (try/catch → invalid pattern falls back to substring includes) tested against contents; miss → `missing pattern <pattern> in <file>`; (c) requireNotContains — same read/regex; hit → `forbidden pattern <pattern> still present in <file>`; (d) requireTests — build the project test list ONCE per cwd per run via a module-level Map<string,string> cache; runner selection from detectProjectCommands (rust → `cargo test --list --format=terse` fallback `-- --list`; node+vitest → `vitest list --json` fallback `npm test -- --list`; jest → `--listTests`; python → `pytest --collect-only -q`); on spawn error/timeout/empty stdout → record `test-list unavailable` for the whole requireTests sub-check and DO NOT block (existence/grep still enforced); each declared name tolerant-matches — try new RegExp(name) against the list, on invalid regex OR no regex match fall back to substring includes, match by EITHER satisfies; miss → `missing test: <name>`. The tolerant substring-OR-regex rule is documented in a code comment. Returns {pass: missing.length===0, missing, ran} where ran lists a human-readable token per check (e.g. `file:.../x.rs`, `contains:screen.rs:fetch_us_data`, `tests:list`). When deliverables is undefined/null/empty → early-return {pass:true, missing:[], ran:[]} immediately (backward compat).

LAYER 2 — spec declaration: schema + normalizer + prompt (AC-04/05). SpecificationData in src/render/schemas.ts extends the phases element Type.Object with deliverables: Type.Optional(Type.Object({ requireFiles: Type.Optional(Type.Array(Type.String())), requireContains: Type.Optional(Type.Array(Type.Object({file:Type.String(), pattern:Type.String()}))), requireNotContains: Type.Optional(Type.Array(Type.Object({file:Type.String(), pattern:Type.String()}))), requireTests: Type.Optional(Type.Array(Type.String())) })) — all-optional preserves backward compat (a spec with no deliverables validates identically). The existing STAGE_MODELS["spec"] registration, template, and additionalDocs are unchanged. normalizePhases(raw) in src/doc-validators.ts (consumed at implementation.ts:81) is widened from Array<{name, description?}> to Array<{name, description?, deliverables?}> so phase.deliverables is statically typed and round-trips. buildSpecPrompt in src/prompts.ts changes the phases data-to-return bullet from `{ name, description }` to `{ name, description, deliverables? }` and adds an explicit instruction that a phase whose deliverable is NOT compiler-checkable (creating a file, wiring a call site X→Y, making new sources reachable, adding a named test) MUST declare deliverables (requireFiles / requireContains+requireNotContains {file,pattern} regex checks — e.g. a wired call site or a non-`_ => {}` match arm / requireTests), AND-ed with build-green — without them a phase compiles green while delivering nothing.

LAYER 3 — AND-semantics wiring in src/stages/implementation.ts (AC-03), the correctness crux and minimal-diff change. Immediately after `const gate = runBuildGate(...)` (implementation.ts:154) and `attemptErrors = gate.errors`, call `const deliverableCheck = runDeliverableCheck(setup.worktreePath, phase.deliverables, { signal: ctx.signal })` and log `Implementation ${phaseId} deliverable-check ${deliverableCheck.pass ? "PASS" : "FAIL"} (missing: ${deliverableCheck.missing.join("; ") || "none"})` next to the existing build-gate log. Change ONLY the GREEN condition at implementation.ts:162 from `if (gate.pass || gate.inScopePass)` to `if ((gate.pass || gate.inScopePass) && deliverableCheck.pass)`. When build-green but deliverables FAIL → do NOT set green=true, do NOT break; instead push the missing list into the next attempt's implParts alongside the existing `## Previous attempt failed the build/test gate` block (implementation.ts:141-143) as `## Deliverables still missing — create/wire these\n- <entry>` so the implementer creates the files / does the wiring / adds the tests. The missing list resets each attempt (mirroring attemptErrors = gate.errors); the loop respects MAX_ATTEMPTS (3). Never throws (runDeliverableCheck never throws). When phase.deliverables is undefined → runDeliverableCheck returns {pass:true} → condition reduces to today's behavior.

CROSS-CUTTING INVARIANTS (the critical constraints): never-throw (every file/list/regex op wrapped); never destructure pi Theme (unrelated to this change but its guards must stay green); no new runtime deps; ONE cached subprocess per cwd per run; AND-semantics is mandatory (a missing deliverable MUST fail even when build-green); backward compatible for specs without deliverables. The deliverable-check is a SEPARATE check AND-ed alongside gate.pass/gate.inScopePass, orthogonal to runRedCheck (which runs pre-implement). Scope is limited to src/build-runner.ts, src/stages/implementation.ts, src/prompts.ts, src/render/schemas.ts, src/doc-validators.ts (normalizer), and one new test file; control-flow, review/integration, and backend selection are untouched.

## Testing Strategy

Local verification is `npm run typecheck` (tsc --noEmit, strict, .ts-import-extension NodeNext) and `npm test` (vitest). All new tests follow the established hermetic pattern: mkdtempSync(join(tmpdir(),"sd-<prefix>-")) temp cwds, fixture file writes, and a module-scope node:child_process.spawnSync mock for the requireTests lister; imports use the .ts extension.

LAYER 1 — tests/build-runner-deliverable-check.test.ts (AC-02): one new suite covering (a) all-present → pass:true, missing:[], ran complete; (b) missing requireFiles → exact `missing file: <path>`; (c) absent requireContains → `missing pattern <pattern> in <file>`; (d) forbidden requireNotContains hit → `forbidden pattern <pattern> still present in <file>`; (e) missing requireTests name (spawnSync mocked fixed list) → `missing test: <name>`; (f) present requireTests with tolerant substring-then-regex match (SCENARIO-006); (g) unreadable file (chmod 000 or mocked throw) → `unreadable: <path>`, no throw, remaining checks still run (SCENARIO-008); (h) no-runner-skip (detectProjectCommands reports no runner) → requireTests records `test-list unavailable`, does not block, existence/grep still enforced (SCENARIO-007); (i) test-list spawn cached — assert the mock fires at most once across two requireTests-bearing phases sharing the cwd (SCENARIO-009); (j) empty/undefined deliverables → {pass:true, missing:[], ran:[]} immediately (SCENARIO-014). Each case asserts the exact missing reason strings and the ran list.

LAYER 2 (AC-04/05): a schema-validation test asserting SpecificationData accepts phases[].deliverables present and validates identically when absent (SCENARIO-019/020); a prompt-content test asserting buildSpecPrompt(...) output contains the `{ name, description, deliverables? }` bullet AND the explicit deliverable-declaration instruction naming requireFiles/requireContains/requireNotContains/requireTests (SCENARIO-018); a normalizePhases test asserting a deliverables object on an input phase round-trips to the output phase (typed).

LAYER 3 (AC-03): a wiring-logic test of the verdict composition and missing-injection — a stubbed build-gate returning gate.pass===true combined with a deliverableCheck whose pass is false must yield a NON-green verdict (SCENARIO-011), and with pass===true → green; the missing-list injection is asserted by capturing the implementer retry prompt and confirming the `## Deliverables still missing — create/wire these` block with the missing entries (SCENARIO-012); the absent-deliverables path asserts no extra log noise and the condition reduces to today (SCENARIO-014); MAX_ATTEMPTS still bounds the retry (SCENARIO-013).

LAYER 4 — tests/implementation-deliverable-regression.test.ts (AC-06): the stockfan Phase-5/6 false-green regression. Case A (the bug, SCENARIO-016): a phase declaring requireFiles:[X] + requireContains:[screen.rs→fetch_us_data] + requireNotContains:[screen.rs→`with_retry\(\|\| fetch_fmp`] + requireTests:[T], with ALL deliverables ABSENT and a stub build-gate reporting gate.pass===true → assert runDeliverableCheck(...).pass===false (non-empty missing) and the combined `(gate.pass || gate.inScopePass) && deliverableCheck.pass` verdict is NOT green. Case B (the happy path, SCENARIO-017): the same phase with deliverables PRESENT → pass===true and the combined verdict IS green — proving AND-semantics both ways.

FULL-SUITE GATE (AC-07): npm run typecheck strict-clean with no new errors; npm test fully green — all 1120 pre-existing tests plus the new suites. Explicitly verify no regression to runRedCheck, npm in-scope detection, the scope-aware cargo gate, themed stream rendering (tests/stream-theme-class-theme.test.ts, tests/render/real-theme-parity.test.ts — guard the never-destructure-Theme rule), mid-run input injection, and the dashboard.
