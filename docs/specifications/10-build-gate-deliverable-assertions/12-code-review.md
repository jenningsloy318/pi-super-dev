# Code Review: Code Review: Per-Phase Deliverable Assertions for the Build Gate

- **Date**: 2026-07-21
- **Author**: super-dev:code-reviewer
- **Verdict**: Changes Requested

---

## Verdict: Changes Requested

The implementation delivers the per-phase deliverable contract end-to-end: `runDeliverableCheck` (src/build-runner.ts) is a never-throwing sibling oracle implementing requireFiles/requireContains/requireNotContains/requireTests with tolerant substring-OR-regex matching, exhaustive (no short-circuit) `missing`, and a {pass,missing,ran} return; the implementation stage's GREEN condition is correctly widened to `(gate.pass || gate.inScopePass) && deliverableCheck.pass` with the missing list injected into the next implementer retry under a "Deliverables still missing" block; and the spec schema (SpecPhase/PhaseDeliverables), `normalizePhases` widening, and `buildSpecPrompt` elicitation are all in place and backward-compatible (no deliverables → early {pass:true}). Typecheck is strict-clean and 53 new tests pass. AND-semantics is sound and well-tested for requireFiles/requireContains/requireNotContains (which re-read the filesystem each call). However, ONE correctness bug undermines the feature for `requireTests` specifically: the module-level `testListCache` is never invalidated between retry attempts, so a test-list cached on attempt 1 stays stale for attempts 2–3 — meaning once `requireTests` fails, no amount of implementer work can flip the verdict (the new test exists on disk but isn't in the cached list), causing the phase to abort after MAX_ATTEMPTS and produce a false-negative that defeats the very retry mechanism the feature was built to enable. There is also a coverage gap against AC-04 (the combined real-checker × stub-green-gate regression is asserted only with a mocked primitive, not the real `runDeliverableCheck`). Neither is Critical (no silent data corruption / security exposure — the failure is noisy and logged), but the cache staleness is a real production correctness regression, so the verdict is Changes Requested.

## Findings

### F-01: testListCache is never invalidated, so requireTests cannot recover across retry attempts within a phase (defeats the core retry mechanism)

- **Severity**: High
- **File**: `src/build-runner.ts`
- **Line**: 1629
`testListCache` (module-level Map, build-runner.ts:1629) is populated on the first `loadTestList(cwd,...)` call and returned verbatim on every subsequent call for the same cwd — there is NO `.delete`/`.clear` anywhere (grep confirms only `.get`/`.set`), and the implementation stage never resets it. SCENARIO-009 explicitly tests that the lister spawns once across two calls sharing a cwd, which is correct for the cross-PHASE case — but it is fatal for the cross-ATTEMPT case. Production scenario: a phase declares `requireTests:["my_test"]`. Attempt 1: the implementer omits the test; `runDeliverableCheck` spawns `cargo test -- --list` (or `vitest list --json`), the list lacks `my_test`, requireTests FAILS, the list is cached, and `## Deliverables still missing — create/wire these / - missing test: my_test` is injected. Attempt 2: the implementer creates the test on disk; but `loadTestList` returns the STALE cached list (pre-creation), so requireTests STILL reports `missing test: my_test` and the phase stays not-green. Attempt 3: same → the phase aborts after MAX_ATTEMPTS and `allGreen=false; break` terminates the ENTIRE run — even though the test now exists and would pass. This is a false-NEGATIVE (opposite of the false-positive the feature exists to fix) and it specifically breaks the documented recovery path for the most common real-world deliverable ('add a named test'). Note the asymmetry: requireFiles/requireContains/requireNotContains re-read the filesystem every attempt (correct), but requireTests does not. FIX: invalidate the cache between attempts — e.g. export `clearTestListCache()` (or `clearDeliverableCaches()`) and call it at the top of each attempt iteration in implementation.ts, OR clear it whenever a deliverable check reports any `missing test:` entry so the next attempt re-lists. Re-listing per failed attempt is correct even though `cargo test --list` compiles, because correctness must outrank the redundant-spawn optimization. Confidence: 0.85.
### F-02: AC-04 combined regression is exercised only with a MOCKED runDeliverableCheck, not the real primitive (spec's implementation-deliverable-regression.test.ts is absent)

- **Severity**: Medium
- **File**: `tests/implementation-deliverable-wiring.test.ts`
- **Line**: 32
Acceptance criterion #4 asks for a single regression proving: a phase declaring requireFiles+requireContains+requireNotContains+requireTests, with deliverables ABSENT and a stub build-gate reporting gate.pass===true, yields `runDeliverableCheck(...).pass===false` AND a combined NOT-green verdict. The actual tests split this: tests/build-runner-deliverable-check.test.ts proves the real primitive returns pass===false for missing deliverables (in isolation), and tests/implementation-deliverable-wiring.test.ts (SCENARIO-011/015) proves the AND-semantics verdict composition — but with runDeliverableCheck fully stubbed via vi.mock (it never executes the real regex/fs/spawn logic). The spec's testing strategy named `tests/implementation-deliverable-regression.test.ts`, which does not exist. Risk: an integration defect (e.g. a shape mismatch between the real DeliverableCheckResult and what the verdict line reads) could pass both suites while breaking the end-to-end guarantee the AC was written to lock in. FIX: add one test that imports the REAL runDeliverableCheck (un-mocked) over a temp cwd with the stockfan-shaped absent deliverables, asserts pass===false, and feeds that real result through the verdict line to assert NOT-green. Confidence: 0.6.
### F-03: loadTestList caches by raw cwd, not resolve()'d to absolute — inconsistent with sibling cargoMetadataCache (cache-key/argv skew risk)

- **Severity**: Low
- **File**: `src/build-runner.ts`
- **Line**: 1707
loadTestList uses `testListCache.get(cwd)` / `.set(cwd, res)` with the caller-supplied cwd verbatim, whereas the neighboring `loadCargoMetadata` deliberately does `const absCwd = resolve(cwd)` and keys the cache by the absolute path (with an explicit code comment warning about relative/symlinked cwd causing duplicate spawns). In practice implementation.ts always passes an absolute `setup.worktreePath`, so today this only risks a redundant spawn under unusual relative/symlink inputs — not a correctness failure. But it violates the module's own stated cache discipline and is a latent footgun if a future caller passes a relative path. FIX: resolve cwd to absolute once in loadTestList and use it for both the cache key and the spawn (mirroring loadCargoMetadata). Confidence: 0.8.
### F-04: runDeliverableCheck spawns the test-lister even when the build gate already failed (wasted compile on a broken build)

- **Severity**: Low
- **File**: `src/stages/implementation.ts`
- **Line**: 161
implementation.ts runs `runDeliverableCheck` unconditionally after `runBuildGate`, so when the build is broken (gate.pass===false, e.g. a compile error), the requireTests path still triggers a `cargo test -- --list` — which for cargo is a full compile + list, repeating the just-failed build's cost (mitigated only by the once-per-run cache and the timeout envelope). It is correctness-safe (runDeliverableCheck never throws; a broken build just yields tests:unavailable or a stale-but-harmless list, and existence/grep sub-checks still run) and is consistent with the spec's 'no short-circuit / exhaustive missing' design, so this is an efficiency note, not a defect. Optional optimization: skip the test-list spawn when `!gate.pass && !gate.inScopePass`, since a not-green gate means the phase will retry regardless of the deliverable verdict. Confidence: 0.7.
