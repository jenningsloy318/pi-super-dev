# Requirements: Per-phase Deliverable Assertions for the Build Gate (false-green root-cause fix)

- **Date**: 2026-07-20
- **Author**: super-dev:requirements-clarifier
- **Type**: bug-fix
- **Priority**: critical
- **Status**: draft

---

## Executive Summary

A phase is currently GREEN whenever its in-scope crates compile and existing tests pass (`gate.pass || gate.inScopePass` in implementation.ts), which is necessary but not sufficient: any phase whose deliverable is not a compile error (creating a file, wiring a call site, making a new source reachable, adding a named test) reports green while delivering nothing — exactly the false-green in run 2026-07-20T11-27-04 (stockfan Phases 5 & 6). This fix introduces an OPTIONAL per-phase `deliverables` contract (requireFiles / requireContains / requireNotContains / requireTests) AND-ed with build-green, so a phase is GREEN only when it both compiles and its declared artifacts actually exist. Backward compatible: phases/specs with no `deliverables` are unchanged.

## Acceptance Criteria

- **AC-01**: A new `runDeliverableCheck(cwd, deliverables, opts?)` function is added to `src/build-runner.ts` alongside `runRedCheck`/`runBuildGate`, reusing `resolveTimeoutMs` and `detectProjectCommands`. It enforces four optional sub-checks: (a) `requireFiles` via `fs.existsSync` resolved relative to cwd — miss → `missing file: <path>`; (b) `requireContains` reads the file (best-effort; unreadable → `unreadable: <path>`) and regex-tests `pattern` — miss → `missing pattern <pattern> in <file>`; (c) `requireNotContains` regex-tests `pattern` — hit → `forbidden pattern <pattern> still present in <file>`; (d) `requireTests` lists the project's tests per runner (cargo → `cargo test --list --format=terse` or `-- --list`; vitest → `vitest list --json` or `npm test -- --list`; jest → `--listTests`; pytest → `--collect-only -q`) and tolerant-matches each declared name by substring OR regex — miss → `missing test: <name>`. The test-list spawn is cached once per cwd per run. It returns `{ pass: boolean, missing: string[], ran: string[] }` and NEVER throws: any unreadable file, failed list-spawn, parse ambiguity, or absent test-runner records a clear `missing`/reason and continues; when no runner is available `requireTests` records `test-list unavailable` and does NOT block (existence/grep still enforced). The tolerant substring-OR-regex match rule is documented in a code comment.
- **AC-02**: `runDeliverableCheck` is fully unit-tested in a new `tests/build-runner-deliverable-check.test.ts` using a temp cwd: existing vs missing files, present vs absent `requireContains` patterns, `requireNotContains` hit (forbidden still present), missing vs present `requireTests` names, an unreadable file path, and the no-runner-skip case (returns `{pass:true, missing:[...recorded...], ran:[...]}` for requireFiles/requireContains/requireNotContains while skipping requireTests). Coverage asserts the exact `missing` reason strings and that `ran` lists the checks performed.
- **AC-03**: In `src/stages/implementation.ts`, immediately after `runBuildGate` produces `gate`, the stage also calls `runDeliverableCheck(setup.worktreePath, phase.deliverables)` (phase iteration comes from `normalizePhases(state.spec?.phases)`, which is extended to PRESERVE a `deliverables` field so `phase.deliverables` is typed). The phase is GREEN only when `(gate.pass || gate.inScopePass) && deliverableCheck.pass`. When build-green but deliverables FAIL, the stage does NOT set `green=true`, does NOT break, and feeds `deliverableCheck.missing` into the next implementer attempt as an additive `## Deliverables still missing — create/wire these` block (alongside the existing `## Previous attempt failed the build/test gate` block that uses `attemptErrors`), so the implementer creates the files / does the wiring / adds the tests. Retries up to `MAX_ATTEMPTS` (3). It logs `Implementation ${phaseId} deliverable-check ${PASS|FAIL} (missing: <semi-colon-joined or none>)` and the combined verdict. Never throws; when `phase.deliverables` is absent/undefined → `deliverableCheck.pass = true` (today's behavior, no-op).
- **AC-04**: `buildSpecPrompt` in `src/prompts.ts` (the specification prompt whose `## Data to return` lists `phases`) is updated so the `phases` item reads `{ name, description, deliverables? }`, and an explicit instruction is added: for any phase whose deliverable is NOT compiler-checkable (creating a file, wiring a call site from X to Y, making new sources reachable, adding a named test), the spec author MUST declare `deliverables` with `requireFiles` (paths that must exist), `requireContains`/`requireNotContains` (`{file, pattern}` regex checks — e.g. a call-site wiring or a non-`_ => {}` match arm), and `requireTests` (test names that must exist), because these are AND-ed with build-green to define phase completion — without them a phase compiles green while delivering nothing.
- **AC-05**: `SpecificationData` in `src/render/schemas.ts` is extended so `phases[].deliverables` is accepted as an OPTIONAL object: `{ requireFiles?: string[], requireContains?: {file:string, pattern:string}[], requireNotContains?: {file:string, pattern:string}[], requireTests?: string[] }`. The existing `STAGE_MODELS["spec"]` registration, template, and additionalDocs are unchanged. A spec with no `deliverables` validates identically to today.
- **AC-06**: A regression test simulates the stockfan Phase-5/6 false-green end-to-end at the checker + wiring level: a phase declaring `requireFiles:[X]` + `requireContains:[screen.rs→fetch_us_data]` + `requireNotContains:[screen.rs→fetch_fmp]` + `requireTests:[T]`. With ALL deliverables ABSENT, `runDeliverableCheck(...).pass === false` (non-empty `missing`) even though a stub `runBuildGate` would report `gate.pass === true`; the combined phase verdict computed by the wiring logic is NOT green. A second test asserts the same phase with deliverables PRESENT yields `pass === true` and a green verdict, proving AND-semantics both ways.
- **AC-07**: `npm run typecheck` is strict-clean (no new errors) and `npm test` is fully green — all existing tests pass (including the themed-stream, real-theme-parity, mid-run-input, dashboard, runRedCheck, npm-in-scope, and scope-aware-cargo-gate suites) PLUS the new deliverable-check unit tests, the AND-wiring test, and the spec-prompt-elicitation test. No regression to `runRedCheck`, npm in-scope detection, scope-aware cargo gate, themed stream rendering, mid-run input injection, or the dashboard.

## Non-Functional Requirements

- Robustness / fail-soft: `runDeliverableCheck` and every file/list/regex operation it performs must NEVER throw — unreadable files, missing files, failed test-list spawns, regex parse errors, and parse ambiguity are all recorded as a `missing` reason string and the check continues. The gate must never crash because a declared file was unreadable or a runner failed to list tests.
- Backward compatibility: phases and full specs that declare NO `deliverables` are byte-for-byte identical to today's behavior (`deliverableCheck.pass = true`, no extra log noise beyond an optional single line, no extra subprocess). Existing specs require zero changes.
- Performance: at most ONE cached test-list subprocess spawn per cwd per run (shared across phases that share the worktree). File existence checks are `fs.existsSync` (no I/O storm); regex checks are bounded single-file reads. No new runtime dependencies. No new spawned processes beyond the single cached test-list per run.
- Scope correctness (out-of-scope guard): this change touches ONLY `src/build-runner.ts`, `src/stages/implementation.ts`, `src/prompts.ts`, `src/render/schemas.ts` (and the `normalizePhases` helper it relies on). It MUST NOT alter control-flow (`nodes.ts`/`workflow.ts`/`pipeline.ts`), the review/integration stages, or backend selection.
- Theme safety: the change must never destructure a pi `Theme` instance (a class relying on `this.fgColors`); always call methods style. The `tests/stream-theme-class-theme.test.ts` and `tests/render/real-theme-parity.test.ts` guards must remain green.

## Open Questions

- requireTests cache scope: confirm the test-list spawn is cached PER CWD PER RUN (shared across all phases of one implementation run in one worktree), not per phase. Recommended: per cwd per run — phases share a worktree, so one `cargo test --list` / `vitest list` per run. Confirm before implementation.
- Deliverable-miss message block: confirm it is ADDITIVE to (not a replacement for) the existing `## Previous attempt failed the build/test gate` block that uses `attemptErrors`. Recommended: both blocks are emitted when both apply (a phase can fail build-green-only-on-deliverables, or fail on both). Confirm.
- Tolerant test-name match precedence for requireTests: when a declared name is neither a valid regex nor a substring match, which wins for the no-match verdict? Recommended: treat the pattern as regex if it parses as one, else substring; a match by EITHER satisfies the requirement; document the rule in-code. Confirm the substring-vs-regex precedence and that the stockfan `module::test` path can be matched by a substring of the test name.
- normalizePhases extension: confirm `normalizePhases` (in src/stages/implementation.ts, currently producing `{name, description}`) is extended to preserve an optional `deliverables` field rather than reading deliverables by index from the raw `state.spec.phases`. Recommended: extend the normalized phase shape so `phase.deliverables` is statically typed. Confirm the exact normalized shape to avoid a second normalization surprise like the earlier `phases.entries is not a function` regression.
