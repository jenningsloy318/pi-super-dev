# Research Report: Per-phase Deliverable Assertions for the build-gate — mechanism, runner test-enumeration, and the documented grep-vs-runtime boundary

- **Date**: 2026-07-20
- **Author**: super-dev:research-agent

---

## Summary

This brief researches how to make a pi-super-dev build-gate phase GREEN only when its declared DELIVERABLES actually exist (not merely when code compiles and existing tests pass — the false-green root cause observed in run 2026-07-20T11-27-04-004Z). The recommended mechanism is a new `runDeliverableCheck(cwd, deliverables, opts)` that AND-s four optional sub-checks with build-green: `requireFiles`/`requireContains`/`requireNotContains` are pure fs+regex (language-agnostic, deterministic, no build), and `requireTests` enumerates the runner's tests and tolerant-matches declared names. The checker should mirror this repo's OWN proven patterns already in src/build-runner.ts: `readMaybe` (never-throw best-effort file read), per-cwd spawn caching (`cargoMetadataCache`), reuse of `detectProjectCommands`/`resolveTimeoutMs`, and module-wide never-throw discipline where any failure records a reason and continues (SRC-05). The four runner test-list CLIs are verified against official docs: cargo libtest `--list` (with `-q`≡`--format=terse`) and pytest `--collect-only -q` enumerate test NAMES like `module::test` (SRC-01, SRC-02); jest `--listTests` returns test FILE PATHS, one per line (SRC-03) — the documented ISS-1 limitation; vitest exposes a `list` subcommand family but its JSON shape is the least version-stable of the four (SRC-04), so the design degrades to "test-list unavailable" rather than crashing. Two open issues are carried forward as FULLY-DOCUMENTED, NON-BLOCKING limitations with verbatim carry text (ISS-1 jest file-level requireTests; ISS-3 grep-is-syntactic-not-runtime). Backward compatibility is preserved by a no-op pass when `phase.deliverables` is absent.

## Options Considered

### A. Pure fs+regex deliverable checks (requireFiles/requireContains/requireNotContains) — chosen

Fast (no build), deterministic, fully language-agnostic, trivially unit-testable with a temp cwd, and matches the false-green root cause directly (missing file = fail; absent wiring pattern = fail). Reuses the existing `readMaybe` never-throw read (SRC-05). COST: grep is SYNTACTIC not SEMANTIC — `requireNotContains` proving a `_ => {}` catch-all arm is gone does NOT prove the new source is routed at runtime; a non-`_ => {}` arm that still returns early, or a wiring that compiles but is dead, satisfies the anti-grep. This is the ISS-3 boundary: execution-based runtime reachability is explicitly out of scope and future work; the grep assertions are the pragmatic version and MUST be stated verbatim as a documented limitation in the implementation summary.
### B. requireTests via native per-runner test-list enumeration + tolerant substring-OR-regex match — chosen

Catches the Phase-6 false-green class (a named e2e test that must EXIST) that fs/grep alone cannot — grep can't assert 'a test named T was added'. Verified CLIs: cargo `-- --list` (SRC-01) and pytest `--collect-only -q` (SRC-02) give test NAMES; jest `--listTests` gives file PATHS not names (SRC-03); vitest `list` family JSON shape is version-unstable (SRC-04). Mitigations baked in: (1) tolerant substring-OR-regex match so spec authors needn't quote cargo's exact `module::test` path; (2) ONE cached list-spawn per cwd per run (mirror `cargoMetadataCache`, SRC-05); (3) NEVER throw — a failed/absent list records a single `test-list unavailable (runner: <r>)` reason and does NOT block existence/grep checks; (4) AC-04 spec-prompt must note requireTests names are runner-specific (cargo=module::test, vitest/pytest=name, jest=path). COST: runner-output variance is real and is exactly why the never-throw + unavailable-reason discipline is mandatory.
### C. AND-semantics for phase GREEN: `(gate.pass||gate.inScopePass) && deliverableCheck.pass` — chosen (vs OR/advisory)

This is the whole point of the fix: a missing deliverable MUST fail the phase even when build-green, closing the false-green. Combined with a no-op pass when deliverables are absent, existing specs behave byte-identically to today (backward compatible). The residual risk — a SPEC with an over-broad/wrong deliverable declaration false-failing a build that is genuinely green — is acceptable and self-correcting within MAX_ATTEMPTS because `deliverableCheck.missing` is fed into the next implementer attempt exactly like today's `attemptErrors`. OR/advisory-only was rejected outright: it leaves the false-green open.
### D. Execution-based runtime reachability assertion (actually run the new code path) — REJECTED / out of scope

Would catch the dead-wiring class that pure grep (Option A) cannot (ISS-3): require the harness to actually execute the target app and assert the new source is reached. COST: requires app-specific harnessing of an arbitrary target repo from within a TS-only pi-super-dev change, is non-deterministic, slow, and couples the harness to every target's runtime shape. Explicitly out of scope per the task; the requireContains/requireNotContains grep assertions are the pragmatic substitute, with the gap documented verbatim. Listed here only to record the decision boundary so the next stage does not re-litigate it.

## Open Issues

- ISS-1 (jest, design-covered, DOCUMENTED NON-BLOCKING LIMITATION — resolved-as-documented): jest `--listTests` returns test FILE PATHS, not test names (SRC-03, jestjs.io/docs/cli: '--listTests … Lists all test files … given the provided options and exits'), so requireTests is inherently FILE-LEVEL for jest repos. Mitigation (carried verbatim into implementation): (1) tolerant substring-OR-regex match against the runner's actual list payload; (2) a single `test-list unavailable (runner: jest)` reason when listing fails/absent; (3) the AC-04 spec-prompt note that requireTests names are runner-specific (cargo=module::test, vitest/pytest=name, jest=path). The never-throw discipline guarantees a jest repo never crashes the gate — it records the reason and still enforces existence/grep.
- ISS-3 (runtime reachability, DOCUMENTED NON-BLOCKING LIMITATION — resolved-as-documented): `requireNotContains` proving the `_ => {}` catch-all arm is gone does NOT prove the new source is routed at runtime — a non-`_ => {}` arm that still returns early, or a wiring that compiles but is dead, satisfies the anti-grep. This MUST be stated verbatim in the implementation summary; execution-based runtime reachability is explicitly out of scope and is future work (Option D, rejected). The requireContains/requireNotContains grep assertions are the pragmatic version of this check; their syntactic-not-semantic nature is the documented ceiling of what this change can guarantee.
- ISS-4 (vitest list JSON shape, LOW — design degrades): vitest's `list` subcommand family exists (SRC-04, vitest.dev/guide/cli) and the `--listTags=json` JSON convention is documented, but the exact `vitest list --json` payload shape is the least version-stable of the four runners. Resolution: the never-throw + cached-once-per-cwd + `test-list unavailable (runner: vitest)` degrade path covers it; requireFiles/requireContains/requireNotContains still enforce for vitest repos even if test-listing fails. No further research needed before implementation — the design is robust to the unknown.
