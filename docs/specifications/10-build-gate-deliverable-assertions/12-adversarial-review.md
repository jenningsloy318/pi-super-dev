# Adversarial Review: Adversarial Review: Build-Gate Per-Phase Deliverable Assertions

- **Date**: 2026-07-21
- **Reviewer**: super-dev:adversarial-reviewer
- **Verdict**: REJECT

---

The implementation adds a never-throwing `runDeliverableCheck` primitive (Layer 1), widens the schema/normalizer/prompt to carry optional per-phase `deliverables` (Layer 2), AND-s it into the implementation stage's GREEN verdict (Layer 3), and ships a stockfan regression test (Layer 4). The design intent — making GREEN mean "compiles AND deliverables exist", the proven root cause of the 2026-07-20 stockfan false-green — is sound: requireFiles/requireContains/requireNotContains/requireTests with tolerant matching, a cached test-list spawn, exhaustive (no-short-circuit) `missing`, and a never-throw contract. The requireNotContains semantics were even hardened mid-implementation (a missing/unreadable target now FAILS rather than silently passes).\n\nHOWEVER, the change CANNOT MERGE as-is. The single load-bearing wiring line is broken: `resetDeliverableCheckCache()` is invoked at src/stages/implementation.ts:180 but is NOT in the import statement at line 14 (it IS exported at build-runner.ts:1665). `npm run typecheck` is RED (`TS2304: Cannot find name 'resetDeliverableCheckCache'`) and the targeted test run is RED (6 failed in implementation.test.ts with `resetDeliverableCheckCache is not defined` — a free variable the source never imports cannot be supplied by a module mock). This directly violates acceptance criteria #5 (typecheck + tests green) AND means the production implementation stage throws ReferenceError on the first phase with deliverables (the call is unconditional). This is a textbook "mock masks the real import" defect that the global test suite would have caught but the Layer-4 gate did not. The fix is one line (add `resetDeliverableCheckCache` to the import) but it is a hard blocker.\n\nBeyond the blocker: the requireTests sub-check — the single most valuable assertion, the one that actually catches "a test was never created" — is also the most fragile: it spawns bare `vitest`/`jest`/`pytest` binaries that are typically NOT on PATH (they live in node_modules/.bin or a venv) and silently degrades to "unavailable" on ENOENT, re-opening the exact false-green class the feature exists to close. The run-boundary reset is enforced only by comment (a global mutable cache + undocumented temporal invariant) and is, ironically, the line that's currently broken. Recommend REJECT until the import is fixed and typecheck+full test suite are green; F2 (requireTests binary resolution) should be fixed or explicitly risk-accepted before this can reliably prevent a recurrence of the stockfan bug.

### F1: Missing import: resetDeliverableCheckCache undefined → typecheck RED, 6 tests RED, runtime crash

- **Severity**: high
- **Lens**: Skeptic
implementation.ts:14 imports `{ runBuildGate, runDeliverableCheck, runRedCheck, type GateOptions, type RedStatus }` from build-runner.ts — `resetDeliverableCheckCache` is ABSENT. Yet line 180 calls `resetDeliverableCheckCache();` unconditionally inside every attempt loop. The function IS exported (build-runner.ts:1665). Consequences, all verified: (1) `npm run typecheck` → `src/stages/implementation.ts(180,5): error TS2304: Cannot find name 'resetDeliverableCheckCache'`. (2) `npx vitest run tests/build-runner-deliverable-check.test.ts src/stages/implementation.test.ts` → 6 failed in implementation.test.ts with `resetDeliverableCheckCache is not defined` at line 180. The module mocks added in implementation.test.ts / implementation-tdd-rust-wiring.test.ts (`resetDeliverableCheckCache: () => {}`) CANNOT rescue this: vi.mock supplies named exports of the mocked module, but it cannot create a binding for an identifier the source never imported — `resetDeliverableCheckCache()` is a free variable → ReferenceError. (3) Production: the implementation stage throws on the first phase processed (the call is unconditional, not deliverable-gated), outside any try/catch, aborting the run. This violates AC-05 (npm test green) and AC-06 (npm run typecheck green) — the two acceptance criteria whose entire purpose is to catch exactly this. Fix: add `resetDeliverableCheckCache` to the line-14 import. This is a one-liner but is a hard merge blocker; the feature is non-functional as committed.
### F2: requireTests spawns bare vitest/jest/pytest binaries → ENOENT → silent 'unavailable' → missing tests not caught

- **Severity**: high
- **Lens**: Skeptic
resolveTestListerArgv returns bare binaries: `["vitest", "list", "--json"]`, `["jest", "--listTests"]`, `["pytest", "--collect-only", "-q"]`. In real/CI worktrees these are rarely on PATH — they live in node_modules/.bin (vitest/jest) or a venv bin (pytest). spawnSync then yields an ENOENT error → loadTestList records `{available:false}` → runDeliverableCheck pushes `tests:unavailable` and does NOT block, so a declared requireTest that was never created is silently NOT reported missing. This is precisely the stockfan Phase-6 failure mode ('create e2e_screen_us_fallback.rs test') that the feature is meant to prevent, and requireTests — the one sub-check that can catch a never-created test — is the most fragile path. runBuildGate/runRedCheck avoid this because they run via `npm test` / `cargo` (resolved by npm/rustup); the lister bypasses that resolution. Fix: resolve via `node_modules/.bin` (join cwd + that dir, PATH-prepend) or `npx --no-install vitest ...`, and for python resolve the venv interpreter; fall back to `npm test -- --list` for the node family. Without this, requireTests is effectively dead for most repos and the AND-semantics guarantee is undercut exactly where it matters most.
### F3: Global test-list cache with comment-only run-boundary contract (and the honoring call site is the broken F1 line)

- **Severity**: medium
- **Lens**: Architect
testListCache is a module-global Map. Correctness of the retry mechanism (a freshly-added test on attempt N must be seen) now DEPENDS on every caller of runDeliverableCheck remembering to call resetDeliverableCheckCache() first. This invariant is documented in comments but is NOT enforced by the type system — any future caller (a new stage, a future test, the worktree-router multi-worktree path) that omits the reset gets silent stale-test-list false-negatives. The coupling is also poorly scoped: reset clears ALL cwd entries, not just the current worktree's. Better design: thread the cache through DeliverableCheckOptions (caller owns lifetime) or scope it per-call so a caller physically cannot forget. Compounding irony: the ONE site supposed to honor this contract is implementation.ts:180 — exactly the missing-import line in F1 — so right now the reset NEVER runs at all, meaning even after F1 is fixed, correctness rests entirely on this undocumented global-state invariant.
### F4: skipTests:!buildGreen defers requireTests on build-red attempts → incomplete feedback to implementer

- **Severity**: low
- **Lens**: Architect
The optimization `skipTests: !buildGreen` (defensible: avoid wasted compile + poisoned cache on a broken build) means that during a build-RED attempt the requireTests sub-check is skipped, so its missing-test entries are absent from deliverableCheck.missing. That missing list is the ONLY signal fed into the next attempt's '## Deliverables still missing — create/wire these' block. Net: on a build-red attempt the implementer is told 'fix the build' but NOT 'also you need to add test T'. If the implementer fixes the build without adding T, the next attempt is build-green → requireTests runs → missing test surfaces, costing an extra full retry iteration (each iteration is a full agent round + gate). It converges, so not a correctness bug, but it is a latency/incomplete-feedback cost on the highest-value deliverable type. Consider: when skipTests is set, still synthesize 'pending: test <name> will be verified once build is green' hints into the implementer prompt from the declared requireTests.
### F5: requireNotContains FAILS on a missing/unreadable target file — surprising for spec authors who'd delete-then-recreate

- **Severity**: low
- **Lens**: Skeptic
The hardened requireNotContains semantics (a missing/unreadable target file now FAILS rather than silently passing) are internally consistent and defensible — a contract forbidding a pattern in <file> cannot be verified when <file> is absent. But it is a footgun for spec authors: 'requireNotContains screen.rs -> fetch_fmp' reads as 'assert fetch_fmp is gone from screen.rs', and deleting screen.rs (or replacing it) is often a legitimate way to remove that call. The author would get a confusing `missing file: screen.rs` failure. The spec-prompt instruction added in prompts.ts documents requireContains/requireNotContains but does NOT warn that the file must still exist. Recommend either (a) documenting the 'file must still exist' contract in the spec prompt, or (b) making requireNotContains a pure absence check that passes when the file is gone. Pick one and document it.
### F6: Deliverables shape declared 2-3× (PhaseDeliverables / DeliverableContract / NormalizedPhase.deliverables)

- **Severity**: low
- **Lens**: Minimalist
Three near-identical type declarations exist for the optional-deliverables shape: PhaseDeliverables (schemas.ts), DeliverableContract (build-runner.ts:1587), and NormalizedPhase.deliverables: PhaseDeliverables (doc-validators.ts). PhaseDeliverables and DeliverableContract are structurally identical but separately authored. Adding a field (e.g. requireLines, requireMinCount) requires touching ≥2 files in lockstep; drift between them is silent (TS structural typing won't flag a missing optional field). Unify on a single exported type (build-runner imports PhaseDeliverables from schemas, or schemas re-exports DeliverableContract).
### F7: Untrusted-input vectors: no regex timeout + no path-traversal guard on file checks

- **Severity**: low
- **Lens**: Skeptic
(1) tolerantMatch compiles spec-declared patterns via `new RegExp(pattern)` and `.test()` against file contents with NO timeout. A catastrophic-backtracking pattern hangs the (single-threaded) deliverable check; the spawn timeout does not cover in-process regex. The spec is LLM-generated and influenceable via task input / mid-run input injection (a known threat in this repo per learned.md). (2) readForDeliverable resolves `join(cwd, file)` with no traversal guard — a spec with `file: "../../../etc/passwd"` is readable. Contents are NOT exfiltrated into output (only `missing pattern <pat> in <file>` / `unreadable: <path>`), so leak is limited to path-existence oracle, but the read itself crosses the worktree boundary. Mitigations: bound regex eval (or reject patterns over N chars / with nested quantifiers), and reject/normalize `file` paths that escape cwd.
### F8: `ran` audit array built unconditionally on every check, consumed by one log line

- **Severity**: low
- **Lens**: Minimalist
DeliverableCheckResult.ran accumulates a human-readable token per sub-check element on EVERY invocation (file:..., contains:..., not-contains:..., tests:list/unavailable) and is surfaced only in a single ctx.log line in the implementation stage. For a phase with many requireContains entries this is a per-element string push on the hot path of every attempt. Value is marginal (logs already print pass/missing). Consider building ran only at debug/trace level, or dropping it from the return type in favor of structured logging inside the primitive. Negligible perf cost; flagged as a token/clarity minimalism nit.
### F9: Heavy meta-commentary ('review finding:', SCENARIO-NNN) inflates a now-1936-line hot file

- **Severity**: low
- **Lens**: Minimalist
The new ~370 lines carry dense JSDoc that cites review-process chatter ('review finding: ...', 'SCENARIO-007', 'SCENARIO-009') inline. These annotations document the review history rather than future-intent and bloat build-runner.ts — which is loaded into agent context — at ~75 chars/comment-line of marginal maintenance value. Trim to WHY/WHAT; move scenario-ID cross-refs to the test file (they already exist there) or a CHANGELOG note. Keeps the file navigable and reduces context token cost.
