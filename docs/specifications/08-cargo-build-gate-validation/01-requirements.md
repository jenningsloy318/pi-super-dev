# Requirements: Scope-aware cargo build gate: include untracked files, validate package specs, and add a spec-declared gate contract

- **Date**: 2026-07-20
- **Author**: super-dev:requirements-clarifier
- **Type**: bug-fix
- **Priority**: high
- **Status**: draft

---

## Executive Summary

spec-07 shipped only Bug A (dir→package-name resolution via cached `cargo metadata`); three remaining defect layers still break the gate: (B) `git diff --name-only` is blind to untracked files, so mandated e2e crates get dropped from scope; (C) the resolver's identity fallback emits invalid `-p <dir>` flags on name/asset drift; and (D) auto-deriving the test surface from a file diff is lossy for backend-wiring features, so the spec must declare its gate contract explicitly. This fix unions untracked files into the touched surface, hardens the resolver to NEVER emit an invalid `-p` (drop-on-mismatch, metadata-failure→workspace-wide), adds an optional `gate` contract (`packages`/`workspace`/`integration`) threaded from the specification stage through RunOptions into `runBuildGate` with documented precedence, and corrects the test suite that still encodes the original bug.

## Acceptance Criteria

- **AC-01**: Layer B — `detectTouchedCargoPackages` (src/build-runner.ts ~L405) UNIONS two sources before extracting `crates/<seg>/` segments: tracked changes `git -C <cwd> diff --merge-base <ref> --name-only` AND new/untracked files `git -C <cwd> ls-files --others --exclude-standard`. Both stdouts are concatenated, then the existing `CRATE_SEGMENT_RE` runs over the union. Either command failing (non-git / bad ref / spawn error / non-zero) contributes nothing rather than throwing; an untracked-only change under `crates/workflows/` still yields `workflows` in the scope.
- **AC-02**: Layer C — `resolveCargoPackageNames` (~L327) DROPS unresolved dirs instead of the current per-element identity fallback: a resolved name is EMITTED only if it exists in the `cargo metadata` member map; an unresolved dir is discarded. On `cargo metadata` failure/missing/non-zero/timeout/bad-JSON (`loadCargoMetadata` returns `{ok:false}`) the function returns `[]` (NOT the touched dir names), so the gate widens to workspace-wide rather than emitting invalid `-p <dir>`. The whole-list identity fallback and per-element identity fallback are both removed. Never throws.
- **AC-03**: Layer C — a final validation gate inside `runBuildGate` (or the scoped-argv builder, e.g. `scopedCargoArgs`): BEFORE constructing any `-p <name>` flag, confirm each candidate name is a known `cargo metadata` member; drop any that aren't; if the surviving set is empty → workspace-wide. This uniform check applies to ALL sources (spec-declared, env, auto-detect). The opaque cargo error (`package ID specification 'data' did not match any packages … similar name: dtoa`) can no longer occur. Never throws.
- **AC-04**: Layer D — the specification-stage schema/model in src/render/schemas.ts gains an OPTIONAL `gate` field: `gate?: { packages?: string[]; workspace?: boolean; integration?: string[] }`. Backward compatible: when `gate` is omitted (non-backend/trivial specs, or repos with no contract), behavior is byte-identical to today. The field lives on the specification-stage OUTPUT (SpecificationData), not on RequirementsData.
- **AC-05**: Layer D — the specification-stage prompt builder (src/stages/*.ts) is updated to INSTRUCT the agent to declare `gate` for backend/integration features: name the cargo packages whose tests must pass, whether workspace-wide is required, and any e2e/integration target paths or binaries. For non-backend/trivial specs the prompt permits omitting `gate` (auto-detect applies).
- **AC-06**: Layer D — `RunOptions` (src/types.ts ~L243) gains `gate?: { packages?: string[]; workspace?: boolean; integration?: string[] }`. implementation.ts / verify.ts / index.ts read `state.spec?.gate` and pass it through `RunOptions.gate` into `runBuildGate` opts. Naming collision note: this is the RunOptions interface in src/types.ts, NOT the control-flow `gate` node referenced elsewhere.
- **AC-07**: Layer D — precedence in `runBuildGate` (HIGHEST→lowest): spec-declared `gate.packages` → `SUPER_DEV_BUILD_TEST_PACKAGES` env → corrected auto-detect (`detectTouchedCargoPackages` with layers B+C) → workspace-wide. `gate.workspace === true` short-circuits to workspace-wide (ignores `packages`). `gate.integration` targets are APPENDED on top of whatever scope was chosen. When a declared `gate.packages` set is empty after validation (all unknown) → degrade to workspace-wide (consistent with AC-03).
- **AC-08**: Layer D — declared `gate.packages` are validated against the `cargo metadata` member map (same resolver); unknown names are DROPPED with a clear log line, never emitted as an invalid `-p`. `gate.integration` targets are appended to the test command (e.g. additional `cargo test --test <stem>` or path invocation) on top of the scoped packages.
- **AC-09**: Layer E — tests/build-runner-autoscope.test.ts is corrected: (1) mock `cargo metadata` to return a PREFIXED workspace (`crates/data`→`stockfan-data`, `crates/tools`→`stockfan-tools`, `crates/workflows`→`stockfan-workflows`) and assert `-p stockfan-data -p stockfan-tools -p stockfan-workflows` (NOT `-p data`); (2) add an untracked-file case where the ONLY `crates/workflows/` change is an untracked `tests/e2e_*.rs` (via `git ls-files --others`) and `stockfan-workflows` still appears in scope; (3) add a drop-unresolved case (a touched dir resolving to NO member → dropped, not emitted; all-drop → workspace-wide argv); (4) add a new spec-declared-gate-contract test: `RunOptions.gate = { packages: ['stockfan-data','stockfan-workflows'], integration: ['crates/workflows/tests/e2e_x.rs'] }` → gate uses the validated names + appends the integration target, ignoring auto-detect.
- **AC-10**: End-to-end stockfan shape: touched dirs `data`/`tools`/`workflows` with an UNTRACKED e2e under `crates/workflows/tests/` resolves to `cargo build/test/clippy -p stockfan-data -p stockfan-tools -p stockfan-workflows` across ALL THREE captured argvs, AND the `workflows` crate is included (its inclusion comes from the untracked-file union in AC-01).
- **AC-11**: `npm run typecheck` passes (strict-clean); `npm test` passes after the corrections in AC-09 — existing-corrected assertions plus the new spec-declared-gate-contract test all green.
- **AC-12**: No regression: non-cargo repos, non-git dirs, and repos with NO `gate` contract produce output byte-identical to today (auto-detect → workspace-wide). The inscope/out-of-scope classification (`partitionErrorsByScope` / `isGreenForScope`) still works. The DEFAULT gate strategy stays auto-detect (only `gate.workspace===true` or a failed resolution widens to workspace); the default is NOT switched to workspace-wide.

## Non-Functional Requirements

- Performance: `cargo metadata --format-version 1 --no-deps` remains cached per-absolute-cwd (existing cache at ~L171). The only new spawn is a single `git ls-files --others --exclude-standard`, reusing the existing timeout envelope; no new runtime deps. The empty-input guard is preserved so metadata is spawned ONLY when there is something to resolve.
- Security/Robustness: NEVER throw from the git helpers, the metadata resolver, the validator, or the gate. All git/metadata spawns use single discrete-argv `spawnSync` with no `shell:true`, so package/path data never reaches a shell. Any failure degrades to workspace-wide (drop invalid → widen), never to an invalid `-p`.
- Backward compatibility: the change is confined to src/build-runner.ts (B+C), src/types.ts RunOptions + src/render/schemas.ts SpecificationData + spec prompt + the stage call sites (D), and tests (E). nodes.ts/workflow.ts/pipeline.ts/render templates/control-flow are NOT touched.
- Theme method-binding constraint (must NOT regress): the real pi `Theme` is a CLASS — `fg(color,text){ return this.fgColors.get(color)... }`. Any prompt/render change that touches theme MUST call method-style (`theme.fg(...)`), NEVER destructure (`const fg = theme.fg; fg(...)` detaches from `this` and throws 'reading fgColors'). Guarded by tests/stream-theme-class-theme.test.ts.

## Open Questions

- gate.integration entry format: the example shows a PATH ('crates/workflows/tests/e2e_screen_us_fallback.rs') but the prose says 'additional cargo test --test <stem> invocations'. Are integration entries PATHS (derive stem = basename minus '.rs', emit `cargo test --test <stem>` under the package scope) or pre-named test targets? Recommend: accept either — if an entry ends in '.rs', derive the stem via basename-without-extension; otherwise treat it as a named target. Confirm with the implementer before finalizing AC-08.
- When declared `gate.packages` is non-empty but ALL fail metadata validation: should we fall through to env → auto-detect → workspace-wide, or short-circuit straight to workspace-wide? Recommend straight-to-workspace-wide (AC-03/AC-07 assume this) for consistency with 'never emit an invalid -p'. Confirm.
- Defense-in-depth uniformity (AC-03): should the env var `SUPER_DEV_BUILD_TEST_PACKAGES` also pass through the final metadata-validation gate (so a stale env name is dropped rather than emitted)? Recommend YES — validate every candidate name regardless of source. Confirm; if no, the env path bypasses the guard and the 'NEVER emit invalid -p' guarantee has a hole.
- Where should the 'dropped unknown package' log line go — `console.warn`, or an existing structured logger convention in src/build-runner.ts? Minor, but confirm the repo's logging convention so AC-08's 'clear log line' is well-defined.
- Confirm the gate contract is read from the PARSED specification-stage output (SpecificationData via STAGE_MODELS['specification']/STAGE_MODELS['spec'] at src/render/schemas.ts ~L233), not from the raw requirements markdown. The RequirementsData schema (~L73) does NOT carry a gate field — only the specification stage does.
