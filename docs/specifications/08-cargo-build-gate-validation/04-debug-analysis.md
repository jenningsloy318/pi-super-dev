# Debug Analysis: Root-Cause Analysis: cargo build-gate emits invalid `-p <dir>` and drops untracked crates (Layers B–E)

- **Date**: 2026-07-20

---

## Summary

The scope-aware cargo build gate (spec-07, already on `main`) fixed only the directory→package-name mapping for the prefix-as-name case (Bug A). Three remaining defects plus a self-encoding test suite keep the gate broken in the real shape that triggered this work (the `stockfan` workspace, where directories `data`/`tools`/`workflows` map to packages `stockfan-*`):

1. UNTRACKED BLINDNESS (Layer B): `detectTouchedCargoPackages` (src/build-runner.ts:405) only spawns `git diff --merge-base <ref> --name-only`. `git diff` is invisible to untracked files (`??`). An untracked `crates/workflows/tests/e2e_screen_us_fallback.rs` therefore contributed nothing, so `workflows` (and its mandated e2e) was dropped from scope.

2. IDENTITY FALLBACK EMITS INVALID `-p` (Layer C): `resolveCargoPackageNames` (src/build-runner.ts:327) has TWO identity fallbacks that re-introduce the original bug — `!meta.ok` returns `dedupePreservingOrder(strDirs)` (whole-list identity, ~L343), and a per-element unmatched dir returns the dir NAME itself (`out.push(matched ? matched.name : d)`, ~L354). For `stockfan` this emits `-p data` (no such package), reproducing the opaque cargo error `package ID specification 'data' did not match any packages … similar name: dtoa`. There is also NO membership validation inside `runBuildGate` before names become `-p` flags.

3. NO SPEC-DECLARED GATE CONTRACT (Layer D): `SpecificationData` (src/render/schemas.ts:221) and `RunOptions` (src/types.ts:243) have no `gate` field. The test surface is auto-derived purely from a file diff, which is lossy for backend-wiring features (acceptance = workspace integration suite + an e2e cannot be inferred from changed files). Call sites implementation.ts:84 / index.ts:53 / verify.ts:87 never read `state.spec?.gate`.

4. TEST SUITE ENCODES THE BUG (Layer E): `tests/build-runner-autospace.test.ts` `routeSpawn` (L84–90) routes EVERY non-`git` command — including `cargo metadata` — to `{status:0, stdout:"", stderr:""}`. Empty stdout makes `loadCargoMetadata` return `{ok:false}` (L256), which triggers the whole-list identity fallback, so the resolver returns `["data"]`. The test then ASSERTS `-p data`, locking the bug in green.

The diagnosis and four-layer fix are already specified by the task ACs; this analysis confirms the exact code locations, the causal chain, and a deterministic reproduction. The single dominant root cause is the IDENTITY FALLBACK semantics: the resolver's failure mode is "guess the dir name" instead of "drop / widen to workspace-wide," and nothing downstream re-validates against the member map.

## Hypotheses

- H1 (85%) — IDENTITY FALLBACK is the root: `resolveCargoPackageNames` returns directory NAMES on both whole-list failure (`!meta.ok` → `dedupePreservingOrder(strDirs)`, build-runner.ts:343) and per-element miss (`matched ? matched.name : d`, ~L354), and `runBuildGate` emits those names as `-p` flags with NO membership check (testPackages flows straight into scopedCargoArgs, runBuildGate ~L855). Prediction: for a prefixed workspace the gate emits `-p data` and cargo rejects it with 'did not match any packages … similar name: dtoa'. FALSIFIABLE: if a member-map validation gate existed before argv construction, no invalid `-p` could ever be emitted regardless of resolver output.
- H2 (90%) — UNTRACKED BLINDNESS independently drops the workflows crate: `detectTouchedCargoPackages` (build-runner.ts:405) spawns ONLY `git diff --merge-base <ref> --name-only` (~L419). Prediction: a worktree whose ONLY change under `crates/workflows/` is an untracked file produces an empty `git diff` stdout, so `workflows` never enters the touched set even though `git status` shows `?? crates/workflows/tests/e2e_*.rs`. FALSIFIABLE: unioning `git ls-files --others --exclude-standard` would surface the untracked path and the segment regex would extract `workflows`.
- H3 (88%) — TEST MOCK ACCIDENTALLY ENCODES THE BUG: `routeSpawn` (build-runner-autoscope.test.ts:84) returns `{status:0, stdout:''}` for all non-`git` commands, so the mocked `cargo metadata` call yields `{ok:false}` and the whole-list identity fallback fires — the test's `-p data` assertion only passes BECAUSE it exercises the failure path. Prediction: correcting the resolver to drop unresolved names would make these assertions fail without test updates (Layer E). FALSIFIABLE: a prefixed-metadata fixture (`crates/data`→`stockfan-data`) would require asserting `-p stockfan-data`.
- H4 (70%) — SECONDARY: even with H1/H2 fixed, auto-detect is structurally lossy for backend-wiring features whose acceptance criteria (workspace integration suite + an e2e) cannot be derived from a file diff. This is what H1/H2 paper over but don't truly fix; the spec-declared `gate` contract (Layer D) is the real remediation. Prediction: any backend feature needing `cargo test --test <e2e>` in addition to scoped packages will be under-tested without an explicit contract.
- H5 (low, rejected as the cause) — theme/method-binding regression detaches `this`. Rejected: no Theme rendering is on this code path; tests/stream-theme-class-theme.test.ts guards it and the bug reproduces without any theme involvement.

## Root Cause

VERIFIED ROOT CAUSE — the resolver's failure semantics are "guess the directory name" rather than "drop / widen to workspace-wide," and the gate never re-validates emitted names against the cargo member map. Three concrete, independently-confirmable defects, each pinned to exact code:

1. IDENTITY FALLBACK (dominant, build-runner.ts:327 `resolveCargoPackageNames`):
   - whole-list fallback: `if (!meta.ok) return dedupePreservingOrder(strDirs);` (~L343) returns directory NAMES verbatim on any metadata failure;
   - per-element fallback: `out.push(matched ? matched.name : d);` (~L354) returns the directory NAME for an unmatched segment.
   Both reproduce `-p data` for `stockfan`. The fix (Layer C) replaces both: return `[]` on `!meta.ok`, and DROP unmatched segments instead of pushing `d`.

2. NO VALIDATION GATE (build-runner.ts `runBuildGate` ~L825–855): `testPackages` from any tier flows straight into `scopedCargoArgs`/`scopedCargoBuildArgs`/`scopedCargoTestArgs`/`scopedCargoClippyArgs` with no membership check. The fix adds a validation pass before argv construction: confirm each name is a known member, drop unknowns, and widen to workspace-wide if the surviving set is empty.

3. UNTRACKED BLINDNESS (build-runner.ts:405 `detectTouchedCargoPackages`, ~L419): only `git diff --merge-base <ref> --name-only` is spawned; `git diff` does not see untracked files. The fix (Layer B) concatenates `git -C <cwd> ls-files --others --exclude-standard` and runs the existing `CRATE_SEGMENT_RE` over the union; either command failing contributes nothing (never throws).

CONFIRMING EVIDENCE the test green-lit the bug: `routeSpawn` (tests/build-runner-autoscope.test.ts:84–90) returns `{status:0, stdout:''}` for every non-`git` command, so the mocked `cargo metadata` produces empty stdout → `loadCargoMetadata` returns `{ok:false}` (build-runner.ts:256, the `out === ""` branch) → whole-list identity fallback → `["data"]` → `-p data`. The test asserts exactly `-p data`, encoding the defect. Layer E replaces this with a prefixed-metadata fixture asserting `-p stockfan-data -p stockfan-tools -p stockfan-workflows` plus untracked-file, drop-unresolved, and spec-declared-contract cases.

RECOMMENDED FIX: apply Layers B+C to src/build-runner.ts (union untracked files; drop unresolved; metadata-failure→[]; add the membership validation gate before argv construction); add the optional `gate?: { packages?; workspace?; integration? }` to SpecificationData (src/render/schemas.ts:221) and RunOptions (src/types.ts:243); instruct the spec prompt to declare `gate` for backend features; thread `state.spec?.gate` through implementation.ts:84 / index.ts:53 / verify.ts:87 into `runBuildGate`; implement precedence spec→env→auto-detect→workspace-wide with `gate.workspace` short-circuit and `gate.integration` appended; correct tests/build-runner-autoscope.test.ts (Layer E). All four fixes degrade silently (never throw) and are byte-identical for non-cargo / non-git / no-contract repos. REGRESSION TEST: the new autoscope cases (prefixed fixture, untracked-only workflows change, all-unresolved→workspace-wide, and the spec-declared contract test) plus `npm run typecheck` strict-clean.
