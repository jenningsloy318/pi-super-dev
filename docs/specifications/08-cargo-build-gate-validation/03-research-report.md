## Summary

All four blocking open issues from the prior attempt are resolved with freshly-fetched sources (this run reached git-scm.com and doc.rust-lang.org via curl; firecrawl is absent but direct web fetch worked). (1) Citation pass complete — git-diff man confirms `--merge-base <commit>` resolves to the merge base of commit & HEAD and `git diff <commit>` compares the WORKING TREE to that commit (so it captures staged files present on disk) [SRC-01]; git-ls-files man confirms `--others` = untracked and `--exclude-standard` honors .gitignore [SRC-02]; cargo-metadata man confirms `packages[]` under `--no-deps` = "workspace members only, don't fetch dependencies" (resolve is null) [SRC-03]; cargo-test man confirms `--test NAME` is target-scoped while the positional `name_filter` is a substring filter across all targets, NOT isolated [SRC-04]. (2) Staged-but-uncommitted files are already captured by `git diff --merge-base` for the common case (staged file is present in the working tree); the only blind spot is staged-then-worktree-reverted, which never occurs in this commit-per-stage pipeline, so we DOCUMENT the committed/worktree-present assumption rather than adding a third `git diff --cached` spawn (which would violate the "git ls-files --others is the only new process" constraint). (3) `gate.integration` entries are emitted as `cargo test --test <stem>` (target-scoped, canonical Cargo Book form), with the stem DERIVED deterministically from the declared value (basename, strip `.rs`/path), then VALIDATED by a filesystem stat of `<memberDir>/tests/<stem>.rs` before emitting — a miss is dropped with a log line (never emits cargo's `test target 'x' not found` error). (4) The `cargo metadata --no-deps` member map contains ONLY workspace members by spec, so the validation set is workspace-members-only by design; path-dependencies are intentionally dropped because building/testing an arbitrary graph node is not a meaningful gate target.

## Options Considered

### Touched-surface source set: union `git diff --merge-base` + `git ls-files --others` (RECOMMENDED)

Two spawns. Covers committed-vs-base, worktree-present-staged changes, AND untracked files — the union closes the stockfan blind spot (untracked e2e under crates/workflows/). Matches the constraint that `git ls-files --others` is the only NEW process (diff --merge-base already exists in spec-07). Misses ONLY the staged-but-worktree-reverted case, which cannot occur in a commit-per-stage pipeline — documented as an explicit assumption rather than closed with a 3rd spawn. Either command failing contributes nothing (never throws). ALTERNATIVE considered: add `git diff --cached --merge-base <ref>` as a 3rd source — closes the theoretical gap but adds a spawn and contradicts the explicit process-count constraint; deferred to a documented future-hardening note if mid-implementation gating is ever needed.
### Integration invocation: `cargo test --test <stem>`, stem derived from declared path, validated via fs stat (RECOMMENDED)

`--test NAME` is target-scoped and canonical (SRC-04); deterministic and isolated. Stem derivation rule: declared value ending in `.rs` or containing a path separator -> basename minus `.rs`; bare token -> used as-is. Validation: `exists(join(memberDir,'tests',stem+'.rs'))` (filesystem stat, NOT a spawn) before emitting; miss -> drop with a log line so cargo's `error: test target 'x' not found` can never surface. Globs (`*?[]`) supported by cargo natively if ever needed. ALTERNATIVES rejected: (b) `cargo test <path>` positional — it's a substring NAME filter across ALL targets, not target-scoped, so it can accidentally run unrelated tests and does not isolate the target; (c) both-via-pattern-detection — more code paths and ambiguous edges with no real benefit since (a) subsumes the valid cases.
### Validation set: workspace members only, from cached `cargo metadata --no-deps` (RECOMMENDED)

SRC-03 confirms `packages[]` under `--no-deps` = workspace members only (`resolve` is null, path-deps/external deps excluded). Reuses the existing per-cwd cached spawn (no new process). A touched dir that resolves to a path-dependency rather than a workspace member returns no match -> DROPPED by Layer C, which is the correct gate semantics: building/testing an arbitrary reachable graph node is not a meaningful gate target. Cargo WOULD accept `-p <path-dep-name>`, but emitting it would be surprising and not what a gate intends, so workspace-members-only is the right, defensive choice. ALTERNATIVE: validate against the full graph (drop `--no-deps`) — heavier spawn, includes all transitive deps, and the broader set has no gate value; rejected.
### Unresolved-dir handling: DROP and widen-to-workspace-wide on empty (RECOMMENDED) vs identity fallback

The current identity fallback (`resolveCargoPackageNames` emits the dir name when no member matches) is the root cause of the stockfan failure — it produced invalid `-p data` that cargo rejected. Layer C replaces it: a resolved name is EMITTED only if present in the member map; unresolved dirs are discarded; metadata failure (`{ok:false}`) returns `[]` -> workspace-wide. `runBuildGate`/`scopedCargoArgs` re-validates every candidate from EVERY source (spec, env, auto-detect) before constructing `-p`, so future naming drift degrades silently to workspace-wide instead of throwing. Never throws from the resolver, validator, or gate (degrade: drop invalid -> workspace-wide). Backward-compatible: no-gate-contract / non-cargo / non-git repos produce byte-identical output to today.
