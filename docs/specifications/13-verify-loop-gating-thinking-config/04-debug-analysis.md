# Debug Analysis: Verify-loop gating & per-agent thinking config: root-cause analysis of 4 loop-gating gaps + missing thinking configuration

- **Date**: 2025-02-14

---

## Summary

This is a defect-cluster in the pi-super-dev pipeline's verify/integration stages (src/stages/verify.ts) plus a missing-feature in the spawn layer (src/pi-spawn.ts, src/session-agent.ts). Five independent root causes were confirmed by reading the source. (A) Stage 11 `integrationLoopNode` is a hand-rolled `for`-loop with no stagnation/oscillation detection — unlike Stage 10 which has `findingsSignature`+`reviewLoopUntil`+`__stagnated`; a run that keeps producing the same test failures burns all 3 rounds with no early break and no `__testStagnated` record. (B) `reviewLoopUntil` exits on `reviewApproved(s)` alone (line 138) and never consults the deterministic `buildGreen(s)` predicate (defined line 42 but only used inside the retry body), so the loop can terminate "successfully" while the build gate is red. (C) Stagnation is detected ONLY by byte-identical signatures (`hist[last] === hist[last-1]`), so a 5→5 plateau or 5→6 scope-drift with differing signatures never triggers a break. (D) At max-rounds exhaustion the loop returns after the LAST iteration ran `reviewStep → fixStepReview → buildGateStep`, meaning the terminal fix from `fixStepReview` is committed but never re-reviewed — `state.review` is stale when the downstream `reviewApproved` merge gate reads it. (Thinking) Neither backend sets any pi thinking level: `buildSpawnArgs` never appends `--thinking`, `runAgentViaSession` never calls `setThinkingLevel`, and there is no `thinkingForAgent` role map analogous to `isCodeWritingAgent`. The fixes are additive, must stay non-fatal (never throw), and must preserve all 1387 tests.

## Hypotheses

- ROOT (100%): Five distinct, independently-verifiable defects — 4 loop-gating gaps in src/stages/verify.ts and 1 missing thinking-config feature spanning src/pi-spawn.ts + src/session-agent.ts + workflow.ts. Each is falsifiable by reading the specific lines cited below.
- A (confirmed): integrationLoopNode (verify.ts ~L233) is a bespoke for(attempt=1..2) loop that only checks `testsGreen(state) && reviewApproved(state)` and lacks any signature tracking. PREDICTION: no reference to __testSignatures/__testStagnated exists in the file — VERIFIED (grep returned none).
- B (confirmed): reviewLoopUntil ends with `return reviewApproved(s)` (verify.ts L138); buildGreen (L42) is invoked only inside the fix/build body, never as an exit precondition. PREDICTION: an approved-but-build-red state exits the loop today — VERIFIED by the return expression ignoring buildGate.
- C (confirmed): stagnation test is `sig !== '' && hist.length>=2 && hist[last]===hist[last-1]` (byte-identical only). PREDICTION: a non-decreasing findings COUNT with differing signatures (5→5 reorder, or 5→6) does NOT trip stagnation today — VERIFIED; no count comparison exists.
- D (confirmed): on exhaustion the loop returns after the body's last reviewStep→fixStepReview→buildGateStep; the terminal fix is never re-reviewed. PREDICTION: state.review reflects the pre-terminal-fix review when the exhaustion path returns — VERIFIED by loop structure (fix runs after the final review).
- Thinking (confirmed): buildSpawnArgs (pi-spawn.ts L112) argv omits --thinking; runAgentViaSession (session-agent.ts L329, createAgentSession L349) never calls setThinkingLevel; no thinkingForAgent exists. PREDICTION: grep for 'thinking' in both files returns nothing — VERIFIED.
- Rejected: 'the max-round cap must change to fix stagnation' — false; AC-01 explicitly forbids changing the 3-round cap, and an early break inside the existing cap suffices.
- Rejected: 'exit paths may throw to abort the pipeline on unrecoverable state' — false; CONSTRAINTS mandate every new loop-exit path stay non-fatal (log + return, never throw).

## Root Cause

The verify stage evolved Stage 10's review loop (with signature-based stagnation via findingsSignature/reviewLoopUntil/__stagnated) but Stage 11's integration loop was written earlier as a raw for-loop and never received the equivalent guard (GAP A). Stage 10's own exit condition was written before a deterministic build gate existed, so `reviewLoopUntil` returns `reviewApproved(s)` without ANDing `buildGreen(s)` — allowing green-review/red-build exits (GAP B). Stagnation detection was implemented as a cheap byte-identity check on signatures, which misses count-plateaus and scope-drift where signatures differ (GAP C). The loop body orders fix AFTER review, so on exhaustion the terminal fix is unreviewed and the downstream `reviewApproved` merge gate reads stale `state.review` (GAP D). Separately, the spawn layer was never given a thinking-level knob: there is no `thinkingForAgent` role map, `buildSpawnArgs` emits no `--thinking` flag, and `runAgentViaSession` never calls `setThinkingLevel`, so all specialists run at the model default regardless of whether they are reasoning-heavy (design/spec/reviewers) or mechanical (commit/cleanup) (THINKING). Recommended fixes: (A) add a `testSignature(s)` (sorted api+ui failures) + track in `state.__testSignatures`, break early on repeat setting `state.__testStagnated{rounds,signature,failures≤12}`, log non-fatal, cap unchanged. (B) change `reviewLoopUntil` success return to `reviewApproved(s) && buildGreen(s)`, keeping the stagnation short-circuit ahead of it. (C) in both loops, also treat as stagnant when current findings/failures count >= previous count across 2 rounds (non-decreasing), OR-ed with the identical-signature trigger. (D) after the Stage 10 loop exits via exhaustion (not approval, not stagnation), run exactly one final budget-checked `reviewStep` (no fix) so `state.review` is fresh; keep non-fatal. (THINKING) add `thinkingForAgent(agent)` in pi-spawn.ts (design/spec-writer/adversarial-reviewer/code-reviewer/debug|debugger/assessment→'high'; implementer/tdd-guide→'medium'; commit/orchestrator-commit/slug/cleanup→'minimal'|'off'; else 'medium'), resolve precedence per-call `thinking` > `SUPER_DEV_THINKING` env > role default, append `--thinking <level>` in buildSpawnArgs, thread `thinkingLevel` through SessionAgentOptions and call `session.setThinkingLevel(level)` in a try/catch after createAgentSession; add unit tests for mapping, both override precedences, and the --thinking arg.
