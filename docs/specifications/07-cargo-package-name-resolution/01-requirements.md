# Requirements: Resolve real cargo package names in the scope-aware build-gate (and forbid --lib-only self-verification)

- **Date**: 2026-07-20
- **Author**: super-dev:requirements-clarifier
- **Type**: bug-fix
- **Priority**: critical
- **Status**: draft

---

## Executive Summary

The scope-aware build gate in src/build-runner.ts derives cargo `-p` flags from workspace DIRECTORY names (e.g. `crates/data/` → `data`) instead of real package names (e.g. `stockfan-data`), so `cargo build -p data` fails instantly with "package ID specification `data` did not match any packages" on any workspace that prefixes crate names — false-failing every gate attempt and aborting the run before anything compiles. Fix 1 adds a `cargo metadata`-backed `resolveCargoPackageNames(cwd, touchedDirs)` that maps dir→real package `name` (with a documented dir-name fallback and a never-throw + per-cwd cache), wired into `detectTouchedCargoPackages`; Fix 2 confirms the already-complete touched-set extraction now yields correct names; Fix 3 strengthens the implementation/QA prompt builders (in src/prompts.ts, invoked by src/stages/implementation.ts and verify.ts) to forbid `--lib`-only verification and require full `cargo test -p <pkg>` plus spec-mandated e2e/integration targets.

## Acceptance Criteria

- **AC-01**: src/build-runner.ts exports `resolveCargoPackageNames(cwd, touchedDirs: string[]): string[]` that runs `cargo metadata --format-version 1 --no-deps --manifest-path <cwd>/Cargo.toml` (bounded by the existing timeout envelope, via discrete argv `spawnSync`, no `shell:true`), parses the JSON, and maps each touched `crates/<dir>/` segment to the workspace package whose `manifest_path` parent directory equals `crates/<dir>/` (compare the directory CONTAINING the package's Cargo.toml, supporting manifest-in-subdir), returning the real package `name`s deduped and in first-seen order.
- **AC-02**: AC-02 — Never-throw + fallback chain: the metadata helper is wrapped in a defensive try/catch that NEVER throws; on spawn error / non-zero exit / timeout / missing cargo / no match, it falls back to returning the input touched directory names verbatim (identity), so workspaces where dir==name and non-cargo dirs degrade to today's behavior identically. The fallback is documented in a JSDoc comment.
- **AC-03**: AC-03 — Per-cwd in-memory cache: a module-level `Map<cwd, MetadataResult>` memoizes the parsed `cargo metadata` output so a single gate run that calls the resolver multiple times spawns `cargo metadata` at most once per cwd. The cache is process-local (cleared on process exit; never persisted across runs to avoid stale results).
- **AC-04**: AC-04 — `detectTouchedCargoPackages(cwd, baseRef?)` keeps its existing `git diff --name-only` spawn + regex `/(?:^|\/)crates\/([^/]+)\//` + dedupe logic, but passes the collected directory segments through `resolveCargoPackageNames(cwd, dirs)` as the FINAL step so it returns REAL package names; on the stockfan shape (dirs data/tools/workflows) it returns `['stockfan-data','stockfan-tools','stockfan-workflows']`, producing `cargo build/test/clippy -p stockfan-data -p stockfan-tools -p stockfan-workflows` end-to-end via the unchanged `scopedCargoArgs`/`scopedCargoBuildArgs`/`scopedCargoTestArgs`/`scopedCargoClippyArgs`.
- **AC-05**: AC-05 — Complete touched set: a touched `crates/workflows/tests/e2e_*.rs` makes `stockfan-workflows` appear in the resolved scope (regex already captures the `workflows` segment; this AC asserts the dir→name mapping covers it, so spec-mandated integration/e2e tests in a touched crate run via `cargo test -p stockfan-workflows`).
- **AC-06**: AC-06 — New unit tests (hermetic, `node:child_process.spawnSync` mocked) covering: (a) dir→name resolution against a stubbed `cargo metadata` JSON with prefixed names + a manifest-in-subdir package; (b) metadata failure (spawn error / non-zero exit / timeout) falls back to the dir-name identity list; (c) a cache-hit scenario where the metadata spawn mock is called only once across two resolver calls for the same cwd; (d) an end-to-end workspace where dir != name produces correct `-p` flags through `scopedCargoArgs`.
- **AC-07**: AC-07 — Fix 3 (prompt-level): the implementation/verify self-verification prompt builders (the authoritative text lives in src/prompts.ts — `buildImplementPrompt` and the QA/verify prompt invoked by src/stages/implementation.ts and src/stages/verify.ts) are strengthened to REQUIRE `cargo test -p <pkg>` with NO `--lib` flag (so integration binaries under `tests/` run), PLUS any spec-mandated e2e/integration target, and to include an explicit instruction: do NOT declare green on `--lib`-only evidence; `--lib` skips the `tests/` integration binaries. This is a prompt-text change only — no control-flow or nodes/workflow/pipeline change.
- **AC-08**: AC-08 — No regression: all existing scope-aware tests pass unchanged (build-runner-touched-crates, build-runner-autoscope, build-runner-inscope-classification, build-runner-scoped-args, build-runner-packages, build-runner-nonregression, build-runner-timeout, build-runner-docs); non-cargo repos (go/python/node/mixed) and non-git dirs remain byte-identical (the metadata tier runs only when `language === 'rust'` and a non-empty scope resolves, exactly as today).
- **AC-09**: AC-09 — Theme method-binding preserved: any new rendering uses method-style `theme.fg(...)` (or a wrapper `const fg=(c,t)=>theme.fg(c,t)`); NEVER destructures `theme.fg`. No new theme tokens are introduced and the rendering layer is untouched. (tests/stream-theme-class-theme.test.ts must still pass.)
- **AC-10**: AC-10 — `npm run typecheck` passes (strict-clean) and `npm test` passes (existing + new) with no new runtime dependencies; the only new spawned process is `cargo metadata --no-deps --manifest-path ...`, cached per cwd per run.

## Non-Functional Requirements

- Performance: `cargo metadata --no-deps` is the single new spawned process, bounded by the existing timeout envelope, and memoized per cwd so a gate run spawns it at most once; cache is process-local (never persisted) to avoid staleness across runs.
- Reliability / fault tolerance: the metadata resolver and the gate itself NEVER throw — every failure path (spawn error, non-zero exit, timeout, missing cargo, malformed JSON, no matching package) degrades gracefully to the documented dir-name identity fallback, preserving today's safe `[]`/workspace-wide semantics; a non-cargo or non-git repo behaves identically to before.
- Correctness / no-false-green: Fix 3 is prompt-only and must not change control flow; the `--lib` prohibition removes a vacuous-green vector (declaring success on unit tests only) without weakening the deterministic gate's out-of-scope classifier (which still never grants a false green).
- Backward compatibility: dir==name workspaces get an identity mapping (no-op), so existing tests and real dir==name monorepos are byte-identical; the scoped-vs-workspace-wide gate STRATEGY is unchanged (touched-crate scoping remains default).
- Security: package names flow as discrete argv elements through `spawnSync` with no `shell:true` (matching the existing spawn pattern in detectTouchedCargoPackages/runBuildGate), so no metadata-derived string ever reaches a shell.
- TypeScript strict-clean + zero new runtime dependencies; changes confined to src/build-runner.ts (Fix 1+2) and prompt text in src/prompts.ts (Fix 3) plus new tests — nodes.ts/workflow.ts/pipeline.ts and the rendering layer are untouched.

## Open Questions

- Fix 3 location: the authoritative prompt text lives in src/prompts.ts (buildImplementPrompt + the QA/verify prompt), which the stages merely call — not inline in src/stages/implementation.ts/verify.ts as the task brief implied. Confirm the `--lib` prohibition should be added in src/prompts.ts (recommended: it is the single source of truth the stages consume) rather than re-stated in each stage.
- Fix 2 scope: the existing regex `/(?:^|\/)crates\/([^/]+)\//` ALREADY captures every touched crate segment (including `workflows` from `crates/workflows/tests/e2e_*.rs`), so no extraction fix is required — Fix 2 is fully delivered by Fix 1's dir→name conversion. Confirm there is no separate single-crate edge case to handle (e.g. root-level workspace Cargo.toml edits touching ALL packages).
- Should `resolveCargoPackageNames` also be exported for unit testing even if `detectTouchedCargoPackages` is the only internal caller? (Recommended: yes, export it so AC-06 tests can target it directly without going through git diff.)
- Cache invalidation: confirm a process-local Map is acceptable (recommended: yes — a gate run is a single process and cross-run staleness is the real risk); no TTL or fs-based cache is needed.
- Metadata matching granularity: confirm matching on the directory CONTAINING the package's Cargo.toml (`manifest_path` parent dir) vs the package `name` itself — recommended to match directory, since the bug is precisely that directory != name and the mapping must bridge them.
