# Adversarial Review: Adversarial Review: Cargo Package Name Resolution for Scope-Aware Build Gate

- **Date**: 2026-07-20
- **Reviewer**: super-dev:adversarial-reviewer
- **Verdict**: CONTEST

---

The core fix is correct and well-executed: `resolveCargoPackageNames` maps touched directory segments to real cargo package names via cached `cargo metadata --no-deps` (discrete argv, no shell), with a documented, never-throw fallback chain (empty→[], metadata-fail→whole-list identity, unmatched-segment→per-element identity) and a process-local per-cwd cache that also memoizes the `{ok:false}` failure sentinel. It is correctly wired as the final step of `detectTouchedCargoPackages` (git-diff + regex + dedupe byte-identical), so the stockfan shape emits `cargo build/test/clippy -p stockfan-data -p stockfan-tools -p stockfan-workflows`, and the `crates/workflows/tests/e2e_*.rs` set is preserved (Fix 2). Fix 3 lands prompt-only discipline in `buildImplementPrompt`/`buildQaPrompt`, correctly gated on `setup.language === "rust"` (not broadcast to non-rust stacks — a review-fix the author applied beyond the spec's literal text). The `classificationScope` augmentation is conservative (only adds directory segments to the in-scope set, making false-green strictly harder) — a thoughtful guard against a HIGH-severity regression. Verification: `npm run typecheck` clean; `npm test` 862/862 green across all 47 files including the new resolution/wiring/prompt suites and the full backward-compat/nonregression/autoscope/touched-crates/scoped-args suites; no new runtime deps; only new spawn is the cached `cargo metadata`. The verdict is CONTEST rather than PASS because of one design-level blind spot (nested crates sharing a top segment can be silently mis-resolved, regressing the old safe-fail into a possible false-green) plus several lower-severity nits; none rise to REJECT (no production-failure/data-loss/security breach in the common layouts the spec targets).

### ADV-01: Nested-crate shadowing: two packages sharing a top `crates/<seg>/` segment can be mis-resolved, regressing a safe false-FAIL into a possible false-GREEN

- **Severity**: medium
- **Lens**: Skeptic
`matchPackageBySegment` (src/build-runner.ts, ~lines 295-315) resolves a touched segment to the package whose FIRST `crates/<seg>/` segment equals it, preferring an exact crate root over a nested one. Consider a workspace with `crates/data/Cargo.toml` (package `core-data`) AND `crates/data/extra/Cargo.toml` (package `extra-data`, a nested crate under `data/`). A change to `crates/data/extra/src/lib.rs` is captured by the git-diff regex as segment `data`, which `matchPackageBySegment` resolves to `core-data` (the exact-root winner). The gate then runs `cargo build/test -p core-data` and NEVER builds `extra-data`, the crate that actually changed → `extra-data`'s tests silently don't run → potential false-green (inScopePass=true on a clean core-data build). This is a genuine regression vs the PRE-fix behavior: the old code returned the directory name `data`, which cargo rejected with exit 101 → safe false-FAIL (run aborts). The fix trades a safe abort for a possible silent miss in this exotic-but-legal layout. The spec's SCENARIO-002 sanctions segment-based matching for a SINGLE nested package, but never contemplates two packages sharing a top segment, and there is no test covering the two-package-same-segment case. Recommendation: either (a) when ≥2 packages share the touched top segment AND none is an exact root, conservatively expand the scope to ALL of them (safe over-build, never a miss), or (b) fall back to identity (dir name) so cargo rejects it (safe-fail) rather than guessing. At minimum add a regression test for the shadowing layout.
### ADV-02: Dead `===` branch in matchPackageBySegment — exact-root check is unreachable for real cargo metadata

- **Severity**: low
- **Lens**: Minimalist
`if (p.manifestDir === \`crates/${seg}\` || p.manifestDir.endsWith(\`/crates/${seg}\`))` — `cargo metadata` always emits ABSOLUTE `manifest_path` values, so `dirname(...)` is absolute (e.g. `/repo/crates/data`). The left operand `manifestDir === 'crates/data'` (relative) can therefore never be true; only the `endsWith('/crates/data')` branch ever fires. The `===` half is effectively dead code for real metadata. Not a correctness bug (the `endsWith` covers it), but it misleads readers into thinking the relative form is reachable. Recommendation: drop the `===` half and rely solely on `endsWith`, or normalize `manifestDir` to relative-first. (Minor Windows-portability note: `endsWith('/crates/...')` assumes forward slashes; `cargo metadata` does emit forward-slash-normalized paths cross-platform via Utf8Path Display, so this is safe in practice, but worth a one-line comment.)
### ADV-03: Cache key uses `path.resolve` which does NOT dereference symlinks → duplicate `cargo metadata` spawns on aliased worktree paths

- **Severity**: low
- **Lens**: Architect
`loadCargoMetadata` keys the cache on `absCwd = resolve(cwd)` (src/build-runner.ts, ~line 232). `path.resolve` normalizes `.`/`..` and makes the path absolute, but it does NOT resolve symlinks (that requires `fs.realpathSync`). If the same physical worktree is addressed via two different paths in one process (e.g. a canonical path and a symlinked alias, or a `/tmp` vs real-path indirection common in CI/worktree harnesses), the cache misses and `cargo metadata` is spawned once per alias — wasting the per-run-once invariant the spec's SCENARIO-005 promises. No correctness impact (each spawn returns the same data), just a redundant spawn. Recommendation: key on `fs.realpathSync(cwd)` inside the existing try/catch (fall back to `resolve(cwd)` if realpath throws), or document the assumption that the caller passes a stable canonical cwd.
### ADV-04: Failure sentinel cannot distinguish transient from permanent failures → a one-off cargo-metadata hiccup reverts the whole run to the original dir-name bug

- **Severity**: low
- **Lens**: Skeptic
`loadCargoMetadata` caches `{ok:false}` for the life of the process on ANY failure (SCENARIO-018 intent: don't re-spawn a missing/broken cargo). This is correct for PERMANENT failures (cargo not installed, corrupt manifest). But a TRANSIENT failure on the FIRST gate of a run — a momentary spawn error, a one-off 30s timeout under load, a fleeting registry/lock hiccup — poisons the cache for every subsequent gate in the SAME pi-super-dev run (verify, implementation, merge). From that point on, `resolveCargoPackageNames` degrades to directory names, and on a dir≠name workspace the gate re-enters the EXACT original bug (`cargo build -p data` → exit 101 → false-fail) for the rest of the run, with no self-heal short of process restart. `cargo metadata --no-deps` is genuinely robust (no compile, no build lock, no registry), so transient failures are unlikely — but the sentinel has no TTL, no retry-once, and no way to tell transient from permanent. Recommendation (optional, since spec sanctions the behavior): consider NOT caching the failure on the specific transient signals (timeout `signal` / ENOENT-that-might-be-PATH-race) while still caching genuine cargo-nonzero/cargo-missing; or expose a `clearCargoMetadataCache()` for the harness to call between independent gates. At minimum document that a transient first-call failure blinds the rest of the run.
### ADV-05: Heavy JSDoc density — roughly half the added lines re-explain the same fallback chain

- **Severity**: low
- **Lens**: Minimalist
The build-runner.ts diff adds ~305 lines, of which an estimated ~150 are JSDoc/comment, much of it restating the identical never-throw / identity-fallback / per-cwd-cache contract three or four times across `loadCargoMetadata`, `resolveCargoPackageNames`, `matchPackageBySegment`, and `classificationScope`. The spec does mandate documenting the fallback (AC-02), so verbosity here is defensible and even helpful for a framework module. But the redundancy (the same SCENARIO-xxx citations and fallback bullet-lists repeated per function) is token-cost on every future read/edit and a maintenance hazard (drift between the duplicated paragraphs). Recommendation: state the fallback contract ONCE on `resolveCargoPackageNames` and have the helpers reference it; keep the per-function docs to their own unique invariants.
