# Debug Analysis: Scope-aware cargo build-gate emits DIRECTORY names as `-p` flags instead of real package names

- **Date**: 2026-07-20

---

## Summary

The scope-aware cargo build-gate in src/build-runner.ts false-fails on any Cargo workspace where a crate's directory name differs from its declared package name (e.g. dir `crates/data/` → package `stockfan-data`). `detectTouchedCargoPackages` (lines ~161-180) runs `git diff --merge-base main --name-only`, extracts the first `crates/<dir>/` segment of each path via regex `/(?:^|\/)crates\/([^/]+)\//`, and returns those directory NAMES verbatim through `dedupePreservingOrder`. There is NO step that resolves directory→package name. `scopedCargoArgs` (line ~213) then passes each entry through as `-p <name>`, producing e.g. `cargo build -p data -p tools -p workflows --quiet`. Cargo's `-p` flag matches against the package `name` field from `Cargo.toml`, not paths, so it rejects `data` instantly with `error: package ID specification 'data' did not match any packages` (exit 101) — before compiling anything. The repair loop cannot fix it because the command is framework-derived and identical on every retry; the run aborts after 3 attempts + pre-merge. On the stockfan shape (dirs data/tools/workflows → real packages stockfan-data/stockfan-tools/stockfan-workflows/stockfan-api) the run 2026-07-20T06-16-52-090Z aborted for exactly this reason. A secondary issue: the implementation agent declared success after `cargo test -p stockfan-data --lib --quiet` (lib unit tests only), so even the correct names would have skipped integration binaries under `tests/` and spec-mandated e2e in `crates/workflows/tests/`. Fix: add `resolveCargoPackageNames(cwd, touchedDirs)` that runs cached `cargo metadata --format-version 1 --no-deps --manifest-path <cwd>/Cargo.toml` (never-throw, dir-name fallback, per-cwd memoization) and wire it as the final mapping step in `detectTouchedCargoPackages`; strengthen `buildImplementPrompt`/`buildQaPrompt` in src/prompts.ts to forbid `--lib`-only verification.

## Hypotheses

- L1-H1 [p=0.85, CONFIRMED] — The gate emits directory names as `-p` flags; cargo rejects them because the real package `name` differs. Evidence: src/build-runner.ts:171 regex captures `m[1]` (the `[^/]+` dir segment), line 180 returns `dedupePreservingOrder(pkgs)` directly with no metadata resolution; grep for `resolveCargoPackageNames`/`cargo metadata` in build-runner.ts returns nothing. Prediction: on the stockfan shape the gate argv contains `-p data` not `-p stockfan-data`.
- L1-H2 [p=0.10, DISPROVEN] — The regex drops the `workflows` crate (e2e dir) so the touched set is incomplete. Evidence: regex `/(?:^|/)crates/([^/]+)//` against `crates/workflows/tests/e2e_foo.rs` captures `workflows` correctly (the `[^/]+` matches up to the next `/`). The e2e crate is captured; it is simply mis-NAMED as `-p workflows`. Fix 2 is transitively resolved by Fix 1's dir→name mapping — no regex change needed.
- L1-H3 [p=0.05, DISPROVEN] — A spawn timeout or empty-scope fallback causes the failure. Evidence: the run log shows exit 101 originating FROM cargo (`package ID specification 'data' did not match any packages`), i.e. a non-empty but wrong argv reached cargo; an empty scope would have produced a workspace-wide command with no `-p` flags, which would not yield that exact cargo error.
- L2-H1a [p=0.95, CONFIRMED] under L1-H1 — No `cargo metadata` resolution step exists; `detectTouchedCargoPackages` returns directory segments as-is. Evidence: full function body lines 161-180 has no spawn of `cargo metadata`, no JSON parse, no manifest lookup.
- L2-H1b [p=0.05, DISPROVEN] under L1-H1 — Resolution exists but is wired incorrectly. Evidence: grep confirms neither `resolveCargoPackageNames` nor `cargo metadata` appears anywhere in build-runner.ts, so there is nothing to mis-wire.
- L3-Fix3 [p=0.90, CONFIRMED] — Self-verification guidance is too weak: src/prompts.ts:93-98 (`buildImplementPrompt`) and :97-98 (`buildQaPrompt`) only say 'Run the full test suite and verify build succeeds' with no `--lib` prohibition and no explicit `cargo test -p <pkg>` requirement. The run log corroborates: the agent declared green on `cargo test -p stockfan-data --lib --quiet`.

## Root Cause

`src/build-runner.ts` `detectTouchedCargoPackages` (lines ~161-180) extracts the first `crates/<dir>/` path segment via regex `/(?:^|\/)crates\/([^/]+)\//` and returns those DIRECTORY names directly (`dedupePreservingOrder(pkgs)`) with NO resolution to real package `name`s. `scopedCargoArgs` (line ~213) emits each as `-p <dir-name>`. Cargo's `-p` flag matches the package `name` declared in `Cargo.toml` (`[package] name = "..."`), not the directory path, so any workspace that prefixes crate names (dir `data` → package `stockfan-data`) produces `cargo build -p data` → `error: package ID specification 'data' did not match any packages` (exit 101) before compilation. The repair loop cannot recover because the argv is framework-derived and identical every retry. EXACT LOCATIONS: (1) src/build-runner.ts:171 regex captures directory segment; (2) src/build-runner.ts:180 returns segments verbatim (the missing `resolveCargoPackageNames` call site); (3) src/build-runner.ts:213 `scopedCargoArgs` passes through as `-p`; (4) src/build-runner.ts:551 `detectTouchedCargoPackages(cwd)` invocation feeds the gate. Secondary (Fix 3): src/prompts.ts:93-98 `buildImplementPrompt`/`buildQaPrompt` provide only generic 'Run the full test suite' guidance with no `--lib` prohibition, allowing the implementation agent to declare success on `cargo test -p <pkg> --lib` (lib unit tests only, skipping `tests/` integration binaries and `crates/workflows/tests/e2e_*.rs`). RECOMMENDED FIX: (a) add `resolveCargoPackageNames(cwd, touchedDirs)` — discrete-argv `spawnSync('cargo',['metadata','--format-version','1','--no-deps','--manifest-path', path.join(cwd,'Cargo.toml')], {encoding:'utf8', timeout envelope})`, parse JSON, match each touched `crates/<dir>/` against the parent directory of each workspace package's `manifest_path`, return real `name`s deduped/first-seen-order; never-throw try/catch with dir-name identity fallback; module-level `Map<cwd, MetadataResult>` cache so one gate run spawns `cargo metadata` at most once per cwd; (b) insert `dirs = resolveCargoPackageNames(cwd, dirs)` as the final step before `dedupePreservingOrder` in `detectTouchedCargoPackages`; (c) no regex change — the existing capture already includes `workflows`; (d) strengthen `buildImplementPrompt`/`buildQaPrompt` text to REQUIRE `cargo test -p <pkg>` with NO `--lib` plus spec-mandated e2e, with an explicit 'do NOT declare green on --lib-only evidence' line. REGRESSION TESTS: hermetic tests stubbing `node:child_process.spawnSync` to (a) return a prefixed-name manifest JSON and assert dir→name mapping incl. manifest-in-subdir; (b) return spawn-error/non-zero/timeout and assert dir-name identity fallback; (c) call resolver twice for same cwd and assert spawn invoked once (cache hit); (d) end-to-end `scopedCargoArgs` on dir!=name shape produces `-p stockfan-data -p stockfan-tools -p stockfan-workflows`. PREVENTION: the gate's job is package-scoped cargo invocation, so package IDs must always be sourced from `cargo metadata` (the source of truth for package names), never inferred from filesystem layout; a one-line guard asserting every emitted `-p` value exists in `cargo metadata`'s `packages[].name` would make any future regression fail loudly instead of silently passing dir names through.

## Reproduction Steps

- Preconditions: a Cargo workspace (default members) where at least one member's directory name differs from its declared package name — e.g. stockfan: `crates/data/Cargo.toml` declares `name = "stockfan-data"`, `crates/tools/` → `stockfan-tools`, `crates/workflows/` → `stockfan-workflows`.
- From a feature branch with changes under `crates/data/`, `crates/tools/`, and `crates/workflows/tests/e2e_*.rs`, trigger the build gate (any SUPER_DEV run that reaches runBuildGate with `language === 'rust'` and a non-empty touched scope).
- Watch the gate invoke `detectTouchedCargoPackages(cwd)`: it spawns `git -C <cwd> diff --merge-base main --name-only`, regex-captures the first `crates/<seg>/` of each line → `['data','tools','workflows']`, and returns those DIRECTORY names with no `cargo metadata` lookup.
- `scopedCargoBuildArgs(['data','tools','workflows'])` returns `['cargo','build','-p','data','-p','tools','-p','workflows','--quiet']`.
- `spawnSync('cargo',['build','-p','data','-p','tools','-p','workflows','--quiet'])` exits 101 with stderr `error: package ID specification 'data' did not match any packages` — emitted BEFORE any compilation, on the first attempt.
- The repair loop re-runs the identical framework-derived command (3 attempts + pre-merge, all byte-identical because the framework, not the agent, owns the `-p` flags); every attempt fails identically; the run aborts.
- MINIMAL deterministic repro (no cargo/git needed): assert `scopedCargoBuildArgs(['data'])` deep-equals `['cargo','build','-p','data','--quiet']` — proving the gate passes the raw directory name straight through with no metadata resolution. The fix flips this to `-p stockfan-data` once `resolveCargoPackageNames` is wired in.
- Secondary repro for Fix 3: in the run log, grep the implementation stage's tool calls — it declared `allTestsPass: true` after running only `cargo test -p stockfan-data --lib --quiet`; no `tests/` integration binary and no `crates/workflows/tests/e2e_*.rs` target was ever executed, confirming the `--lib`-only green path the strengthened prompts must forbid.
