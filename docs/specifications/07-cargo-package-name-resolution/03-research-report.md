# Research Report: Scope-aware cargo build-gate: resolve REAL package names via `cargo metadata` (deep-research / open-issues gate)

- **Date**: 2026-07-20
- **Author**: super-dev:research-agent

## Summary

The build gate's `detectTouchedCargoPackages` (src/build-runner.ts:161) maps `git diff` paths `crates/<seg>/` to the DIRECTORY name `<seg>` and `scopedCargoArgs` (:213) passes those verbatim as `-p <name>`, so any workspace that prefixes crate names (stockfan: dir `data` → package `stockfan-data`) makes cargo abort with "package ID specification `data` did not match any packages" BEFORE compiling — a framework-derived false fail the repair loop cannot fix. The single authoritative fix is to map each touched dir → the workspace package whose Cargo.toml directory matches it, using Cargo's OWN introspection. **Best practice (BP-1):** `cargo metadata --format-version 1 --no-deps` (SRC-1, doc.rust-lang.org/cargo/commands/cargo-metadata.html) is the Cargo-recommended, network-free, deterministic way to enumerate a workspace; its JSON exposes `workspace_root`, `workspace_default_members`, and `packages[].name` + `packages[].manifest_path` (absolute). **BP-2:** pin `--format-version 1` (the `format-version` field exists precisely so consumers pin and avoid breakage). **BP-3:** `--no-deps` keeps it fast and deterministic (no dep-graph resolution, no network). **BP-4:** follow the repo's own never-throw convention — `detectTouchedCargoPackages` already swallows every error into `[]`; the new `resolveCargoPackageNames` must reuse the established `{ cwd, timeout: timeoutMs, encoding:"utf8" }` spawn envelope (build-runner.ts:~581) with discrete argv (no `shell:true`) and degrade to the dir-name identity list (today's behavior) on spawn error / non-zero exit / timeout / missing cargo / no match. **BP-5:** memoize the parsed metadata in a module-level `Map<cwd, MetadataResult>` (process-local; a gate run calls the resolver multiple times and metadata is not free). **BP-6 (Fix 3):** `cargo test -p <pkg>` with NO `--lib` (SRC-2, cargo test Test Selection) — `--lib` restricts to `src/` unit tests and silently skips the `tests/` integration binaries; omitting the filter runs lib + integration targets. The authoritative prompt text lives in src/prompts.ts (`buildImplementPrompt`:93, `buildQaPrompt`:97), invoked from src/stages/implementation.ts:10/:73 and verify.ts — both need an explicit "do NOT declare green on `--lib`-only evidence; require full `cargo test -p <pkg>` + spec-mandated e2e/integration" instruction. All four prior research issues are RESOLVED and carry forward as implementation guidance, so there are NO open research blockers.

## Options Considered

### Option 1 (RECOMMENDED): `cargo metadata --format-version 1 --no-deps` dir→name resolver + per-cwd cache + never-throw dir-name fallback

Authoritative: Cargo's own introspection resolves real package names and handles prefixed names, manifest-in-subdir, workspace `name.workspace = true` inheritance, and path-deps correctly (SRC-1). Network-free and deterministic via `--no-deps`; schema-pinned via `--format-version 1` (BP-2/BP-3). Adds exactly ONE cached spawn per cwd per gate run (BP-5); reuses the existing spawnSync timeout envelope (build-runner.ts:~581) with discrete argv so package/path data never reach a shell. ACCEPTED DEGRADATION (prior ISS-3, now closed): a half-broken target-repo manifest makes `cargo metadata` exit non-zero → the never-throw fallback returns the dir-name identity list for THAT repo only (= byte-identical today's behavior); document this in a JSDoc on `resolveCargoPackageNames`. Wire it as the FINAL step of `detectTouchedCargoPackages` so `scopedCargoArgs`/scopedBuild/Test/ClippyArgs (build-runner.ts:213+) need NO change and the whole `-p` flag family is fixed end-to-end. No new runtime deps.
### Option 2: Hand-parse each `crates/<dir>/Cargo.toml` to read `name =`

No spawn, but fragile: workspace inheritance (`[workspace.package]` + `name.workspace = true`) forces you to resolve inheritance by hand or pull in a TOML parser dep (rejected — repo is strict and dep-free for this); also mishandles manifest-in-subdir and virtual manifests. Worse correctness/fragility tradeoff than Cargo's own metadata. Rejected.
### Option 3: Abandon scoping — `cargo build/test/clippy --workspace`

Always name-correct, but OUT OF SCOPE (task explicitly forbids changing the scoped-vs-workspace-wide gate strategy) and far slower on large workspaces — compiles/tests the entire workspace on every gate run, defeating the touched-crate scoping that is the current default. Rejected for strategy + perf reasons.
### Option 4 (Fix 3 sub-decision, RECOMMENDED): Prompt-level verification enforcement in buildImplementPrompt + buildQaPrompt

Strengthen the authoritative prompt builders (src/prompts.ts:93 and :97) to REQUIRE `cargo test -p <pkg>` with NO `--lib` flag plus any spec-mandated e2e/integration target, with an explicit 'do NOT declare green on `--lib`-only evidence; `--lib` skips the `tests/` integration binaries' instruction (SRC-2). Keeps the change within the 'do NOT touch nodes/workflow/pipeline/render' constraint (prompt-text only, no control-flow change). Relies on the LLM agent honoring the instruction — the same trust model the rest of the gate already uses. Heavier control-flow enforcement is out of scope. Also defensively strengthen buildQaPrompt even though verify.ts does not currently import it (prior implementer action item, now folded into AC-07).
