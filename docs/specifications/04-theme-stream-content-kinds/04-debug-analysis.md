# Debug Analysis: All stream content kinds flattened into one undifferentiated text block — no LineKind classification, no per-kind Theme tokens, no tool-bubble backgrounds

- **Date**: 2025-11-21

---

## Summary

In pi-super-dev, every line the super-dev pipeline emits — commands (`→ …`), agent thinking (sink.text), phase banners (`▶ …`), corrective notices (`↻ …`), errors (`❌ …`/`failed after`), and normal stage logs — is stored and rendered as an opaque plain string with NO kind metadata. The three sinks (`sink.phase`/`sink.log`/`sink.text`) push raw strings into `transcript: string[]` (src/extension.ts:158), `flush()` joins them with `\n` verbatim (extension.ts:177-180), the on-disk log writes the raw join (extension.ts:262), `details.transcriptTail` is `transcript.slice(-50)` of plain strings (extension.ts:278), and `buildResultComponent` §1 paints the ENTIRE detail log with a single `fg("dim", line)` (dashboard.ts:339-340). The dedicated pi Theme tokens required for the pi-native look are never used: `thinkingText`, `toolTitle`, `toolPendingBg`, `toolSuccessBg`, `accent`/`bold` for phases, `error` red, `success`/`warning` are all bypassed. The new module `src/render/stream-theme.ts` does not exist, `classifyLine`/`themeLine`/`commandBackground` do not exist, and `DashboardTheme` (dashboard.ts:31) declares only `fg` + optional `bold` with no `bg()` — so even the real Theme's background tokens are structurally unreachable. Net effect: the live `onUpdate` stream and the `renderResult` §1 detail log are one flat, dim, monochrome wall of text — the exact opposite of pi-native's tool-bubble + thinking-color + accent-phase + error-red multi-way split.

## Hypotheses

- H1 (90%): NO classification layer exists. There is no `LineKind` type, no `classifyLine`, no `themeLine`, no `commandBackground`, and no `src/render/stream-theme.ts` module (confirmed: file does not exist). Without a `LineKind`, no code path can choose a distinct Theme token per kind, so everything collapses to the one token §1 hardcodes. Prediction: grep for `LineKind|classifyLine|themeLine|commandBackground` returns zero hits. FALSIFIED if these symbols already exist and are wired.
- H2 (85%): `flush()` emits raw joined text with no theming and no TUI/non-TUI branching. Evidence: extension.ts:177-180 builds `body` purely from `all.join("\n")` (or trimmed join) and hands it to `onUpdate` — it never touches `ctx?.ui?.theme` and never branches on `ctx?.mode === "tui"`. Prediction: flush() references neither `theme` nor `ctx.mode`. This explains why the LIVE stream is undifferentiated and also means any future classification must be injected here (TUI-only styling) with a raw-text fallback for print/json/RPC to keep byte-clean output (AC-05 / AC-08).
- H3 (90%): `renderResult` §1 unconditionally dims every line and never constructs a tool-bubble `Text`. Evidence: dashboard.ts:338-343 loops `for (const line of details.transcriptTail ?? []) container.addChild(new Text(fg("dim", line), 0, 0))` — `new Text` is always called with the 3-arg form (text, paddingX=0, paddingY=0); the 4th `customBgFn` argument is never passed. Prediction: no call site in buildResultComponent passes a 4th arg to `new Text`, so command lines cannot receive `toolPendingBg`/`toolSuccessBg` backgrounds. FALSIFIED if a 4-arg Text call for command lines already exists.
- H4 (95%): The kind metadata never travels end-to-end because the data structures carry plain strings. Evidence: transcript is `string[]` (extension.ts:158), sink.phase pushes a template string (extension.ts:228), sink.log pushes a template string (extension.ts:229), sink.text's `live` is a raw string finalized via `transcript.push(live)` (extension.ts:169-173), `transcriptTail` is typed `string[]` (extension.ts:295), and `ResultDetails.transcriptTail` is `string[]` (dashboard.ts:292). Prediction: there is no `{ kind, text }` shape anywhere in the transcript→flush→details→renderResult path. This is why even a perfect `themeLine` helper could not help — the kind is discarded at the sink. FALSIFIED if any of these already carry a `kind` field.
- H5 (60%): `DashboardTheme` is structurally too narrow to satisfy the real pi Theme's `bg()` surface, so backgrounds are impossible without an interface change. Evidence: dashboard.ts:31-34 declares only `fg(token, text)` + optional `bold(text)`; no `bg`. Since `commandBackground` needs `theme.bg("toolPendingBg", _)` / `theme.bg("toolSuccessBg", _)`, the interface MUST be extended with an optional `bg?(token, text)` (AC-03) or the real Theme's `bg` is invisible to the typed code. FALSIFIED if `bg` is already present (it is not). Note: the `fg` token-string pass-through already works for `toolTitle`/`thinkingText` because the real Theme dispatches by token name — the missing piece is purely `bg` + actually CALLING those tokens.

## Root Cause

The pipeline's stream surface has NO line-classification layer and NO per-kind Theme mapping, so every content kind is stored, transported, and rendered as an opaque plain string. Concretely: (1) `src/render/stream-theme.ts` — the module that should export `LineKind`, `classifyLine`, `themeLine`, and `commandBackground` — DOES NOT EXIST, so there is no `LineKind` type and no function that maps a kind to `theme.fg(\"toolTitle\"/\"thinkingText\"/\"accent\"/\"error\"/\"success\"/\"warning\"/\"muted\"/\"dim\"/\"text\", …)` or `theme.bg(\"toolPendingBg\"/\"toolSuccessBg\", …)`. (2) `transcript: string[]` (src/extension.ts:158), `sink.phase`/`sink.log` (extension.ts:228-229), `live` (extension.ts:169), `flush()` (extension.ts:177-180), the disk log write (extension.ts:262), and `details.transcriptTail`/its type (extension.ts:278,295) all carry plain strings — the kind is discarded the moment a line is emitted at the sink, so no downstream consumer can recover it. (3) `flush()` builds its `body` from `all.join(\"\\n\")` with NO branching on `ctx?.mode === \"tui\"` and never reads `ctx?.ui?.theme`, so the LIVE onUpdate stream is unstyled raw text regardless of mode (and, as required, must STAY raw in non-TUI to keep print/json/RPC byte-clean — theming must be added only in the TUI branch). (4) `DashboardTheme` (dashboard.ts:31-34) declares only `fg` + optional `bold` with NO `bg()`, making the real pi Theme's `toolPendingBg`/`toolSuccessBg` background tokens structurally unreachable from typed code — the interface must be extended with optional `bg?(token, text)`. (5) `buildResultComponent` §1 (dashboard.ts:338-343) paints EVERY `transcriptTail` entry with a single `fg(\"dim\", line)` and constructs `new Text(styled, 0, 0)` with the 3-arg form — the 4th `customBgFn` argument is never passed, so command lines can never receive the tool-bubble background that gives pi-native its multi-way visual split. The combination of (1)-(5) is why commands, thinking, phase markers, and errors are visually indistinguishable: the kind is lost at emit time, the transport carries only text, and both render paths apply one uniform dim style. RECOMMENDED FIX: create `src/render/stream-theme.ts` (pure `classifyLine`/`themeLine`/`commandBackground`, classify AFTER trimming leading whitespace because sink.log stores `  ${message}` so `→ …` arrives as `  → …`; match `→ structured_output ✓` before generic `→ `; return raw text + undefined background when theme is absent); extend `DashboardTheme` with optional `bg?`; change `transcript` to `Array<{ kind: LineKind; text: string }>`, tag at the sink (phase→\"phase\"/`▶ ${label}`, log→classifyLine(message), text's live→\"thinking\"), thread `{kind,text}` through `flush`/disk-log(raw text only)/`details.transcriptTail`; in `flush()` style via `themeLine(kind,text,ctx.ui.theme)` ONLY when `ctx.mode===\"tui\"`, else raw join; in `buildResultComponent` §1 render each line via `themeLine(line.kind,line.text,theme)` and emit COMMAND/COMMAND-DONE as `new Text(styled,0,0,commandBackground(line.kind,theme))` to get the tool-bubble bg. REGRESSION TEST: drive the sink through phase/log/text events with NO theme and assert the joined transcript + on-disk log contain ZERO ANSI escape bytes; assert the TUI path contains the expected `toolTitle`/`thinkingText`/`accent`/`error` fg tokens and that command Texts carry a customBgFn. PREVENTION: require every new stream-content emit site to route through a typed sink whose push type is `{kind,text}` (not string), so a kind can never be silently dropped again.
