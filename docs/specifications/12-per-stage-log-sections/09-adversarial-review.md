# Adversarial Review: Adversarial Review — Per-Stage Log Sections (spec-12): Phases 1–3 only; integration wiring absent, Phase 4 unstarted, headline feature is dead code in production

- **Date**: 2025-01-15
- **Reviewer**: super-dev:adversarial-reviewer
- **Verdict**: REJECT

---

The committed Phases 1 (TranscriptLine stage tagging) and 2 (groupByStage partitioner) are sound and unit-test cleanly. Phase 3 (per-stage sections in flush()) is implemented in src/render/live-stream.ts but is INCOMPLETE (1 own RED test) and, critically, UNREACHABLE in production: the spec's Phase 5 single wiring point — calling stream.sink.stage(info) from the dashboard `stage:` handler in extension.ts — does not exist (grep confirms zero `.stage(` calls). As a result `stageReceived` stays false for the entire real run, `flush()` always takes the legacy rolling-tail branch, `currentStageId`/`currentStageLabel` stay at the sentinel "setup"/"pre-stage", and every transcript entry is tagged with that sentinel — collapsing groupByStage to a single group. Phase 4 (buildResultComponent per-stage blocks with customBgFn toolPendingBg/Success/Error) is not started at all: §1 is still the single merged "detail log (last 50 lines)" loop. The headline deliverable — "render each pipeline STAGE as its OWN log section" — is therefore not achieved in either the streaming view or renderResult; the ~120 LOC of section-rendering exists only in unit tests that invoke h.sink.stage() directly. Header confirms "Phases Completed: 2/6". Three lenses concur: Architect (integration boundary not crossed, Phase 4 absent), Skeptic (own test RED, stage-drop bug, sentinel-perpetually-running), Minimalist (unreachable code reviewed prematurely, fragile re-tag invariant). Verdict REJECT: shipping this ships a no-op feature.

### F1: Phase 5 wiring absent: stream.sink.stage(info) is never called → Phase 3 section-stack is dead code in the production extension path

- **Severity**: Critical
- **Lens**: Architect
The spec (Phase 5 / AC-05) mandates extending the dashboard `stage:` handler in src/extension.ts to ALSO call `stream.sink.stage(info)`. It does not: `grep -rn "\.stage(" src/extension.ts src/render/dashboard.ts src/render/render.ts` returns ZERO matches; the only stage handling is `stage: (info) => {...}` at extension.ts:381 which updates dashboardOrder/dashboardStages/renderDashboard (widget tracker only). Consequence chain: (1) LiveStreamSink.currentStageId/currentStageLabel stay at the sentinel "setup"/"pre-stage" for the entire run; (2) `stageReceived` (live-stream.ts) stays false forever, so `flush()` ALWAYS routes to flushRollingTail and NEVER to flushSectionStack; (3) every pushed transcript entry inherits the sentinel tag, so groupByStage (even when eventually invoked) yields exactly ONE group. The entire Phase-3 surface — flushSectionStack, renderSectionHeader, stageStatus map, RUNNING_TAIL_LINES/COMPLETED_TAIL_LINES caps (~120 LOC) — is unreachable except in unit tests that call h.sink.stage() directly. Acceptance Criteria 1, 2, and 4 are not met in the integrated system. Fix: in the existing `stage:` handler add `stream.sink.stage(info);` (one line) and thread details.stages → statusOf into buildResultComponent. This is the linchpin that makes Phases 1–4 observable at all.
### F2: Phase 4 not implemented: buildResultComponent §1 is still the single merged 'detail log (last 50 lines)' loop — no per-stage blocks, no customBgFn

- **Severity**: Critical
- **Lens**: Architect
dashboard.ts:360 buildResultComponent is unchanged from the pre-spec baseline. §1 (lines ~376-386) still renders `fg("dim", "── detail log (last 50 lines) ──")` then a single `for (const line of details.transcriptTail ?? [])` loop emitting per-kind Text children via themeLine + commandBackground. There is NO call to groupByStage anywhere in dashboard.ts (grep confirms), NO per-stage Container/Text block stack, NO status-themed bold header (running ●/ok ✓/failed ✗/skipped ↷), and NO per-stage background via pi-tui Text's 4th customBgFn arg using theme.bg("toolPendingBg"/"toolSuccessBg"/"toolErrorBg"). The spec calls this §1 stack 'the high-impact surface' and it is entirely untouched. Acceptance Criterion 3 is not met. Without this, renderResult looks identical to before the entire spec-12 effort.
### F3: Own Phase-3 RED test still failing: a stage that emits zero log lines is silently dropped from the section stack

- **Severity**: High
- **Lens**: Skeptic
`npx vitest run tests/live-stream-flush-sections.test.ts` → 1 failed, 45 passed. The failing case (test:261, 'renders a completed stage with zero visible lines as header-only') sets an `EmptyCompleted` stage (status ok) with NO log lines, then a running ImplB stage with one log, and asserts the body contains 'EmptyCompleted'. The body contains only the ImplB header + live-now. Root cause: LiveStreamSink.stage() (live-stream.ts) sets currentStageId/currentStageLabel and re-tags the last entry, but pushes NO transcript entry of its own; groupByStage partitions ONLY actual transcript entries. Therefore any stage that emits zero log lines — skipped stages, fast stages, a stage that only streams via `text()` then never finalizes — produces no group and no header, silently vanishing from the §1 stack. This directly contradicts the dashboard widget (which shows every stage) and breaks the 'each stage is a distinct section' intent. Fix: the sink (or the statusOf resolver keyed off details.stages) must synthesize a header-only group for every stage present in the dashboard tracker, not only stages that logged.
### F4: ResultDetails.transcriptTail type NOT widened — spec's end-to-end stage-tag type contract is violated (silently papered over by structural subtyping)

- **Severity**: Medium
- **Lens**: Architect
dashboard.ts:320 still declares `transcriptTail?: Array<{ kind: LineKind; text: string } | string>`, not the spec-mandated `Array<{kind;text;stageId?;stageLabel?}|string>`. It typechecks (`npm run typecheck` clean) ONLY because TS structural subtyping accepts a wider TranscriptLine[] where the narrower element type is expected (excess-property checks don't apply to non-literal arrays). Runtime tolerates the extra fields by ignoring them. But the contract drift is real: any future caller that constructs a ResultDetails object LITERALLY gets a type error if it sets stageId/stageLabel, and the spec's AC-4 ('thread stage tags end-to-end') is not honored at the type boundary. Fix: widen the ResultDetails type additively as specified so the contract reflects what extension.ts actually passes.
### F5: Sentinel 'pre-stage' group renders as perpetually RUNNING (accent header + forever-spinning braille glyph) even after the run completes

- **Severity**: Medium
- **Lens**: Skeptic
Pre-stage setup logs (anything emitted before the first 'Stage 1' banner) carry the sentinel stageId 'setup' / stageLabel 'pre-stage'. The dashboard tracker never records a status for 'setup', so stageStatus.get('setup') is undefined; renderSectionHeader (live-stream.ts) treats undefined status identically to 'running' — accent foreground, bold, and runningGlyph(Date.now()/100) (animated braille). On a fully completed run's §1, this surfaces as a '▌ ⠇ pre-stage' header that never stops spinning, falsely signaling in-flight work. This also affects F1's collateral damage: because stage() is never called, the ENTIRE real transcript resolves to this single perpetually-running sentinel section. Fix: resolve a terminal status for the sentinel stage on finalize, or render unknown-status-but-run-complete groups as neutral/dim rather than running.
### F6: TOTAL_SECTION_CAP front-splice silently drops the EARLIEST stage sections (including headers), desyncing §1 from §2 stage-progress

- **Severity**: Medium
- **Lens**: Architect
live-stream.ts flushSectionStack applies `bodyLines.splice(0, bodyLines.length - TOTAL_SECTION_CAP)` (drops from the front) as an aggregate safety bound. For a large multi-stage run (the test fabricates 60×40), the oldest sections — headers included — are truncated, while §2 stage-progress (buildResultComponent) still lists every stage. The §1 block stack therefore becomes headless and inconsistent with §2. A faithful trim policy would drop oldest COMPLETED sections first (preserve running/failed/expanded headers), not blindly lop the front. Severity medium: cosmetic/UX, not a crash, but undermines the scannability the spec explicitly targets.
### F7: phase-line re-tag assumes the phase entry is strictly the most-recent transcript entry — any intervening commit silently defeats the correction

- **Severity**: Low
- **Lens**: Skeptic
sink.stage() re-tags ONLY transcript[length-1] when it is a `phase` line whose label matches info.label (live-stream.ts). If anything commits between sink.phase(label) and sink.stage(info) — a finalizeLive() thinking commit, or an interim log/userInput — the 'last entry' is no longer the phase line and the re-tag silently no-ops, leaving the phase banner tagged with the PREVIOUS stage. The comment asserts 'control-flow emits phase strictly BEFORE stage:{running}' (RESOLVED-1) but that invariant is neither enforced nor tested at the boundary; it is a load-bearing assumption with no guard. Fix: search back through recent entries for the matching phase line within a small window, or stamp the phase entry with a pending-stage marker at push time.
### F8: ~120 LOC of section-rendering is presently unreachable in production — reviewing it now reviews unit-test-only surface

- **Severity**: Low
- **Lens**: Minimalist
Because of F1 (Phase 5 wiring absent), flushSectionStack / renderSectionHeader / stageStatus / the per-stage caps and trim-notice-per-stage path can execute ONLY under direct h.sink.stage() invocation in tests. This is not a deletion recommendation — the code WILL be needed once F1 lands — but it means the present review's evidence for the production behavior of Phase 3 is indirect (unit-test structural assertions via a mock theme, not a real extension run). Combined with F2 (Phase 4 absent) and F3 (one of those unit tests is RED), the effective, shipped behavior of the whole feature is zero. Note also the parallel: the spec marked Phases 3 and 4 as parallelizable, but only 3 was started; recommend landing F1+F2+F3-fix before treating Phase 3 as done.
