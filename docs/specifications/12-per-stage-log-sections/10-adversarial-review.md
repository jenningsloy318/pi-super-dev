# Adversarial Review: Adversarial Review — Per-Stage Log Sections (spec-12)

- **Date**: 2025-07-22
- **Reviewer**: super-dev:adversarial-reviewer
- **Verdict**: CONTEST

---

The implementation is functionally complete and green: `npm run typecheck` is strict-clean and all 1341 tests across 79 files pass. The stage-tagging sink (AC-01), pure `groupByStage` partitioner (AC-02), streaming `flush()` section stack (AC-03), `buildResultComponent` per-stage blocks (AC-04), and `extension.ts` wiring (AC-05) all land as specified, with method-style theme access preserved and byte-clean non-TUI output. However, this is a CONTEST, not a PASS, for three reasons. (1) AC-06 explicitly mandated a dedicated `tests/render/per-stage-result.test.ts` plus a `customBgFnOf`-based assertion of per-child backgrounds — neither exists, so the highest-impact surface (the `buildResultComponent` §1 block stack: status glyphs, per-stage `customBgFn`, expand/compact, legacy fallback) has NO dedicated unit coverage; it is exercised only incidentally through the streaming-flush tests and a few real-theme-parity assertions. (2) The status taxonomy is encoded independently in three places (live-stream `renderSectionHeader`, dashboard `stageBlockGlyph`/`statusThemeToken`/`statusBackground`), and the `statusOf` resolver has asymmetric semantics between surfaces (live view treats the sentinel "setup" stage as running/accent; result view hardcodes it to "ok"/green), so the same logical state renders differently and a pre-stage failure would be mis-themed green in the final report. (3) Minor coupling and speculative-complexity smells (runtime import of `runningGlyph` from dashboard into live-stream; the scan-back-4 phase re-tag window; heavy process-artifact comments referencing ephemeral SCENARIO/AC/RESOLVED tokens). None of these risk production failure or data loss, so REJECT is not warranted, but the author should respond to the coverage gap and the status-asymmetry before merge.

### AR-01: AC-06 dedicated test file tests/render/per-stage-result.test.ts is missing — buildResultComponent §1 block stack has no direct coverage

- **Severity**: medium
- **Lens**: Skeptic
Acceptance criterion AC-06 explicitly requires `tests/render/per-stage-result.test.ts` asserting block stack, status glyphs, per-child `customBgFn` (via an existing `customBgFnOf` helper), expand/compact, and legacy fallback. That file does not exist (`ls tests/render/per-stage-result.test.ts` → not found), and `grep -rn customBgFnOf tests/` returns zero matches — the helper is referenced nowhere. The new `buildResultComponent` per-stage branch (src/render/dashboard.ts, the `else` block after `hasStageTags`) is the spec's 'high-impact surface' yet is covered only incidentally by `live-stream-flush-sections.test.ts` (which drives the STREAMING `flush()`, a different code path) and a handful of assertions in `real-theme-parity.test.ts`. Concretely untested: the status→glyph map (✓/✗/↷/●), the `statusBackground` customBgFn wiring per status (toolPendingBg/Success/Error), the expand-vs-compact line slicing (`expanded ? group.lines : group.lines.slice(-1)`), and the `hasStageTags` legacy fallback. This is a gap against an explicit acceptance criterion, not a code defect. Recommendation: add the file with a mock theme whose `bg` emits a searchable marker and assert (a) one header Text per stage with the correct glyph, (b) `customBgFn` is present on each header and resolves to the right token per status, (c) failed/running blocks emit all lines while completed blocks emit ≤1, (d) an all-`setup`/string tail collapses to the single legacy '── detail log ──' block.
### AR-02: statusOf resolver is asymmetric across surfaces: 'setup' renders running (live) but forced ok/green (result), masking pre-stage failures

- **Severity**: medium
- **Lens**: Architect
In `flushSectionStack` (src/render/live-stream.ts) the injected resolver is `(id) => stageMeta.get(id)?.status`, so the sentinel 'setup'/'pre-stage' group resolves to `undefined` → treated as running → accent + animated braille. In `buildResultComponent` (src/render/dashboard.ts) the resolver is `(stageId) => stageId === 'setup' ? 'ok' : idToStatus.get(stageId)`, so the SAME sentinel group is forced to 'ok' → green ✓ with `toolSuccessBg`. Two consequences: (1) the same logical state is represented inconsistently between the live view and the final report; (2) if the run actually fails during the pre-stage window (all error lines land in the 'setup' group because no structured `stage` event has arrived), the result view paints them with a success glyph and success background — actively misleading. Recommendation: either resolve 'setup' status from the real stage tracker when available, or render it as a neutral/unknown state (dim, no glyph) rather than hardcoding 'ok'. At minimum document why pre-stage is always 'ok' in the result view.
### AR-03: Status→color/glyph taxonomy duplicated in three independent places

- **Severity**: medium
- **Lens**: Architect
The mapping ok→success/✓, failed→error/✗, skipped→warning/↷, running→accent/● is encoded independently in: `renderSectionHeader` (src/render/live-stream.ts, header theming), `statusThemeToken` + `stageBlockGlyph` (src/render/dashboard.ts, result block header), and `statusBackground` (src/render/dashboard.ts, customBgFn token). `grep` for the 'ok' branch alone hits 6 locations across the two files. Adding a new status (e.g. 'retry', 'cancelled') requires touching all three with no compiler help tying them together — a status string valid in one mapping but unhandled in another silently falls through to a default. Recommendation: extract one `STATUS_STYLE: Record<string, {fgToken, glyph, bgToken?}>` table (in stream-theme.ts alongside the other presentation primitives) and have all three call sites read from it, with a default that is visibly wrong (e.g. throw or a '?' glyph) so a miss is caught immediately.
### AR-04: runningGlyph runtime import from dashboard.ts into live-stream.ts muddies the render-layer boundary

- **Severity**: low
- **Lens**: Architect
src/render/live-stream.ts now does `import { runningGlyph, type DashboardTheme } from './dashboard.js'` — previously it imported only the `type DashboardTheme`, so live-stream had no runtime dependency on dashboard. `runningGlyph` is a generic presentation primitive (a braille-frame picker) that has nothing to do with the dashboard specifically; it logically belongs next to `themeLine`/`classifyLine` in stream-theme.ts (which already owns the 'RUNNING_FRAMES / runningGlyph' header per dashboard.ts:10). There is no circular import today (dashboard does not import live-stream, confirmed), so this is a boundary/placement smell, not a defect. Recommendation: move `runningGlyph` (and its frame table) into stream-theme.ts and import it from both consumers.
### AR-05: Phase-line re-tag scans back a fixed window of 4 — correct today, but the bound is magic and undocumented against real traces

- **Severity**: low
- **Lens**: Skeptic
In `stage()` (src/render/live-stream.ts) the RESOLVED-1 phase-before-stage correction scans back `Math.max(0, transcript.length-4)` entries to find and re-tag the matching `▶ <label>` phase line, breaking early if an entry already tagged with this stage is reached. The window of 4 is justified in a comment by 'an intervening commit (e.g. a finalizeLive thinking push)'. If more than 4 entries ever land between the `phase` banner and the `stage:{running}` event (e.g. several rapid `log` calls flushed in between, or multiple thinking commits), the phase banner is silently mis-attributed to the PREVIOUS stage and the new stage's header is synthesized empty instead. There is no assertion or trace evidence that ≤4 always holds. Recommendation: either (a) make the scan unbounded but break on the first already-tagged-this-stage entry (the existing break makes this safe and O(distance) not O(n)), removing the magic 4 entirely, or (b) add a regression test that pushes >4 intervening entries and asserts the phase line is still re-tagged, pinning the current bound.
### AR-06: Aggregate-cap accounting is off-by-one (counts N separators, emits N-1) — harmless but imprecise

- **Severity**: low
- **Lens**: Minimalist
In `flushSectionStack`, `total = sections.reduce((sum, sec) => sum + sec.length + 1, 0)` charges +1 per section for the blank separator, but the join loop only emits a blank `if (i > start)` — i.e. (kept-1) separators, not (kept). The budget is therefore over-counted by one line, so the trim loop may drop one more leading section than strictly necessary. The error is conservative (trims slightly more), so it never violates the `≤ TOTAL_SECTION_CAP` guarantee pinned by the test, but the accounting does not match the emitted byte count. Recommendation: either compute `total` as `sum(sec.length) + (sections.length-1)` or assert the final `bodyLines.length` (not the pre-trim `total`) against the cap.
### AR-07: Comments embed ephemeral spec-process tokens (SCENARIO-XXX / AC-0X / RESOLVED-X) that will not survive delivery

- **Severity**: low
- **Lens**: Minimalist
Across stage-grouping.ts, live-stream.ts, dashboard.ts, and the new test files, doc-comments routinely cite 'SCENARIO-010', 'AC-04', 'RESOLVED-1', 'Phase 1/2/3/4/5'. These are artifacts of THIS specification's BDD/process vocabulary; once merged they reference documents a future maintainer does not have open, so they read as noise rather than domain knowledge (e.g. 'SCENARIO-012: synthesize an empty group…' explains the WHY via a ticket id instead of via the user-facing reason). The code itself is clear; the comments add length without enduring context. Recommendation: rewrite the enduring comments to state the user-facing invariant ('a stage that started but logged nothing still shows its header so the user sees it ran'), and drop or move the SCENARIO/AC ids into git commit messages / the spec doc only.
### AR-08: Legacy fallback path duplicates the pre-change §1 renderer verbatim — acceptable for byte-parity, but note the duplication

- **Severity**: low
- **Lens**: Minimalist
In `buildResultComponent`, the `if (!hasStageTags)` branch re-implements the previous '── detail log (last 50 lines) ──' + per-kind `themeLine` + `commandBackground` loop essentially verbatim alongside the new per-stage `else` branch. This is a deliberate choice (the spec demands byte-for-byte legacy parity, pinned by dashboard-result.test.ts / dashboard-result-perkind.test.ts, and duplication is safer than a shared abstraction here). Flagging only so it is a conscious decision: if the per-kind theming or command-bubble background logic ever changes, both branches must be updated in lockstep. A one-line `// NOTE: intentionally duplicates the legacy path for byte-parity — keep in sync with the per-stage branch` would make the coupling explicit.
