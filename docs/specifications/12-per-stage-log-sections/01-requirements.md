# Requirements: Per-stage log sections: render each pipeline stage as its own themed section (streaming + renderResult)

- **Date**: 2026-07-21
- **Author**: super-dev:requirements-clarifier
- **Type**: enhancement
- **Priority**: high
- **Status**: draft

---

## Executive Summary

Today every pipeline stage's log lines collapse into one flat rolling transcript (one onUpdate text blob in streaming + one merged `── detail log ──` block in buildResultComponent §1), even though stage TITLES are already themed and the dashboard already tracks per-stage status. This change tags each transcript entry with its stage, groups by stage, and renders each stage as its own headed, status-colored section — the pi-native tool-call-bubble look — in both the live streaming flush and the final renderResult. It is a pure TS change to this repo (no new deps, no control-flow / change-tracker / backend changes), backward-compatible for non-TUI and legacy tail shapes.

## Acceptance Criteria

- **AC-01**: The `TranscriptLine` shape (src/render/live-stream.ts) gains `stageId: string` and `stageLabel: string`. The live-stream sink tracks a `currentStageId`/`currentStageLabel` defaulting to `"setup"`/`"pre-stage"` before the first phase banner; `phase(label)` updates it on each banner. Every pushed entry (phase/log/thinking/trim/user-input) inherits the current stage tag. Implementation sub-phases emitted WITHIN Stage 9 collapse to stage `implementation` (no finer sub-phase grouping — don't over-engineer). Stage-id resolution MUST come from the structured dashboard `stage` event, NOT by parsing the human `▶ Stage N` label (see open questions).
- **AC-02**: A new pure helper `groupByStage(transcript): Array<{ stageId; stageLabel; status?; lines: Array<{kind,text}> }>` partitions tagged entries preserving first-appearance stage order. Untagged / legacy entries (missing `stageId`) collapse into a single fallback group; empty input → empty array. Each group's `status` is resolved from the dashboard tracker (running/ok/failed/skipped) where available, else `undefined`. Unit-tested (ordering, untagged fallback, empty, all-one-stage).
- **AC-03**: Streaming `flush()` (live-stream.ts → onUpdate body) renders a STACK of per-stage sections in TUI: each section has a status-themed header (running=accent/bold + spinner, ok=success, failed=error, skipped=warning/dim), then that stage's lines themed per-kind via `themeLine` and indented 2, with a blank line between sections. The running stage shows up to `RUNNING_TAIL_LINES` (~15) recent lines; completed stages render COMPACT (header + ≤`COMPLETED_TAIL_LINES` (~3) tail lines, or header-only) so the live view stays focused on active work; caps are named constants with a total cap. Per-stage trim notices (kind `trim`) are preserved per section. In non-TUI (ctx.mode !== "tui" or no theme) it emits RAW TEXT — plain `▶ Stage N` headers + indented logs, ZERO ANSI bytes (byte-clean; AC-08 no-leak contract preserved).
- **AC-04**: `buildResultComponent` (src/render/dashboard.ts) §1 is rebuilt into a STACK of per-stage blocks replacing the single merged dim log. Each block is a Container/Text: (a) a bold header Text themed by STATUS via method-style `theme.fg(...)` (e.g. `● Stage 9 — Implementation`); (b) the stage's log lines as Text children themed per-kind via `themeLine`; (c) a per-stage BACKGROUND via pi-tui `Text`'s 4th `customBgFn` arg colored by status — running→`theme.bg("toolPendingBg")`, ok→`theme.bg("toolSuccessBg")`, failed→`theme.bg("toolErrorBg")`, skipped→none/dim — mirroring pi-native's tool-call bubbles using ONLY the public Text customBgFn API (do NOT import internal pi-core components). Failed/running stages render EXPANDED; completed stages render COMPACT (header + status + 1-line tail) so the final report is scannable. §2 (stage progress) and §3 (Markdown summary) are unchanged. Legacy `transcriptTail` (missing stage tags OR string entries) falls back to today's single-section behavior — no throw.
- **AC-05**: `details.transcriptTail` (extension.ts → buildResultComponent) carries `{kind,text,stageId,stageLabel}[]` end-to-end. The `ResultDetails.transcriptTail` type is updated (additive — keep accepting legacy `string` entries). The change-tracker (spec-11), deliverable gate, change gate, mid-run input, scope-aware gate, and Markdown §3 are NOT disturbed and remain green.
- **AC-06**: New tests added and green: `groupByStage` (ordering, untagged fallback, empty, all-one-stage); per-stage section rendering (correct header token per status, a background `customBgFn` present per stage, per-stage tail caps enforced); no-ANSI-in-non-TUI regression for both flush and buildResultComponent; and real-Theme parity for the new section rendering via `tests/helpers/real-theme.ts` `withRealTheme`. The class-Theme regression guards `tests/stream-theme-class-theme.test.ts` AND `tests/render/real-theme-parity.test.ts` MUST stay green (no destructured Theme methods — every new theme access calls method-style `theme.fg`/`theme.bg`/`theme.bold`).
- **AC-07**: `npm run typecheck` is strict-clean and `npm test` is ALL green (existing + new). No regression to: the dashboard widget (aboveEditor), themed stream LineKinds/classification, scope-aware gate, deliverable/change gates, mid-run input handling, or Markdown §3 summary.

## Non-Functional Requirements

- Performance: per-stage tail budgets are bounded by named constants (RUNNING_TAIL_LINES ~15, COMPLETED_TAIL_LINES ~3, plus a TOTAL_CAP) so flush()/buildResultComponent cost stays O(total lines) and never grows unbounded; the existing flush throttle (FLUSH_MS=80) and dashboard throttle (WIDGET_MS=200) are preserved.
- Theme safety (CRITICAL): the real pi `Theme` is a class whose `fg()`/`bg()` read `this.fgColors`/`this.bgColors`; NEVER destructure these methods. All new theme access calls method-style `theme.fg(...)`, `theme.bg(...)`, `theme.bold(...)` (or method-bound wrappers). Verified by real-Theme parity tests.
- Backward compatibility: print/json/RPC/headless modes emit RAW TEXT with zero ANSI bytes — per-stage headers are plain `▶ Stage N`, logs indented; legacy `transcriptTail` shape (string entries / missing stage tags) falls back to a single merged section. No new runtime dependencies (reuse stream-theme LineKind/themeLine/commandBackground, the dashboard stage tracker, pi-tui Text/Container/Markdown + theme.bg tokens).
- Maintainability: new pure helpers (groupByStage, the section builders) are unit-testable in isolation with lightweight structural theme mocks, matching the existing pure-helper discipline in stream-theme.ts/dashboard.ts.
- Accessibility/scanability: completed stages render compact, failed/running expanded, so both the live view and the final report foreground active/failed work over noise.

## Open Questions

- stageId resolution: phase banners carry human labels (`▶ Stage 9 — Implementation`) but the dashboard tracker keys on structured `stage.id` from the `stage` event. Which arrives first, and what is the canonical mapping? Recommended: thread the structured `stage.id`/`stage.label` from the `stage(info)` event into the sink (set currentStageId/Label there) rather than parsing the `▶` label string — avoids fragile regex and keeps the sink and dashboard tracker on the same key.
- Per-stage background scope: confirm pi-tui applies a section-level background (Container bg) or whether the status background must be applied per Text line within the block via the 4th customBgFn arg. The task assumes per-Text customBgFn on header/lines; confirm this reproduces the canonical pi ToolExecutionComponent bubble look without importing internal pi-core components.
- Running-stage animation in the FINAL renderResult: the spinner animates in the live streaming view, but renderResult is static. Confirm a static glyph (e.g. `●`) is correct for the running-but-not-yet-terminal case in the final report (a run that ended while a stage was still running).
- Default per-stage tail cap values: are RUNNING_TAIL_LINES≈15 / COMPLETED_TAIL_LINES≈3 and the total cap acceptable defaults, or does the team want different budgets for very long implementation stages?
