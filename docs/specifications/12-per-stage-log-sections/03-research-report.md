## Summary

Every one of the 7 items the prior attempt left in `openIssues` is resolved against the actual code, each with a concrete low-risk fix — so this brief treats them as resolved findings, not open questions. (RESOLVED-1, ordering) Confirmed in src/nodes.ts: `ctx.events.emit("phase", stage.label)` (L123) fires immediately BEFORE `ctx.events.emit("stage", {id,label,status:"running"})` (L127), so a transcript line pushed by `sink.phase` would inherit the PREVIOUS `currentStageId`. Clean fix is sink-side: in `sink.stage(info)` update currentStageId/Label AND re-tag the most-recent transcript entry when it is a phase line matching `info.label` (zero control-flow change); optionally also swap the two adjacent emits as belt-and-suspenders. (RESOLVED-2, Stage-9 sub-phases) `stage` events emit ONLY from the top-level stage wrapper (L94 `record()` for skipped/ok/failed + L127 for running); the implementation stage is a single node emitting `stage` once with one `id`, and TDD red/green/refactor banners are `phase` lines that inherit `currentStageId="implementation"` automatically — no collapse map is needed, satisfying the spec's "don't over-engineer" by construction. (RESOLVED-3, pi-tui bg API) `buildResultComponent` already builds `Text(content,0,0,customBgFn)` using the 4th public arg (via `commandBackground`, dashboard.ts L381); applying `theme.bg("toolSuccessBg",…)`/`toolPendingBg`/`toolErrorBg` method-style as that 4th arg per-stage header (and optionally per body line) satisfies AC "background customBgFn present per stage" and is read directly by the existing `customBgFnOf(child)` parity helper — NO nested `Box(setBgFn)` is required. (RESOLVED-4, real-theme parity) `withRealTheme(fn)` exposes a real `Theme` class instance whose `bg` is a method (structurally satisfies `DashboardTheme.bg?`); every new status-bg call MUST be method-style `theme.bg(token,text)` (never destructured), the new §1 section test runs under `withRealTheme`, and `tests/stream-theme-class-theme.test.ts` (covers `commandBackground`→`theme.bg` at L67) is extended to also cover `buildResultComponent`'s new bg path. (RESOLVED-5, legacy fallback) Widening `transcriptTail` to `Array<{kind;text;stageId?;stageLabel?}|string>` is purely additive → TS-strict-clean; `groupByStage` assigns a single sentinel to all untagged/string entries → ONE fallback group (today's single dim block); empty → `[]`. (MINOR, perf — non-blocking, folded into Option 2 tradeoffs) Applying `customBgFn` to every body-line Text for a contiguous block look adds one fn-per-line — negligible, but verify render latency on a long multi-stage run if that path is chosen. (OUT OF SCOPE) Interactive collapse/expand (Ctrl+O) is explicitly deferred per spec; static compact-completed / expanded-running is sufficient. Net: all 7 resolved, no remaining blockers.

## Options Considered

### Phase-banner attribution: sink-side re-tag in sink.stage (RECOMMENDED primary)

Zero control-flow change (spec says don't touch control-flow); on `sink.stage(info)` set currentStageId/Label then re-tag the most-recent transcript entry when it's a phase line matching info.label. Verified safe because phase (nodes.ts L123) strictly precedes the running stage event (L127), so 'patch the last phase entry' is always the right line. Alternative is swapping the two adjacent emits so `stage:{running}` fires before `phase` (~2 lines, temporally correct) but that edits the control-flow emit site — keep it as optional belt-and-suspenders, not the primary fix. Tradeoff: sink-side re-tag adds one conditional per stage transition (negligible) and keeps nodes.ts untouched.
### Per-stage background scope: customBgFn on header + each body-line Text (block look)

Most faithful to the pi-native tool-call bubble intent (each stage visually pops as a solid block). Pure public Text 4th-arg API (already proven by commandBackground in dashboard.ts L381), readable per-child by `customBgFnOf`. MINOR perf note (non-blocking): one fn-per-body-line added; negligible but worth a render-latency sanity check on a long multi-stage run. Fallback Option A (header-only customBgFn) is the safe minimum that still literally satisfies AC 'background customBgFn present per stage' and is the lower-risk choice if the block look proves fiddly.
### Stage-9 implementation sub-phases: rely on single `stage` event, no collapse map (RECOMMENDED)

Verified by grep: only nodes.ts L94 (record) and L127 (running) emit `stage`, both in the top-level stage wrapper; implementation is one stage node emitting `stage` once with one `id`, so TDD red/green/refactor banners (all `phase` lines) inherit `currentStageId='implementation'` for free — the spec's 'collapse to implementation' is satisfied by construction. Matches the spec's explicit 'don't over-engineer'. Reserve an explicit collapse map (implTdd/implFix→implementation) only if a future change introduces nested stage emits, which the current code does not.
### Legacy/untagged fallback: sentinel stageId='setup', stageLabel='pre-stage'

Additive type widening `Array<{kind;text;stageId?;stageLabel?}|string>` is TS-strict-clean (no breaking change to ResultDetails). groupByStage coalesces every untagged object-or-string entry into ONE group with the sentinel, so renderResult degrades to exactly today's single dim block; empty input → []. Choosing the sentinel label to mirror the live-stream default-before-first-banner convention ('setup'/'pre-stage') keeps streaming + result naming consistent; either label is acceptable and only affects the legacy fallback header text.
### Real-Theme parity + class-Theme guard: method-style everywhere, extend both guards

`withRealTheme` hands a real `Theme` instance with `bg` as a method (satisfies `DashboardTheme.bg?`), so all new status-bg calls stay method-style `theme.bg(token,text)` — never `const {fg}=theme`. New §1 section test runs under `withRealTheme` asserting no `reading 'fgColors'/'bgColors'` throw; `tests/stream-theme-class-theme.test.ts` (commandBackground→bg at L67) is extended to cover `buildResultComponent`'s new bg path so any future destructuring is caught. `tests/render/real-theme-parity.test.ts` stays green. No widening of DashboardTheme.bg is needed (already optional with method syntax). Tradeoff: strictly method-style discipline everywhere is the cost of keeping both regression guards green.
