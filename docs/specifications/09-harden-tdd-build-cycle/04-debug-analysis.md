# Debug Analysis: Root-Cause Analysis: Four Gaps in the super-dev TDD/Build Cycle (red-oracle, --lib discipline, npm scope, framework-type parity)

- **Date**: 2025-07-22

---

## Summary

Stage 9's skeleton is correct — a deterministic per-phase `tdd-guide → implementer → runBuildGate` loop (hard build/test oracle) committing on `gate.pass || gate.inScopePass` — but four independent gaps let real bugs slip to runtime this session. The unifying root cause across all four is that super-dev **trusts LLM self-report or unscoped/default classification at exactly the weak seams**: (1) the TDD "red" phase is never verified mechanically (the tdd-guide's `{testsWritten, testFiles, allFailing}` return is awaited and discarded at `src/stages/implementation.ts:49`, so already-passing or absent tests pass as "TDD"); (2) the TDD prompt never forbids `--lib`-only self-verification, so the RED check only sees cargo lib unit tests and silently skips the `tests/` integration binaries; (3) out-of-scope failure classification (the cargo `inScopePass` feature that lets pre-existing breakage in untouched crates not abort in-scope work) is gated to Rust only — the npm/vitest/jest path computes `classScope = []` and `classifyOutOfScopeErrors(errors, [])` conservatively marks EVERY error in-scope, so any pre-existing failing test file the branch never touched hard-blocks the gate; (4) the `fgColors` crash class (detaching a method from the class-based pi `Theme` so `fg()` reads an undefined `this.fgColors`) is guarded by exactly ONE regression test (`tests/stream-theme-class-theme.test.ts`) over `themeLine` only — there is no shared real-object parity helper and no test over the rest of the render layer (`commandBackground`, `buildResultComponent`, `packDashboardLines`, `createDashboardWidgetFactory`), so a mock-only coverage of a class-based dependency is a known false-green everywhere else. All four were confirmed by reading the source. None requires any change to the control-flow engine (`nodes.ts`/`workflow.ts`/`pipeline.ts`) or to `MAX_ATTEMPTS`; each is a localized TS addition/orchestration in `src/build-runner.ts`, `src/stages/implementation.ts`, `src/prompts.ts`, plus one test helper + one parity test + one doc. The fixes are backward-compatible and never-throw: greenfield repos (no runner/no tests) and untouched-passing suites behave byte-identically today; every new oracle/helper degrades to a non-blocking status on any ambiguity rather than aborting the pipeline.

## Hypotheses

- H1 (Gap 1, p≈0.95) — the TDD 'red' phase is never verified mechanically. Prediction: in `src/stages/implementation.ts` the `tdd-guide` agent call awaits but discards its control value, and no `runRedCheck` symbol exists in `src/`. CONFIRMED: `implementation.ts:49` does `await ctx.agent({ id: ..., agent: "tdd-guide", prompt: buildTddPrompt(...) })` with no `const tdd = ...`, and `buildTddPrompt` (src/prompts.ts) explicitly returns `{testsWritten, testFiles, allFailing}` that nothing consumes. rg `runRedCheck|MAX_RED_RETRIES` in src/ → no matches. So tests that already pass, or are never written, sail through as 'TDD'. Falsifiable: a phase whose tdd-guide returns `allFailing:false` still proceeds straight to the implementer today.
- H1.a (Gap 1 fallback, p≈0.85) — when a red oracle is added it MUST not stall greenfield pipelines or loop forever. Prediction: the correct design returns a 4th status `unknown` (no runner / no targets / parse ambiguity) that proceeds, and caps re-prompt retries at a small constant. This matches `runBuildGate`'s existing greenfield non-fatal pattern (`ran:[], pass:true`) and the cargo `inScopePass` never-throw contract.
- H2 (Gap 3, p≈0.9) — the no-`--lib` self-verify discipline is wired into the implementer/QA prompts but NOT the TDD prompt. Prediction: `rustDiscipline(s)` (or `RUST_SELF_VERIFY_DISCIPLINE`) appears in `buildImplementPrompt` and `buildQaPrompt` but is absent from `buildTddPrompt`'s instruction list. CONFIRMED: src/prompts.ts defines `RUST_SELF_VERIFY_DISCIPLINE` + `rustDiscipline(s)` and spreads `...(rust ? [rust] : [])` only in `buildImplementPrompt` and `buildQaPrompt`; `buildTddPrompt` emits only `langInstructions`. So a tdd-guide that self-verifies with `cargo test -p <pkg> --lib` reports RED based solely on lib unit tests and the `tests/` integration binaries are never exercised before the gate.
- H3 (Gap 4, p≈0.9) — out-of-scope classification is Rust-only by construction, so npm pre-existing failures always block. Prediction: in `runBuildGate`, the classification scope `classScope` is non-empty ONLY on the `cmds0.language === "rust" && testPackages.length > 0` branch; on the npm/vitest/jest branch `testPackages` is `[]`, so `classifyOutOfScopeErrors(errors, [])` hits its empty-scope branch and returns ALL errors as in-scope. CONFIRMED: `runBuildGate` computes `testPackages=[]` for non-rust repos (only the rust branch assigns touched/auto packages), then `const classScope = cmds0.language === "rust" && testPackages.length > 0 ? classificationScope(...) : testPackages` → `[]`; `classifyOutOfScopeErrors` returns `outOfScopeErrors:[]` whenever `scope.size === 0`; and `inScopePass = pass || (errors.length>0 && outOfScopeErrors.length===errors.length)` is therefore false whenever any npm test fails, regardless of which file. There is also no failing-test-FILE parsing (vitest `❯ path/to/file.test.ts` / jest `FAIL src/file.test.ts`) and no shared `touchedFilePaths(cwd, baseRef)` helper — the touched-file union (`git diff --merge-base` + `git ls-files --others`) is currently inlined inside `detectTouchedCargoPackages` and applied only to `crates/<seg>/` lines.
- H4 (Gap 2, p≈0.8) — the fgColors crash class is guarded by a single regression test over one render function, with no real-object parity coverage for the rest of the render layer. Prediction: only `tests/stream-theme-class-theme.test.ts` exists for this bug class, it exercises `themeLine` (src/render/stream-theme.ts:139) only, and there is no `tests/helpers/real-theme.ts` and no test running `commandBackground` (stream-theme.ts:206), `buildResultComponent` (render/dashboard.ts:360), `packDashboardLines` (dashboard.ts:208), or `createDashboardWidgetFactory` (dashboard.ts:295) against the real class-based Theme. CONFIRMED via rg: the only `initTheme`+fgColors regression is stream-theme-class-theme.test.ts; `withRealTheme` does not exist anywhere. So any render path that a future refactor attaches a plain-object mock Theme to can crash at runtime on a detached `theme.fg(...)` while unit tests stay green.
- H5 (alternate, p≈0.05) — one of the four failures is actually a backend/control-flow regression (signal abort, budget exhaustion, or a stage emitting empty `control:{}`). RULED OUT: `runBuildGate` already try/catches every spawn (SCENARIO-034/AC-02) and degrades non-fatally; `MAX_ATTEMPTS=3` and the Stage 10 review / Stage 11 integration structure are explicitly out of scope and untouched. The four gaps are localized prompt/orchestration/classification defects, not pipeline-engine failures.

## Root Cause

Four INDEPENDENT defects, each a localized trust-seam in `src/` (no control-flow-engine change required):

(1) GAP 1 — RED phase never verified. `src/stages/implementation.ts:49` runs `await ctx.agent({ id: `pipeline.implementation.${phaseId}.tdd.a${attempt}`, agent: \"tdd-guide\", prompt: buildTddPrompt(...) })` and DISCARDS the result; `buildTddPrompt` (src/prompts.ts) returns `{testsWritten, testFiles, allFailing}` that is never read, and `runRedCheck`/`MAX_RED_RETRIES` do not exist (rg → 0 matches in src/). The loop trusts the tdd-guide's self-report that tests were written and are failing. FIX: add `runRedCheck(cwd, testTargets, opts)` to `src/build-runner.ts` (reuse `detectProjectCommands` for runner/language, `resolveTimeoutMs` for the `spawnSync({timeout})` envelope, cached cargo metadata, and the existing scoped-argv + `resolveIntegrationStems`-style stem derivation) returning exactly one of `red`/`green`/`broken`/`unknown`, NEVER throwing (`unknown` = no runner/no targets/parse ambiguity → proceed); wire `implementation.ts` to read the tdd-guide's `testFiles`, call `runRedCheck`, and `while status === green || broken` re-prompt the tdd-guide capped at `MAX_RED_RETRIES = 2`, logging `Implementation ${phaseId} red-oracle: ${status} (ran: …)`, proceeding on `red`/`unknown` (unknown + retry-cap proceed with a loud warning), and telling the implementer prompt the tests are confirmed-red.

(2) GAP 3 — `--lib` discipline missing from the RED prompt. `rustDiscipline(s)` (src/prompts.ts) spreads `RUST_SELF_VERIFY_DISCIPLINE` into `buildImplementPrompt` and `buildQaPrompt` but NOT `buildTddPrompt` (it emits only `langInstructions`). FIX: append `rustDiscipline(s)` to `buildTddPrompt` too so the RED check runs `cargo test -p <pkg>` WITHOUT `--lib` plus any spec-mandated `cargo test --test <stem>`.

(3) GAP 4 — npm out-of-scope classification disabled by construction. In `runBuildGate` (src/build-runner.ts) `testPackages` is non-empty ONLY on the rust branch; npm/vitest/jest leaves it `[]`, so `classScope` (line `cmds0.language === \"rust\" && testPackages.length > 0 ? classificationScope(...) : testPackages`) is `[]`, and `classifyOutOfScopeErrors(errors, [])` returns ALL errors in-scope (its empty-scope conservative branch). FIX: extract the git union now inlined in `detectTouchedCargoPackages` into a shared `touchedFilePaths(cwd, baseRef)` (no NEW git spawn beyond what touched-file detection already does), parse failing TEST FILE paths from runner output (vitest `❯ path/to/file.test.ts`; jest `FAIL src/file.test.ts`) when a non-cargo test run failed, classify a failing file OUT-OF-SCOPE iff NOT in the touched set, and populate `outOfScopeErrors`/`inScopePass` exactly like the cargo path; on any parse ambiguity treat as IN-SCOPE (conservative, no false green). Cargo behavior stays byte-identical.

(4) GAP 2 — framework-type parity covered for one function only. Only `tests/stream-theme-class-theme.test.ts` exercises the real class-based pi `Theme` (`fg()` reads `this.fgColors`; a detached method call throws \"reading 'fgColors'\"), and only over `themeLine` (src/render/stream-theme.ts:139). No `tests/helpers/real-theme.ts` `withRealTheme<T>(fn)`, no whole-render-layer parity test. FIX: add `tests/helpers/real-theme.ts` exporting `withRealTheme<T>(fn: (theme: Theme)=>T): T` (calls idempotent `initTheme()`, passes the real proxy, NEVER destructures methods), a `tests/render/real-theme-parity.test.ts` running `themeLine`, `commandBackground` (stream-theme.ts:206), `buildResultComponent` (render/dashboard.ts:360), `packDashboardLines` (dashboard.ts:208), and `createDashboardWidgetFactory` (dashboard.ts:295) through it asserting no-throw + ANSI output, and `docs/testing-parity.md` documenting: any module wrapping a framework type behind a structural interface MUST have ≥1 parity test via `withRealTheme`/`initTheme`; mock-only coverage of a class-based dependency is a known false-green. (A fully-mechanical graph-based 'no framework mocks' gate is NOT feasible without a type-dependency graph and is explicitly acknowledged as future work.)

All four are backward-compatible and never-throw; `npm run typecheck` strict-clean and `npm test` green (existing build-runner suite + new red-oracle, parity, and npm-inscope suites) are the acceptance signal.
