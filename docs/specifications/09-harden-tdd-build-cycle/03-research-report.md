# Research Report: Hardening the super-dev TDD/implement/build cycle: deterministic RED oracle, npm in-scope gating, and test-parity for class-based dependencies

- **Date**: 2025-11-19
- **Author**: super-dev:research-agent

## Summary

All five prior open issues are now resolved against live sources and the repo's own code. Network IS reachable this session (HTTP 200), so runner output strings are confirmed current: pytest reports ERROR (collection/setup failure ⇒ broken), FAILED (test ran, assertion failed ⇒ red), PASSED (green) [SRC-03, docs.pytest.org/en/stable/how-to/output.html]; jest prints a per-file `FAIL src/file.test.ts` prefix [SRC-02, jestjs.io/docs/cli]; vitest prints `❯ path/to/file.test.ts` per failing file plus `fail`/`failed` counts [SRC-01, vitest.dev/guide/cli.html]. The repo already encodes the correct primitives: `resolveTimeoutMs` is applied per-`spawnSync` (build-runner.ts:1034), the `exec` closure try/catches ANY thrown spawn and degrades to an error entry (never escapes the gate), and `detectTouchedCargoPackages` already computes the `git diff --merge-base` + `ls-files --others` touched-file union — so a shared `touchedFilePaths(cwd, baseRef)` is a pure extraction, not new spawning [SRC-04]. The four gaps map to: (1) a `runRedCheck` that reuses `detectProjectCommands`+`resolveTimeoutMs`+`resolveIntegrationStems`+the cached cargo metadata, loops per-stem with a per-invocation timeout, and never throws; (2) an implementation.ts RED-enforcement loop capped at MAX_RED_RETRIES=2 inside the existing MAX_ATTEMPTS=3 attempt; (3) mirroring the existing `RUST_SELF_VERIFY_DISCIPLINE` constant into `buildTddPrompt` via the already-correct `rustDiscipline(s)` helper (confirmed buildTddPrompt does NOT yet call it [SRC-04]); (4) generalizing cargo `inScopePass`/`outOfScopeErrors` to npm/vitest/jest via path-normalized touched-set membership. The canonical TDD red-before-green discipline [SRC-05, Beck 2002] makes the RED oracle the load-bearing fix. The options below pin each previously-ambiguous invariant with a targeted test, so a future edit cannot silently widen broken→red, read a workspace-relative path as in-scope, nest the retry loop into a cost blow-up, or let a mid-stem timeout throw.

## Options Considered

### Pin RED-classifier strings as named regex constants + full-blob status tests (resolves broken-vs-red drift, open issue #2)

Define classifier regexes (e.g. PYTEST_ERROR=/\bERROR\b/, PYTEST_FAILED=/\bFAILED\b/, PYTEST_PASSED=/passed|PASSED/; jest /^FAIL /m; vitest failed/❯ + collection-error /failed to load|cannot find module/i) as module-level named constants in build-runner.ts, and assert the FULL mocked-spawnSync output blob → expected status in the red-oracle unit test. PRO: a runner upgrade that changes phrasing breaks the pinning test loudly instead of silently widening broken→red (a defective test masquerading as a real red); classification order (broken-regex checked BEFORE failed-regex) is also pinned. CON: a few extra test fixtures; strings must be touched when a runner changes its reporter — but that breakage is exactly the desired signal. Status precedence: broken (compile/collection error) → red (≥1 FAILED) → green (all passed) → unknown (parse ambiguity/no targets/no runner), matching AC-01.
### Per-stem spawnSync with per-invocation timeout + SIGTERM→unknown degradation (resolves multi-stem timeout, open issue #5)

runRedCheck mirrors runBuildGate's exec exactly: a for loop over stems (cargo: cargo test -p <pkg> --test <stem>; vitest/jest: single vitest run <files>/npm test -- <files>; pytest: pytest <files>), each spawnSync(argv, {cwd, timeout: timeoutMs}) wrapped in try/catch. A timeout sets r.signal==='SIGTERM'/r.status===null → classify unknown (proceed), NEVER throw and NEVER emit a SyncError. PRO: matches the existing proven pattern byte-for-byte; each scoped test is cheap (single stem) so a per-invocation envelope is correct, not the whole loop; mid-loop timeout degrades cleanly per AC-01/SCENARIO-005. CON: N stems × timeout budget worst-case, but stems are derived from the tdd-guide's own reported testFiles (small N), and runRedCheck is advisory (never blocks on unknown). The vitest/jest/pytest paths run ONCE (files passed as args), so only cargo loops — and cargo scoped test binaries are individually fast.
### Path-normalize touched-set membership via path.resolve/relative before set comparison (resolves npm in-scope normalization, open issue #3)

Extract touchedFilePaths(cwd, baseRef): string[] from the existing git union in detectTouchedCargoPackages (returns repo-rooted paths), then build the membership set as new Set(touched.map(p => path.resolve(cwd, p))). Parse failing-test paths from runner output (vitest ❯ <p>; jest FAIL <p>), normalize each via path.resolve(cwd, raw) (handles cwd-/workspace-relative) and test membership against the resolved set. A failing file NOT in the set ⇒ out-of-scope. PRO: resolve collapses all of ./, ../, cwd-relative, and absolute-under-cwd to one canonical form, so a workspace-relative vitest path matches the repo-rooted git path; the npm-inscope test exercises a cwd-relative failing path to pin this. CON: on parse ambiguity (unparseable path line) → classify IN-SCOPE (conservative: never grants a false green, per AC-04). NEVER throws: wrap parsing in try/catch, default-to-in-scope on failure. Cargo inScopePass stays byte-identical.
### Assert ≤(MAX_RED_RETRIES+1) runRedCheck calls/attempt + retry-exhaustion proceeds (resolves cost-bound non-multiplicativity, open issue #4)

MAX_RED_RETRIES=2 lives INSIDE one attempt; worst case per attempt = 1 initial tdd-guide + runRedCheck, then up to 2 retry rounds (re-prompt tdd-guide + runRedCheck) = 3 runRedCheck spawns max/attempt, plus 1 implementer + 1 gate. Across MAX_ATTEMPTS=3 that is ≤9 advisory red-oracle spawns total (non-multiplicative with the gate). Pin a unit test that stubs runRedCheck to always return 'green' and asserts it is called exactly MAX_RED_RETRIES+1 (=3) times per attempt, AND that after exhaustion the phase proceeds to the implementer + gate (loud warning) rather than looping the outer attempt. PRO: a future edit that accidentally nests the retry loop inside the attempt loop (multiplying cost) breaks this test. CON: a test counter coupling to the constant; acceptable since the constant is load-bearing and documented. Implementation.ts treats gate.pass || gate.inScopePass as commit-green — unchanged. unknown always proceeds immediately (0 retries), red proceeds immediately (0 retries); only green/broken retry.
### Mirror RUST_SELF_VERIFY_DISCIPLINE into buildTddPrompt via the existing rustDiscipline(s) helper (Gap 3)

The RUST_SELF_VERIFY_DISCIPLINE constant and rustDiscipline(s) helper already exist and are shared by buildImplementPrompt + buildQaPrompt; buildTddPrompt confirmed to NOT yet use them [SRC-04]. Fix is a one-line append of rustDiscipline(s) to buildTddPrompt's Instructions array, gated on s.language==='rust' (no change to non-Rust stacks). PRO: zero new constants, single source of truth for the no-`--lib` discipline; RED checks then run full cargo test -p <pkg> (covering tests/ integration binaries) not just lib unit tests. CON: none — it is the documented AC-03 intent; buildImplementPrompt text needs no change beyond confirming both share the same constant.
### withRealTheme parity helper + whole-render-layer parity test (Gap 2; honest about graph-gate limits)

tests/helpers/real-theme.ts exports withRealTheme<T>(fn) that calls initTheme() (idempotent) and passes the real class-based proxy theme; callers use method-style theme.fg() (NEVER destructure — the fgColors-class guard). tests/render/real-theme-parity.test.ts runs themeLine, commandBackground, buildResultComponent, packDashboardLines, createDashboardWidgetFactory through it asserting no-throw + ANSI. PRO: catches the fgColors detached-method class of bug that plain-object mocks miss (a known false-green), and generalizes tests/stream-theme-class-theme.test.ts to the whole render layer. CON: a fully-mechanical 'no framework mocks' graph-based gate is NOT feasible without a type-dependency graph — acknowledge as future work in docs/testing-parity.md; the parity convention is the durable, proportionate fix (≥1 parity test per module wrapping a framework type behind a structural interface).
