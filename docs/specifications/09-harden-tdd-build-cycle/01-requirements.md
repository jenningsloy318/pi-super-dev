# Requirements: Harden super-dev TDD/implement/build cycle: RED oracle, no-`--lib` parity, scope-aware npm gate, render-layer test parity

- **Date**: 2026-07-20
- **Author**: super-dev:requirements-clarifier
- **Type**: bug-fix
- **Priority**: critical
- **Status**: draft

---

## Executive Summary

Four proven gaps in the Stage 9 TDD→implement→build-gate loop are fixed as a single pure-TS change to pi-super-dev. (1) The TDD "red" phase is currently never verified — `src/stages/implementation.ts:70` calls the `tdd-guide` agent and discards its result, so "TDD" is really "T+D"; we add a deterministic `runRedCheck` oracle and a bounded re-prompt loop. (2) The `buildImplementPrompt` no-`--lib` rule already exists, but `buildTddPrompt` lacks it, so RED checks can miss integration binaries; we mirror it. (3) The cargo `inScopePass`/`outOfScopeErrors` concept is cargo-only, so pre-existing npm/vitest/jest failures block the gate; we generalize it via a shared `touchedFilePaths` git helper. (4) The fgColors-style "real runtime shape vs. mock" bug class passed every gate because unit tests used a mock Theme; we deliver a reusable `withRealTheme` parity helper + whole-render-layer regression + convention doc. All gates degrade safely (never throw); greenfield/in-scope behaviors are unchanged; no new runtime deps; typecheck clean + `npm test` all green. A fully-mechanical "no framework mocks" graph-based gate is explicitly future work.

## Acceptance Criteria

- **AC-01**: `runRedCheck(cwd, testTargets, opts)` exists in `src/build-runner.ts` alongside `runBuildGate`, reusing `detectProjectCommands` (for the runner + `pm`/`language`), `resolveTimeoutMs` (for `spawnSync({timeout})`), the cached cargo-metadata, and the existing scoped-test argv helpers. It runs the project's test command SCOPED to the tdd-guide's reported `testFiles`/targets and returns exactly one status: `"red"` (tests compiled/collected AND ≥1 failed — the desired pre-implement state); `"green"` (tests ran and ALL passed — weak/already-implemented); `"broken"` (compile/collection error — defective test); `"unknown"` (no detectable runner, no test targets, or parse ambiguity — greenfield fallback, MUST NOT block). Per-language invocation: cargo → `cargo test -p <pkg> --test <stem>` per stem (derive stems from testFiles like `resolveIntegrationStems`); vitest/jest → `vitest run <files>` / `npm test -- <files>`; pytest → `pytest <files>`. The function NEVER throws (any error/ambiguity → `unknown` or a conservative status).
- **AC-02**: `src/stages/implementation.ts` enforces RED inside the existing `attempt` loop: after the tdd-guide agent, it reads the agent's `testFiles` and calls `runRedCheck`, then loops `while status === "green" || status === "broken"` re-prompting the tdd-guide to write genuinely-failing/non-broken tests, capped at `MAX_RED_RETRIES = 2` within the current attempt. On `"red"` OR `"unknown"` it proceeds to the implementer. Every outcome is logged as `Implementation ${phaseId} red-oracle: ${status} (ran: …)`. It NEVER stalls the pipeline: `unknown` always proceeds immediately; on retry-cap exhaustion it proceeds with a loud warning. The implementer prompt is informed that the tests are confirmed-red (so its goal is to green them). The outer `MAX_ATTEMPTS = 3` structure and the `gate.pass || gate.inScopePass` commit condition are unchanged.
- **AC-03**: `buildTddPrompt` (in `src/prompts.ts`) mirrors the no-`--lib` discipline already present in `buildImplementPrompt` via `RUST_SELF_VERIFY_DISCIPLINE`: the RED phase must run `cargo test -p <pkg>` WITHOUT `--lib` and run any spec-mandated integration target (`cargo test --test <stem>`), so integration tests are covered during the RED check, not just lib unit tests. The existing `rustDiscipline(s)` helper is reused (Rust-only, gated on `s.language === "rust"`); no change to the already-correct `buildImplementPrompt` text is required beyond confirming both builders share the same source string.
- **AC-04**: `runBuildGate` generalizes in-scope/out-of-scope classification to npm/vitest/jest projects. After a failed non-cargo test run it parses failing TEST FILE paths from runner output (vitest `❯ path/to/file.test.ts`; jest `FAIL src/file.test.ts`), computes the touched-file set via a NEW shared `touchedFilePaths(cwd, baseRef)` helper that extracts the existing git `diff --merge-base` + `ls-files --others` union currently embedded in `detectTouchedCargoPackages` (so NO new git spawn beyond what touched-file detection already does), classifies each failing test file as OUT-OF-SCOPE if NOT in the touched set, and populates `outOfScopeErrors`/`inScopePass` exactly like the cargo path. NEVER throws; on any parse ambiguity → treat as IN-SCOPE (conservative, grants no false green). The cargo `inScopePass` behavior is unchanged byte-for-byte.
- **AC-05**: `tests/helpers/real-theme.ts` exports `withRealTheme<T>(fn: (theme: Theme) => T): T` that calls `initTheme()` (idempotent) and passes the real class-based proxy `theme` to `fn`. It NEVER destructures Theme methods (guards `this.fgColors`); callers use method-style `theme.fg(...)` or a wrapper. `tests/render/real-theme-parity.test.ts` runs `themeLine`, `commandBackground`, `buildResultComponent`, `packDashboardLines`, and `createDashboardWidgetFactory`'s output through `withRealTheme` and asserts no-throw + ANSI output, generalizing `tests/stream-theme-class-theme.test.ts` into a shared regression for the WHOLE render layer. `docs/testing-parity.md` documents the convention: any module wrapping a framework type behind a structural interface MUST have ≥1 parity test via `withRealTheme`/`initTheme`; mock-only coverage of a class-based dependency is a known false-green.
- **AC-06**: Backward compatibility preserved: greenfield repos (no runner/no tests) → `runRedCheck` returns `unknown` → proceeds exactly as today (no new spawn aborts); repos with no test failures are unchanged; cargo `inScopePass` is unchanged; npm repos with only in-scope failures still block correctly; repos whose suite has pre-existing failures in untouched files now pass via `inScopePass`. `npm run typecheck` is strict-clean. `npm test` is ALL green — the existing large build-runner suite (scoped-args, touched-crates, autoscope, inscope-classification, resolver-validation, nonregression, package-wiring) plus the new red-oracle + parity + npm-inscope suites. Any helper-signature change that affects existing tests is updated IN THE SAME change.
- **AC-07**: No regression to existing behaviors: dashboard widget, themed stream, scope-aware cargo gate, mid-run input injection, and Markdown §3 artifacts. No change to the subprocess/session backend or control-flow engine (`nodes.ts`/`workflow.ts`/`pipeline.ts`); no change to `MAX_ATTEMPTS` (3) or the Stage 10 review / Stage 11 integration structure; no new runtime dependencies.

## Non-Functional Requirements

- Performance: `runRedCheck` spawns one scoped test command per attempt using the same cached timeout envelope (`resolveTimeoutMs`) as `runBuildGate`; the npm in-scope path adds NO new git process beyond what `detectTouchedCargoPackages` already spawns (it reuses the extracted `touchedFilePaths` helper). RED re-prompts are capped at `MAX_RED_RETRIES = 2` so the worst-case cost per phase is `2 tdd-guide + 2 red-check + 1 implementer + build-gate`, bounded and never infinite.
- Reliability / fail-safe: EVERY new gate/oracle/git helper degrades instead of throwing — red oracle → `unknown` → proceed; in-scope parse ambiguity → conservative IN-SCOPE (never a false green); git/spawn errors → empty touched set → conservative in-scope. No path can stall or abort the pipeline except the existing genuine in-scope gate failure after `MAX_ATTEMPTS`.
- Correctness: the fix targets the specific failure class (fgColors) — a class-based framework type whose methods read `this.fgColors` and crash only on a detached call. Mocks hide this; the real Theme proxy via `withRealTheme` exposes it deterministically.
- Type safety: TypeScript strict-clean (`npm run typecheck`); no `any` leaks across the new boundaries; new helpers reuse existing typed interfaces (`ProjectCommands`, `BuildGateResult`, `GateOptions`).
- Maintainability: the convention is codified in `docs/testing-parity.md` so future render-layer modules have a checklist, and the shared `touchedFilePaths` helper eliminates the duplicated git-union logic.
- Scope/blast-radius: pure-TS change inside pi-super-dev only; zero changes to the control-flow engine, session backend, or external service contracts.

## Open Questions

- Should `runRedCheck`'s `red` classification require a MINIMUM number of failing tests (e.g. ≥1 is the spec default), or should it additionally warn when the tdd-guide reported `testsWritten > 0` but `testFiles` resolved to zero runnable targets (a likely defective guide output)? Recommended: warn-and-treat-as-`broken` to force a re-prompt, since the guide's own contract says it wrote tests.
- AC-04 touches the npm path but pytest output formatting is runner/version-dependent. Recommended scope: explicitly support vitest + jest file-path markers now (the two `pm`-driven node runners this repo uses), and degrade to conservative IN-SCOPE for any unrecognized pytest output rather than over-fitting — confirm pytest is in-scope or out-of-scope for THIS repo before shipping.
- For `runRedCheck` on cargo: the spec's `gate.integration` stems are validated at build-gate time via `resolveIntegrationStems`. Should the RED oracle re-validate the same stems, or trust the tdd-guide's raw `testFiles`? Recommended: trust the guide's `testFiles` for the RED check (it is testing freshly-written tests, which may not yet be valid stems), and keep validation only at the build gate — avoids a false `broken` from over-eager validation.
- The `MAX_RED_RETRIES = 2` constant: confirm it should be a module-level const in `src/stages/implementation.ts` (mirroring `MAX_ATTEMPTS`) and not configurable via spec/gate. Recommended: module-level const, no new config surface, matching the task's explicit `MAX_RED_RETRIES = 2` directive.
